---
title: What Nub replaces
description: A per-command map from Nub's surface to the tools it stands in for, from the file runner and package manager to the bin runner, script runner, watcher, and Node version manager.
---

Nub is one binary covering the jobs a typical Node project spreads across a runtime wrapper, a package manager, a bin runner, a script runner, a watcher, and a version manager. Each row below maps one Nub command or subsystem to the tools it stands in for, and the "what it does" cell names the limits where Nub is a partial substitute rather than overclaiming.

| Nub | What it does | Replaces |
|---|---|---|
| `nub <file>` | Runs `.js`/`.ts`/`.jsx`/`.tsx` on stock Node, transpiling TypeScript and JSX, loading `.env*` files, and polyfilling modern globals per Node-version band. A drop-in for `node <file>`. | `node`, `tsx`, `ts-node`, `dotenv-cli`, `node --env-file`, `deno run`, `bun run` (for the TS/JSX/`.env` cases) |
| `nub install` / `i` / `ci` | Resolves and links `node_modules` from the project's existing lockfile. pnpm-compatible CLI; round-trips npm, pnpm, and bun lockfiles; reads yarn. | `npm install`, `pnpm install`, `yarn install`, `bun install`, `npm ci` |
| `nub add` / `remove` / `update` | Mutates dependencies and the lockfile (`add`/`remove`/`update`, plus `dedupe`, `prune`, `rebuild`, `link`/`unlink`, `import`, `patch`). | `npm add`/`rm`, `pnpm add`/`remove`/`update`, `yarn add`/`remove`, `bun add`/`remove` |
| `nub outdated` / `why` / `audit` | Inspects the dependency graph: `outdated`, `why`, `audit` (with `--fix`), `licenses`, `query`, `info`/`view`. | `npm outdated`/`why`/`audit`, `pnpm outdated`/`why`/`audit`, `yarn audit` |
| `nub pack` / `publish` | Packages and publishes: `pack`, `publish`, `version`, `dist-tag`, `deprecate`, `owner`, `login`/`logout`. | `npm pack`/`publish`/`version`, `pnpm publish`, `yarn pack`/`npm publish` |
| `nubx` / `nub dlx` | Runs a binary from `node_modules/.bin`, fetching it from the registry on a local miss. | `npx`, `pnpm dlx`, `pnpm exec`, `yarn dlx`, `bunx` |
| `nub run <script>` | Runs a `package.json` script with the full lifecycle-hook and workspace surface, including recursive runs with `--filter` and topological ordering. Not a task-graph runner with cross-run caching. | `npm run`, `pnpm run`, `yarn run`, `bun run`, `pnpm -r` (recursive scripts) |
| `nub watch` | Restarts the entry point on changes to its resolved import graph, `.env*` files, `tsconfig.json`, and manifest. Restart-mode, on Node's own `--watch`; not HMR. | `nodemon`, `node --watch`, `tsx watch`, `bun --watch` |
| `nub node` | Provisions and pins Node versions, fetching from nodejs.org on demand (`install`, `ls`, `uninstall`, `pin`). | `nvm`, `fnm`, `n`, `volta`, `asdf` (the Node plugin) |
| `nub pm use` | Provisions and runs the exact package-manager version a project pins in `packageManager`. | `corepack` |
| `nub upgrade` | Self-updates the `nub` binary to the latest release. | the tool's own installer |

The package-manager rows are split by job — install, mutate, inspect, publish — but they are one engine and one lockfile. The CLI surface follows pnpm; lockfile compatibility follows whatever the project already uses, so npm, pnpm, and bun projects round-trip cleanly and yarn projects are read-only. See [install](/docs/install) for per-incumbent lockfile fidelity.

Two limits are worth restating from the table. `nub run` runs scripts with full workspace fan-out and topological ordering, but it does not maintain a cross-run task cache the way a dedicated build orchestrator like Turborepo does. `nub watch` restarts the process on change; it does not hot-swap modules in a running process.
