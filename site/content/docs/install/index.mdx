Nub has its own package manager: an install engine that resolves dependencies and links `node_modules`. The CLI is pnpm-shaped, and the on-disk lockfile stays in your project's native format — npm, pnpm, and Bun round-trip cleanly; Yarn is read-only. The engine is the vendored [aube](https://github.com/jdx/aube) engine, embedded as a library and driven by Nub's own CLI.

<Callout title="You don't need to use Nub's package manager">
The installer is entirely optional. Nub works alongside whatever package manager your project already uses — keep running `npm`, `pnpm`, `yarn`, or `bun` exactly as you do today, and reach for Nub for everything else: running files, scripts, and binaries. Nothing about Nub requires its installer.
</Callout>

## `nub install`

Install resolves the dependency graph and links `node_modules`. The verb, its aliases, and its flags follow pnpm:

```bash
nub install                      # alias: nub i
nub install --frozen-lockfile    # fail if the lockfile is out of date
nub install -P                   # --prod / --production
nub install -D                   # dev only
nub install --node-linker hoisted
nub ci                           # clean install from the lockfile
```

The accepted flags are pnpm's:

```bash
# accepted flags (pnpm spellings)
--frozen-lockfile
--no-frozen-lockfile
--prefer-frozen-lockfile
--prod, -P            # production install
--dev, -D
--ignore-scripts
--no-optional
--offline
--prefer-offline
--lockfile-only
--force
--node-linker
--registry
--dir, -C             # pnpm's spelling, not npm's --prefix
```

Where pnpm and npm spell a flag differently, Nub takes pnpm's — the `--dir`/`-C` spelling rather than `--prefix`, and `--frozen-lockfile` rather than npm's `ci`-only behavior.

## `nub add`

Add resolves a package, links it, and writes the dependency into `package.json` — pnpm-shaped, including the dependency-type and workspace flags:

```bash
nub add <pkg>                  # alias: a
nub add -D <pkg>               # --save-dev
nub add -E <pkg>               # --save-exact (pin, no ^)
nub add -O <pkg>               # --save-optional
nub add --save-peer <pkg>      # peerDependencies
nub add -g <pkg>               # global install
nub add -w <pkg>               # write to the workspace root
nub add --save-catalog <pkg>   # add into the workspace catalog
nub add --allow-build=<pkg>    # pre-approve its build scripts for this install
nub add --no-save <pkg>        # link without persisting to package.json
```

## `nub remove`

Remove drops a dependency from `package.json` and relinks `node_modules`. Aliases follow pnpm:

```bash
nub remove <pkg>           # rm / uninstall / un / uni
nub remove -D <pkg>        # remove only from devDependencies
nub remove -g <pkg>        # remove a global package
nub remove -w <pkg>        # remove from the workspace root
```

## `nub update`

Update re-resolves dependencies within their ranges; `--latest` rewrites the `package.json` ranges to the newest resolved versions:

```bash
nub update                 # up — refresh all deps within range
nub update <pkg>           # update a single dependency
nub update -L              # --latest: move past the manifest range
nub update -E -L           # pin the rewritten range to an exact version
nub update -D              # devDependencies only
nub update -P              # production only
nub update --lockfile-only # refresh the lockfile, leave node_modules alone
```

## `nub dedupe`

Dedupe collapses duplicate versions in the lockfile to fewer, shared resolutions:

```bash
nub dedupe          # rewrite the lockfile with deduped resolutions
nub dedupe --check  # CI: exit non-zero if dedupe would change anything
```

## `nub import`

Import converts another package manager's lockfile to Nub's `pnpm-lock.yaml`, without installing:

```bash
nub import          # read package-lock.json / yarn.lock / bun.lock → pnpm-lock.yaml
nub import --force  # overwrite an existing pnpm-lock.yaml
```

## `nub pm`

The install engine is distinct from `nub pm`, the [package meta-manager](/docs/pm), which provisions and runs the exact pnpm/npm/yarn your project *pins* (corepack's job). For "fetch and run the project's pinned PM," reach for `nub pm`; for "install dependencies," this engine. The two compose: `nub pm shim` lets bare `npm` / `pnpm` / `yarn` route through the pin while you keep using whatever installer you prefer.

## Per-package-manager compatibility

What Nub reads and writes is keyed to the project's incumbent package manager. Each manager gets its own compatibility page — the lockfile fidelity, the config surfaces honored, and the gaps:

- [pnpm](/docs/install/pnpm) — the closest mirror: native `pnpm-lock.yaml` v9 read+write, isolated layout, `.pnpmfile.cjs` hooks.
- [npm](/docs/install/npm) — `package-lock.json` v2/v3 round-trip byte-for-byte, `.npmrc` at project and user scope.
- [Bun](/docs/install/bun) — `bun.lock` text round-trip, `trustedDependencies` build trust.
- [Yarn](/docs/install/yarn) — read-only: `yarn.lock` (Classic and Berry) is consumed, never rewritten.

## Post-install

This behavior is shared across pnpm, npm, and Bun, so it lives here once. Nub does **not** run dependency build scripts indiscriminately the way npm does. The posture is **deny-by-default, with a gated automatic-trust floor on top** (`defaultTrust=true` under Nub by default — distinct from upstream aube, which keeps it off).

A dependency's `preinstall`/`install`/`postinstall` scripts run only if one of these holds:

1. **An explicit allow** — the package is on `pnpm.onlyBuiltDependencies` / `pnpm.allowBuilds`, Bun's `trustedDependencies`, or you `nub approve-builds`'d it. An explicit `false` (`allowBuilds: { pkg: false }`, `neverBuiltDependencies`) always wins and carves the package out of every layer below.
2. **The default-trust floor vouches for it** — the package is on aube's curated default-trust list (Bun's vendored `default-trusted-dependencies.txt`) **and** every gate holds:
   - **registry provenance** — it resolved from a registry (git / file / link / tarball sources never qualify; an npm alias can't borrow a listed name's trust),
   - **advisory vetting** — an OSV `MAL-*` advisory gate ran against this install's graph (default `advisoryCheck=on`, fires on any fresh resolution), *or* the graph was inherited from an unchanged lockfile that was advisory-checked when it was written (a frozen install / `nub ci` / a teammate clone),
   - **cooling window** — the resolved version's recorded publish time is older than `minimumReleaseAge` (default 1440 minutes / 24h). **Unknown publish time fails closed.**

Otherwise the script is skipped and Nub prints `WARN_NUB_IGNORED_BUILD_SCRIPTS` naming the package, with `nub approve-builds` as the remedy.

The floor's reach is therefore bounded by the lockfile in play. A pnpm/npm install where Nub resolves freshly (or reinstalls a lockfile Nub itself wrote, which carries the `time:` block) gives the floor everything it needs, so curated packages like `esbuild` build automatically:

```console
# captured: nub 0.0.44, pnpm-incumbent, esbuild@0.21.5 — no allowBuilds entry
$ nub install
WARN defaultTrust: running build scripts for 1 default-trusted package(s): esbuild@0.21.5 code=WARN_NUB_DEFAULT_TRUST_BUILDS count=1 packages=["esbuild@0.21.5"]
dependencies:
+ esbuild@0.21.5
```

But on an **existing foreign lockfile that carries no publish-time data** — notably an incumbent `bun.lock` — the cooling-window gate has nothing to read, fails closed, and the floor stays inert, so the same package is skipped (see the [Bun page](/docs/install/bun#trusteddependencies) for the captured A/B). This is the correct, conservative behavior: Nub never trusts a build script it can't age-vet.

The OS-level **build jail** (network-blocked, filesystem-scoped sandbox) is compiled in but **off by default**; it is opt-in via the neutral `paranoid` / `npm_config_paranoid` setting (macOS + Linux only; Windows is a passthrough). Do not assume installs are sandboxed unless you've enabled it.

## Package manager inference

Everything below — which lockfile format Nub reads and writes, which config surfaces it honors, whether a write is allowed — keys off one decision: **which package manager owns this project.** Nub never asks; it infers the incumbent from the project itself, then mirrors it. The signals, in precedence order:

1. **The `packageManager` field** (`package.json`) — the Corepack standard. A declaration is a *stronger* signal than any lockfile on disk: a pnpm project that picked up a stray `package-lock.json` (a contributor ran `npm install`) keeps resolving as pnpm, not silently flipping format.
2. **The `devEngines.packageManager` field** — the fallback declaration, object form (`{ "name": "pnpm" }`) or array form. Outranked by `packageManager` when both are present; a same-tool version disagreement or a different-tool disagreement between the two is a hard error, not a silent pick.
3. **The lockfile on disk** — consulted only when no declaration names a manager. Filename precedence is `pnpm-lock.yaml` (Nub's own canonical, which it shares with a pnpm incumbent) → `bun.lock` → `yarn.lock` → `npm-shrinkwrap.json` → `package-lock.json`.
4. **Nothing present** — a fresh project with no declaration and no lockfile is **nub identity**: Nub writes its own canonical `pnpm-lock.yaml` (v9 bytes) and honors only un-branded, cross-tool config.

A declaration that names `nub` (what `nub pm use nub` writes) is nub identity too, and accepts whatever lockfile the project already has.

In a workspace, only the **root** carries the declaration and lockfile; Nub walks up from the working directory to find it, so a member package with no lockfile of its own resolves to the root's identity.

<Callout title="Contradictions are loud, never silently resolved">
When the signals disagree, Nub stops rather than guess:

- **Declaration vs lockfile mismatch** — `package.json` declares one manager but only *other* managers' lockfiles are present. Code `ERR_NUB_LOCKFILE_DECLARATION_MISMATCH`.
- **Ambiguous lockfiles** — two or more managers' lockfiles sit side by side with no declaration to break the tie. Code `ERR_NUB_LOCKFILE_AMBIGUOUS`.

The remedy for both is the same: name the owner explicitly with `nub pm use <pm>`, or remove the stale lockfile. Under nub identity, Nub's own `pnpm-lock.yaml` does **not** silently outrank a foreign lockfile beside it — that pairing is the ambiguity error, not a quiet win.
</Callout>

<Callout title="Inference vs the pinned PM">
This inference picks the *install engine's* incumbent — the format Nub reads and writes. It is distinct from the version `nub pm` provisions: the meta-manager resolves a *pin* (`.yarnrc.yml` `yarnPath` → `packageManager` → `devEngines`) to fetch and run an exact pnpm/npm/yarn binary. The two share the declaration signals but answer different questions: "which format do I install in?" versus "which PM binary do I run?".
</Callout>

## Store and disk layout

Regardless of the incumbent, Nub installs through a global content-addressed store and links into an isolated virtual store — the same scheme aube uses, under Nub's own directory names.

### Global content store

Package files are deduplicated by content hash in a global store at `$XDG_DATA_HOME/nub/store/v1/` (defaulting to `~/.local/share/nub/store/v1/`). Every install imports from it, so the same package version on disk once is shared across every project. Run `nub store path` to print the resolved location. Files materialize into `node_modules` by reflink (APFS/btrfs), hardlink (ext4), or copy — whichever the filesystem supports — so a populated tree costs little extra disk.

### Virtual store

The default `node_modules` layout is **isolated**: direct dependencies sit at the top level, and transitive packages link into a per-project virtual store so phantom dependencies fail instead of resolving by accident. Nub's virtual store is `node_modules/.nub/` (where pnpm uses `node_modules/.pnpm/`) — equivalent in shape, but not byte-shared, so alternating tools relinks the tree. The `--node-linker hoisted` flag produces a flat, npm-style layout instead; npm, Yarn, and Bun incumbents default to `hoisted`, while pnpm and nub-identity projects keep `isolated`.

### Offline and the global virtual store

A global virtual store reuses fully materialized package directories across projects (on by default for local installs, off under CI). A handful of build-heavy frameworks opt out of it by default (`@sveltejs/kit`, `next`, `nuxt`, `parcel`). Installs run offline when the store already holds every needed package; `--offline` forces it and `--prefer-offline` tries the cache first. See [aube's store docs](https://aube.jdx.dev/package-manager/global-virtual-store) for the full model.
