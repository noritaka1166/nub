The pnpm engine is the one Nub mirrors most closely. The CLI surface is modeled on pnpm's, the isolated `node_modules` layout matches pnpm's symlink scheme, and when pnpm is the incumbent, Nub reads and writes pnpm's own `pnpm-lock.yaml` in pnpm's native format. A repo can move between `pnpm install` and `nub install` without either tool rewriting the other's lockfile.

## Lockfile

When pnpm is the incumbent â€” detected from a `packageManager: "pnpm@â€¦"` field or an existing `pnpm-lock.yaml` â€” Nub reads and writes `pnpm-lock.yaml` directly. The format is pnpm lockfile **v9** (`lockfileVersion: '9.0'`), what pnpm 9 and 10 emit.

A lockfile Nub writes installs cleanly under pnpm with `--frozen-lockfile`:

```bash
nub install                      # writes pnpm-lock.yaml (lockfileVersion: '9.0')
pnpm install --frozen-lockfile   # "Lockfile is up to date, resolution step is skipped"
```

Real round-trip, captured against nub 0.0.44 (`is-odd@3.0.1`):

```console
$ nub install
dependencies:
+ is-odd@3.0.1

$ grep lockfileVersion pnpm-lock.yaml
lockfileVersion: '9.0'

$ pnpm install --frozen-lockfile
Lockfile is up to date, resolution step is skipped
```

The round-trip holds in both directions, for single packages and for workspaces with `catalog:` and `workspace:` entries. A pnpm-written v9 lockfile reads cleanly under `nub install --frozen-lockfile`, verified against pnpm 10.15.1.

One cosmetic difference: Nub records a `time:` block (per-package publish timestamps, used by the resolver's release-age floor) at the top of the lockfile. pnpm tolerates and preserves it.

The v9 reader and writer are the supported path. **Older pnpm lockfile formats are not handled.** A `lockfileVersion: '6.0'` lockfile (pnpm 8) or `5.4` (pnpm 7) keeps its root dependencies under a top-level `dependencies:` map rather than the v9 `importers:` map; Nub's reader looks only at `importers:`, so it sees an empty project, treats the install as already satisfied, links nothing, and exits successfully:

```console
# captured: nub 0.0.44 on a lockfileVersion: '6.0' lockfile
$ nub install
nub 0.0.44 Â· âś“ Already up to date

$ ls node_modules/is-odd
ls: node_modules/is-odd: No such file or directory      # nothing linked
```

Run a one-time `pnpm install` (or `pnpm dedupe`) with pnpm 9+ to migrate the lockfile to v9 before handing the project to Nub.

## Config

When pnpm is the incumbent, Nub honors pnpm's configuration surface:

- `pnpm-workspace.yaml` â€” workspace `packages:` globs and the `catalog:` / `catalogs:` maps are read.
- `package.json` `pnpm.overrides` â€” version pins are applied during resolution and written into the lockfile's `overrides:` block.
- `pnpm.onlyBuiltDependencies` / `pnpm.neverBuiltDependencies` / `pnpm.allowBuilds` â€” the build-script allow/denylists feed Nub's lifecycle-script policy.
- `.pnpmfile.cjs` / `.pnpmfile.mjs` â€” the `readPackage` and `afterAllResolved` hooks run. Nub shells out to Node to execute them, so existing pnpm hooks work unchanged.

```yaml
# pnpm-workspace.yaml â€” read when pnpm is the incumbent
packages:
  - 'packages/*'
catalog:
  lodash.merge: 4.6.2
```

```json
// package.json
{
  "pnpm": { "overrides": { "is-number": "7.0.0" } }
}
```

Both resolve as pnpm would, and the result round-trips through `pnpm install --frozen-lockfile`. (Verified: a `pnpm.overrides` pin of `is-number` to `7.0.0` is applied during resolve and written into the lockfile's `overrides:` block; a `.pnpmfile.cjs` `readPackage` hook that rewrites a dependency takes effect.)

This config surface is **gated on pnpm being the incumbent.** In a project whose incumbent manager is npm, Yarn, or Bun â€” or a nub-identity project â€” Nub does not read `pnpm-workspace.yaml`, the `package.json` `pnpm.*` namespace, pnpm's global `~/.config/pnpm/auth.ini`, or a stray `.pnpmfile.cjs`. Each is another tool's state, and a `.pnpmfile.cjs` in particular shapes resolution, so suppressing it is a correctness call. When such a file is present but ignored, Nub prints one warning naming it and the fix (delete it, name it explicitly with `--pnpmfile`, or `nub pm use pnpm`). An explicit `--pnpmfile <path>` always loads.

## Install behavior

The default `node_modules` layout is **isolated** â€” the same symlink-into-a-virtual-store scheme pnpm uses. Nub's virtual store directory is `node_modules/.nub/` rather than pnpm's `node_modules/.pnpm/`; the two never collide, but they are not interchangeable, so switching tools relinks the tree. The `--node-linker hoisted` flag gives the flat npm-style layout. There is **no `pnp` linker** â€” `--node-linker pnp` (and `nodeLinker: pnp`) is rejected with a clear error:

```console
$ nub install --node-linker pnp
  Ă— node-linker=pnp is not supported by nub; use `isolated` (default) or
  â”‚ `hoisted`
```

Lifecycle scripts for dependencies are **skipped by default**, exactly as pnpm 10 does. A package runs its install scripts only when it is allowlisted via `pnpm.onlyBuiltDependencies` (or `pnpm.allowBuilds`), or when Nub's gated default-trust floor vouches for it (see [post-install](/docs/install#post-install)); `nub approve-builds` is the interactive way to add an entry. `pnpm.neverBuiltDependencies` is an explicit denylist that wins over any allow â€” and over the floor.

## Gaps

- **Lockfile format is v9-only.** v6 (pnpm 8) and v5.4 (pnpm 7) lockfiles are not parsed â€” Nub silently treats them as an empty/satisfied project. Migrate with pnpm 9+ first.
- **No `nodeLinker: pnp`.** Only `isolated` (default) and `hoisted`; `--node-linker pnp` errors.
- **Virtual store is `node_modules/.nub/`, not `.pnpm/`.** Layouts are equivalent in shape but not byte-shared; alternating tools relinks.
- **pnpm config surface is incumbent-gated.** A non-pnpm-incumbent project ignores `pnpm-workspace.yaml`, `pnpm.*`, and `.pnpmfile.cjs` (with a warning for the pnpmfile).
- **A `time:` block appears in the lockfile.** Nub writes per-package timestamps pnpm does not; harmless and preserved across the round-trip, but a `git diff` against a pnpm-written lockfile will show it.
