---
title: Yarn
description: Yarn is supported read-only — Nub reads yarn.lock (Classic v1 and Berry v2+) to install and run a project, but never writes it. Treat a Yarn project as something Nub consumes, not maintains.
---

Yarn is supported **read-only**. Install and run work; any command that would rewrite `yarn.lock` refuses before the engine touches anything:

```console
$ nub add lodash
Error: nub add: refusing to modify yarn.lock — adding a dependency re-resolves and rewrites yarn.lock
  Run it with yarn directly:
    yarn add lodash
$ echo $?
1
```

Nub reads `yarn.lock` to install and run a project but never writes it. Keep using `yarn` for anything that changes the dependency graph.

Both formats are read:

- **Classic** — `yarn.lock` with no `__metadata` block (Yarn 1.x).
- **Berry** — the `__metadata` / `version` format (Yarn 2+).

## Running a Yarn project

```bash
nub install        # reads yarn.lock, links node_modules, runs lifecycle scripts
nub ci             # frozen install: yarn.lock is law; drift is a hard error
nub run <script>   # runs package.json scripts
nubx <bin>         # one-off binary execution
```

Top-level `resolutions` in `package.json` is honored — Nub applies it the way Yarn does. The `packageManager` field is what tells Nub the project is a Yarn project.

## Refused commands

Run these with `yarn`:

```bash
nub add <pkg>       # → yarn add
nub remove <pkg>    # → yarn remove
nub update          # → yarn upgrade
nub dedupe          # → yarn dedupe
```

The same gate fires on an `install` / `ci` that *would* rewrite the lockfile rather than just read it: a project that declares Yarn but has no `yarn.lock` yet, `nub install --force` / `--no-frozen-lockfile` / `--lockfile-only`, or a `yarn.lock` that no longer satisfies `package.json`. Run `yarn install` for those.

There is no path that converts *to* `yarn.lock`. The `nub import` command produces Nub's own `pnpm-lock.yaml`:

```console
$ nub import        # in a yarn project
Imported 2 packages from yarn.lock to pnpm-lock.yaml
```

<Callout title="nub pm use yarn is a different surface">
The `nub pm use yarn` command is the [meta-manager](/docs/pm) provisioning command, and it *does* work: it fetches and pins Yarn via the `packageManager` field. It does not produce a `yarn.lock` or convert your install engine; it just provisions the Yarn binary.
</Callout>

## Config files

Nub reads **`.npmrc` only** for registry, auth, scopes, CA, and proxy settings. It does not read Yarn's config files for any package-management setting:

- `.yarnrc.yml` (Berry) — project **and** global (`~/.yarnrc.yml`) — is read for exactly one key, `yarnPath`, and nothing else. When the project commits a Yarn release and points `yarnPath` at it, `nub pm` runs that release directly. Every *config* key in the file (registry, auth, scopes, CA, proxy, `nodeLinker`, …) is ignored.
- classic `.yarnrc` (Yarn 1) is not parsed.

In practice, none of these Yarn-config keys take effect under Nub:

```yaml
# .yarnrc.yml — ALL IGNORED by nub
npmRegistryServer: https://registry.internal/   # ❌ ignored — nub reads .npmrc `registry=`
npmScopes:                                       # ❌ ignored — per-scope registries unread
  myorg:
    npmRegistryServer: https://npm.myorg.dev/
    npmAuthToken: ...                            # ❌ ignored — private-dep auth NOT applied
npmAuthToken: ...                                # ❌ ignored
httpsCaFilePath: ./ca.pem                        # ❌ ignored
httpProxy: http://proxy:8080                     # ❌ ignored
```

If your private-registry auth or custom registry lives in `.yarnrc.yml` (which for Berry is the *only* place it lives), Nub will not pick it up and private installs fail with 401/403. Put the equivalent settings in `.npmrc`, which Nub does read.

## Node linkers

Nub produces a **hoisted `node_modules`** for a Yarn project regardless of any `nodeLinker` setting — that setting lives in `.yarnrc.yml`, which Nub does not read:

```yaml
# .yarnrc.yml
nodeLinker: pnp     # ❌ nub installs hoisted node_modules instead
nodeLinker: pnpm    # ❌ also hoisted
```

**Nub-the-runtime fully supports Plug'n'Play.** If a project was *already* installed by Yarn in PnP mode, Nub can run it — `nub <file>`, `nub run`, and `nubx` honor `.pnp.cjs` across all supported Node versions. What Nub cannot do is *produce* a PnP install. So the supported workflow for a PnP project is: install with `yarn`, run with `nub`.

## Gaps

Several Yarn `package.json` and `.yarnrc.yml` fields have no effect under Nub:

- `dependenciesMeta.<pkg>.built` — Yarn's per-package build gate is not read; Nub uses its own build-script approval (`nub approve-builds`),
- `packageExtensions` — Yarn's resolution patches are not applied,
- `supportedArchitectures` — Yarn's platform/arch download controls are not read.
