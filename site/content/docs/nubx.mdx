Run a command-line tool by name with `nubx`: a local bin from `node_modules/.bin` first, a registry fetch when it isn't installed. It's a drop-in for `npx` and `pnpm dlx`/`pnpm exec` — existing invocations carry over, and it accepts pnpm `exec`'s core workspace flags (`-r`, `--filter`, `--parallel`).

```bash
nubx eslint .                          # resolve from node_modules/.bin and run
nub exec eslint .                      # alias: nub exec === nubx
nubx --filter @org/api tsc --build     # run a local bin across a filtered workspace
nub dlx -p cowsay -c 'cowsay hi | tr a-z A-Z'  # one-off tool, no install, shell pipeline
```

For a locally-resolved bin, `nubx` does the `node_modules/.bin` walk-up in Rust and exec's the resolved binary directly, so the wrapper overhead that `npx` and `pnpm exec` pay on every call — a full Node bootstrap — approximately disappears: 19× lighter than `npx` on a native CLI like `esbuild` (and roughly 17× lighter than `pnpm exec`). The tool that actually runs is identical to what `npx` would run; only the wrapper gets out of the way.

## Run a local CLI

Point `nubx` at the bin name and let it resolve from `node_modules/.bin`. No `node` bootstrap in the wrapper, so resolution returns in single-digit milliseconds.

```bash
nubx eslint .
nubx prisma generate
nubx tsc
```

## Pass arguments

Everything after the bin name goes straight to the underlying tool, untouched. There is no `--` separator to remember — `nubx` doesn't interpret your tool's flags.

```bash
nubx eslint . --fix --max-warnings 0
nubx tsc --noEmit
nubx prisma migrate dev --name init
nubx vitest run --coverage
```

## --node

Pass `--node` to run the tool with runtime augmentation disabled — see [the runtime overview](/docs/runtime#--node) for the full contract.

```bash
nubx --node prisma generate
nub exec --node tsc --noEmit
```

What still happens under `--node`: the workspace-aware `node_modules/.bin` walk-up, the registry DLX fallback, and `nubx`'s own CLI machinery. Only the runtime augmentation is disabled. Reach for it when a tool's shebang chain needs byte-exact plain Node, when bisecting a Nub-vs-Node issue, or when CI wants exact Node runtime behavior with `nubx`'s resolution.

## Download and execute

When a bin isn't on the `node_modules/.bin` chain, `nubx` falls back to a registry fetch — it resolves the package, installs it into a throwaway project, and runs it, then drops the scratch dir. This is the same local-first-then-DLX model as `npx` and `pnpm dlx`; nub is a complete package manager, so it does the fetch itself rather than shelling out to a foreign PM. Following `pnpm dlx` semantics, there is no interactive confirm-prompt — it just fetches and runs.

```console
$ nubx cowsay "Hello"
dependencies:
+ cowsay@1.6.0

 _______
< Hello >
 -------
        \   ^__^
         \  (oo)\_______
            (__)\       )\/\
                ||----w |
                ||     ||
```

This fallback is unique to the `nubx` (and `nub exec`-as-`nubx`) entry point. Plain `nub exec` is no-network: a missing bin there prints an install suggestion and exits 127 instead of reaching the registry.

## Version specifiers

A version specifier like `nubx cowsay@1.5.0` goes straight through the DLX fetch path: `nubx` resolves and runs that exact version from the registry, no local install required.

```console
$ nubx cowsay@1.5.0 "pinned"
dependencies:
+ cowsay@1.5.0  latest 1.6.0

 ________
< pinned >
 --------
        \   ^__^
         \  (oo)\_______
            (__)\       )\/\
                ||----w |
                ||     ||
```

## Supply-chain cooling window

The DLX fetch runs through Nub's normal install resolver, so the same `minimumReleaseAge` cooling window that protects `nub install` also protects `nubx` — a fetched tool's resolved version must be older than the window before it's used. The default is **24 hours**, the same as `install`; set `minimumReleaseAge=0` to disable it.

```console
# .npmrc: minimumReleaseAge tightened past every release, minimumReleaseAgeStrict=true
$ nubx cowsay "hi"
ERR_NUB_NO_MATURE_MATCHING_VERSION
  × dlx install failed
  ╰─▶ no version of cowsay matching latest is older than … minute(s) (minimumReleaseAgeStrict=true)
  help: blocked by age gate: 1.6.0
```

The window is resolved from the same `.npmrc` scopes that `npx` reads: the built-in default, your user-scope `~/.npmrc` (and `NPM_CONFIG_MINIMUM_RELEASE_AGE`), and a **project-local `.npmrc`** in the directory you run `nubx` from. So a per-project override applies to a `nubx` fetch even though the tool installs into a throwaway scratch project — alongside the project's registry, scoped registries, and auth, matching `npx`. By default the resolver falls back to the next-oldest satisfying version; `minimumReleaseAgeStrict=true` fails instead. See [the default-trust floor](/docs/install#default-trust-floor) for the full posture.

## Yarn Plug'n'Play

Tools that only walk `node_modules/.bin/` — `npx`, `pnpm exec`, `bunx` — miss Yarn Berry's PnP bins. `nubx` resolves them: when the `.bin` walk-up misses in a PnP tree, it falls through to a PnP-aware runner that finds and executes the bin. See [module resolution](/docs/runtime/resolution#yarn-plugnplay) for how the shared PnP machinery works.

## Flag differences

- **`-c`/`--shell-mode`** — the shell-string form lives on `nub dlx -c` (shown above); `nub exec`/`nubx` run a resolved bin directly.
- **Recursive scope** — `nub exec` covers `-r`, `--filter`, and `--parallel`; the richer recursive flags (`--resume-from`, `--reporter`, `--stream`) live on `nub run`.
