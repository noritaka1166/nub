---
title: Replacing corepack
description: Nub is the version manager and package-manager dispatcher you should never have to think about. The right Node and the right package manager are provisioned and run for you, with no shim process between you and your install.
---

Corepack and nvm sit in front of every package-manager command you run: nvm picks the Node, corepack's shim spawns a second Node just to launch your real package manager, and only then does the install start. Nub is both of those, folded into one binary — and it dispatches the install in-process, so nothing sits in the middle.

```console
$ which nub
/Users/you/.nub/bin/nub

$ nub install
dependencies:
+ is-odd@3.0.1
```

That is the whole interaction. No `corepack enable`, no activation prompt, no second Node booting to hand off to a third — `nub install` resolves the project's package manager, version-manages the Node it runs on, and performs the install itself.

## The shim tax

Corepack is a shim. When you run `pnpm install` through it, corepack boots a Node process, reads your `packageManager` pin, locates (or downloads) that exact pnpm, and execs it — a second Node startup that runs before pnpm has done any work. On first use of a pinned version it also prints a download notice to stderr:

```console
$ pnpm install
! Corepack is about to download https://registry.npmjs.org/pnpm/-/pnpm-9.15.4.tgz
# ❌ a second Node boots before pnpm runs — every invocation
```

Nub carries no shim process. The package-manager engine is compiled into the Nub binary and called in-process, so dispatching an install is a function call, not a subprocess. The interposition that costs corepack a full Node startup costs Nub a few milliseconds.

This is the dispatch overhead only — the extra Node startup corepack pays *before* your package manager runs, not the install itself. Your package manager's own work is identical either way; Nub removes the wrapper in front of it.

## You should never type it

The point of folding corepack and nvm into Nub is that the version management disappears. Pin a Node version and a package manager once, and every later `nub install`, `nub add`, or `nub run` provisions and runs exactly what the project declared — no enable step, no activation, no per-command ceremony.

When you do need to set or change the pin, one command writes it:

```console
$ nub pm use pnpm
Fetching pnpm 11.7.0 (4 MB)...
using pnpm@11.7.0
  package.json: packageManager = pnpm@11.7.0 (+sha512)
  package.json: devEngines.packageManager = { name: "pnpm", version: "^11.7.0", onFail: "warn" }
  no lockfile — the next install writes pnpm-lock.yaml
```

That writes the same `packageManager` field corepack reads, so the project stays portable to anyone still running corepack — but under Nub the field is just data, never an activation gate.

### Bailouts

The provisioning is the product; the explicit commands exist for the rare moment the automatic path can't cover.

- [`nub pm`](/docs/pm) — provision and run the exact pinned pnpm, npm, or yarn deliberately, inspect the resolved version, or point it at a private registry.
- [`nub node`](/docs/node) — warm the Node cache before you run, list installed versions, or reclaim disk.
- `nub pm shim` — make **bare** `pnpm` / `npm` / `yarn` commands route through the project's pin, for muscle memory or tools you don't control. This is `corepack enable`, except each shim *is* the Nub binary, so deciding which manager to run costs zero Node boots. See [Package meta-manager → shims](/docs/pm#nub-pm-shim).
