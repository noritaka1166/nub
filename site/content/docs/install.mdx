Nub has its own package manager: an install engine that resolves dependencies and links `node_modules`. The CLI is pnpm-shaped no matter which manager the project came from, and the on-disk lockfile stays in the incumbent's native format â€” npm, pnpm, and Bun round-trip cleanly; Yarn is read-only. The engine is the vendored [aube](https://github.com/jdx/aube) engine, embedded as a library and driven by Nub's own CLI.

<Callout title="You don't need the package manager to use Nub">
The installer is optional. Keep using your existing `npm` / `pnpm` / `yarn` / `bun` and reach for Nub for everything else â€” running files, scripts, and binaries â€” by installing the [`nub pm shim`](/docs/pm#nub-pm-shim). The shims route bare PM commands through the [package meta-manager](/docs/pm), which provisions and runs the exact version your project pins.
</Callout>

## `nub install`

Install resolves the dependency graph and links `node_modules`. The verb, its aliases, and its flags follow pnpm:

```bash
nub install                      # alias: nub i
nub install --frozen-lockfile    # fail if the lockfile is out of date
nub install -P                   # --prod / --production
nub install -D                   # dev only
nub install --node-linker hoisted
nub ci                           # clean install from the lockfile
```

The accepted flags are pnpm's: `--frozen-lockfile`, `--no-frozen-lockfile`, `--prefer-frozen-lockfile`, `--prod`/`-P`, `--dev`/`-D`, `--ignore-scripts`, `--no-optional`, `--offline`, `--prefer-offline`, `--lockfile-only`, `--force`, `--node-linker`, `--registry`, and `--dir`/`-C`. Where pnpm and npm spell a flag differently, Nub takes pnpm's: `--dir`/`-C` rather than `--prefix`, `--frozen-lockfile` rather than npm's `ci`-only behavior.

## `nub add` / `remove` / `update` / `dedupe` / `import`

Graph-mutating verbs route through the engine, pnpm-shaped:

```bash
nub add <pkg>          # alias: a
nub remove <pkg>       # rm / uninstall / un / uni
nub update             # up
nub dedupe
nub import             # convert another PM's lockfile to pnpm-lock.yaml
```

The full registered verb set also covers `why`, `outdated`, `list`/`ls`, `patch` / `patch-commit` / `patch-remove`, `approve-builds`, `prune`, `rebuild`, `fetch`, `link`/`unlink`, `audit`, `licenses`, `bin`, `root`, `store`, `config`, `pkg`, `publish`, `pack`, `dlx`, `create`, and more â€” the complete list lives in the engine verb registry.

## `nub pm`

The install engine is distinct from `nub pm`, the [package meta-manager](/docs/pm), which provisions and runs the exact pnpm/npm/yarn your project *pins* (corepack's job). For "fetch and run the project's pinned PM," reach for `nub pm`; for "install dependencies," this engine. The two compose: `nub pm shim` lets bare `npm` / `pnpm` / `yarn` route through the pin while you keep using whatever installer you prefer.

## Install-script trust

This behavior is shared across pnpm, npm, and Bun, so it lives here once. Nub does **not** run dependency build scripts indiscriminately the way npm does. The posture is **deny-by-default, with a gated automatic-trust floor on top** (`defaultTrust=true` under Nub by default â€” distinct from upstream aube, which keeps it off).

A dependency's `preinstall`/`install`/`postinstall` scripts run only if one of these holds:

1. **An explicit allow** â€” the package is on `pnpm.onlyBuiltDependencies` / `pnpm.allowBuilds`, Bun's `trustedDependencies`, or you `nub approve-builds`'d it. An explicit `false` (`allowBuilds: { pkg: false }`, `neverBuiltDependencies`) always wins and carves the package out of every layer below.
2. **The default-trust floor vouches for it** â€” the package is on aube's curated default-trust list (Bun's vendored `default-trusted-dependencies.txt`) **and** every gate holds:
   - **registry provenance** â€” it resolved from a registry (git / file / link / tarball sources never qualify; an npm alias can't borrow a listed name's trust),
   - **advisory vetting** â€” an OSV `MAL-*` advisory gate ran against this install's graph (default `advisoryCheck=on`, fires on any fresh resolution), *or* the graph was inherited from an unchanged lockfile that was advisory-checked when it was written (a frozen install / `nub ci` / a teammate clone),
   - **cooling window** â€” the resolved version's recorded publish time is older than `minimumReleaseAge` (default 1440 minutes / 24h). **Unknown publish time fails closed.**

Otherwise the script is skipped and Nub prints `WARN_NUB_IGNORED_BUILD_SCRIPTS` naming the package, with `nub approve-builds` as the remedy.

The floor's reach is therefore bounded by the lockfile in play. A pnpm/npm install where Nub resolves freshly (or reinstalls a lockfile Nub itself wrote, which carries the `time:` block) gives the floor everything it needs, so curated packages like `esbuild` build automatically:

```console
# captured: nub 0.0.44, pnpm-incumbent, esbuild@0.21.5 â€” no allowBuilds entry
$ nub install
WARN defaultTrust: running build scripts for 1 default-trusted package(s): esbuild@0.21.5 code=WARN_NUB_DEFAULT_TRUST_BUILDS count=1 packages=["esbuild@0.21.5"]
dependencies:
+ esbuild@0.21.5
```

But on an **existing foreign lockfile that carries no publish-time data** â€” notably an incumbent `bun.lock` â€” the cooling-window gate has nothing to read, fails closed, and the floor stays inert, so the same package is skipped (see the [Bun section](#trusteddependencies) for the captured A/B). This is the correct, conservative behavior: Nub never trusts a build script it can't age-vet.

The OS-level **build jail** (network-blocked, filesystem-scoped sandbox) is compiled in but **off by default**; it is opt-in via the neutral `paranoid` / `npm_config_paranoid` setting (macOS + Linux only; Windows is a passthrough). Do not assume installs are sandboxed unless you've enabled it.

## Package manager inference

Everything below â€” which lockfile format Nub reads and writes, which config surfaces it honors, whether a write is allowed â€” keys off one decision: **which package manager owns this project.** Nub never asks; it infers the incumbent from the project itself, then mirrors it. The signals, in precedence order:

1. **The `packageManager` field** (`package.json`) â€” the Corepack standard. A declaration is a *stronger* signal than any lockfile on disk: a pnpm project that picked up a stray `package-lock.json` (a contributor ran `npm install`) keeps resolving as pnpm, not silently flipping format.
2. **The `devEngines.packageManager` field** â€” the fallback declaration, object form (`{ "name": "pnpm" }`) or array form. Outranked by `packageManager` when both are present; a same-tool version disagreement or a different-tool disagreement between the two is a hard error, not a silent pick.
3. **The lockfile on disk** â€” consulted only when no declaration names a manager. Filename precedence is `lock.yaml` (Nub's own canonical) â†’ `pnpm-lock.yaml` â†’ `bun.lock` â†’ `yarn.lock` â†’ `npm-shrinkwrap.json` â†’ `package-lock.json`.
4. **Nothing present** â€” a fresh project with no declaration and no lockfile is **nub identity**: Nub writes its own canonical `pnpm-lock.yaml` (v9 bytes) and honors only un-branded, cross-tool config.

A declaration that names `nub` (what `nub pm use nub` writes) is nub identity too, and accepts whatever lockfile the project already has.

In a workspace, only the **root** carries the declaration and lockfile; Nub walks up from the working directory to find it, so a member package with no lockfile of its own resolves to the root's identity.

### Contradictions are loud, never silently resolved

When the signals disagree, Nub stops rather than guess:

- **Declaration vs lockfile mismatch** â€” `package.json` declares one manager but only *other* managers' lockfiles are present. Code `ERR_NUB_LOCKFILE_DECLARATION_MISMATCH`.
- **Ambiguous lockfiles** â€” two or more managers' lockfiles sit side by side with no declaration to break the tie. Code `ERR_NUB_LOCKFILE_AMBIGUOUS`.

The remedy for both is the same: name the owner explicitly with `nub pm use <pm>`, or remove the stale lockfile. Note that under nub identity, Nub's own `lock.yaml` does **not** silently outrank a foreign lockfile beside it â€” that pairing is the ambiguity error, not a quiet win.

<Callout title="Inference vs the pinned PM">
This inference picks the *install engine's* incumbent â€” the format Nub reads and writes. It is distinct from the version `nub pm` provisions: the meta-manager resolves a *pin* (`.yarnrc.yml` `yarnPath` â†’ `packageManager` â†’ `devEngines`) to fetch and run an exact pnpm/npm/yarn binary. The two share the declaration signals but answer different questions: "which format do I install in?" versus "which PM binary do I run?".
</Callout>

## pnpm

pnpm is the package manager Nub mirrors most closely. The CLI surface is modeled on pnpm's, the isolated `node_modules` layout matches pnpm's symlink scheme, and when pnpm is the incumbent, Nub reads and writes pnpm's own `pnpm-lock.yaml` in pnpm's native format. A repo can move between `pnpm install` and `nub install` without either tool rewriting the other's lockfile.

| `pnpm-lock.yaml` (v9) | `pnpm-workspace.yaml` | `.pnpmfile.cjs` | `pnpm.overrides` | `resolutions` | `allowBuilds` / `onlyBuiltDependencies` | `catalog:` | `workspaces` | `.npmrc` |
|:---:|:---:|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| âś“ read+write | âś“ | âś“ | âś“ | âś“ | âś“ | âś“ | âś“ | âś“ |

(All pnpm-named surfaces are honored only when pnpm is the incumbent. pnpm prefers `pnpm.overrides`; top-level `resolutions` is still honored â€” pnpm supports it for Yarn compatibility â€” but top-level `overrides` is npm/bun's field and is ignored under pnpm.)

### Lockfile

When pnpm is the incumbent â€” detected from a `packageManager: "pnpm@â€¦"` field or an existing `pnpm-lock.yaml` â€” Nub reads and writes `pnpm-lock.yaml` directly. The format is pnpm lockfile **v9** (`lockfileVersion: '9.0'`), what pnpm 9 and 10 emit.

A lockfile Nub writes installs cleanly under pnpm with `--frozen-lockfile`:

```bash
nub install                      # writes pnpm-lock.yaml (lockfileVersion: '9.0')
pnpm install --frozen-lockfile   # "Lockfile is up to date, resolution step is skipped"
```

Real round-trip, captured against nub 0.0.44 (`is-odd@3.0.1`):

```console
$ nub install
dependencies:
+ is-odd@3.0.1

$ grep lockfileVersion pnpm-lock.yaml
lockfileVersion: '9.0'

$ pnpm install --frozen-lockfile
Lockfile is up to date, resolution step is skipped
```

The round-trip holds in both directions, for single packages and for workspaces with `catalog:` and `workspace:` entries. A pnpm-written v9 lockfile reads cleanly under `nub install --frozen-lockfile`, verified against pnpm 10.15.1.

One cosmetic difference: Nub records a `time:` block (per-package publish timestamps, used by the resolver's release-age floor) at the top of the lockfile. pnpm tolerates and preserves it.

The v9 reader and writer are the supported path. **Older pnpm lockfile formats are not handled.** A `lockfileVersion: '6.0'` lockfile (pnpm 8) or `5.4` (pnpm 7) keeps its root dependencies under a top-level `dependencies:` map rather than the v9 `importers:` map; Nub's reader looks only at `importers:`, so it sees an empty project, treats the install as already satisfied, links nothing, and exits successfully:

```console
# captured: nub 0.0.44 on a lockfileVersion: '6.0' lockfile
$ nub install
nub 0.0.44 Â· âś“ Already up to date

$ ls node_modules/is-odd
ls: node_modules/is-odd: No such file or directory      # nothing linked
```

Run a one-time `pnpm install` (or `pnpm dedupe`) with pnpm 9+ to migrate the lockfile to v9 before handing the project to Nub.

### Config

When pnpm is the incumbent, Nub honors pnpm's configuration surface:

- `pnpm-workspace.yaml` â€” workspace `packages:` globs and the `catalog:` / `catalogs:` maps are read.
- `package.json` `pnpm.overrides` â€” version pins are applied during resolution and written into the lockfile's `overrides:` block.
- `pnpm.onlyBuiltDependencies` / `pnpm.neverBuiltDependencies` / `pnpm.allowBuilds` â€” the build-script allow/denylists feed Nub's lifecycle-script policy.
- `.pnpmfile.cjs` / `.pnpmfile.mjs` â€” the `readPackage` and `afterAllResolved` hooks run. Nub shells out to Node to execute them, so existing pnpm hooks work unchanged.

```yaml
# pnpm-workspace.yaml â€” read when pnpm is the incumbent
packages:
  - 'packages/*'
catalog:
  lodash.merge: 4.6.2
```

```json
// package.json
{
  "pnpm": { "overrides": { "is-number": "7.0.0" } }
}
```

Both resolve as pnpm would, and the result round-trips through `pnpm install --frozen-lockfile`. (Verified: a `pnpm.overrides` pin of `is-number` to `7.0.0` is applied during resolve and written into the lockfile's `overrides:` block; a `.pnpmfile.cjs` `readPackage` hook that rewrites a dependency takes effect.)

This config surface is **gated on pnpm being the incumbent.** In a project whose incumbent manager is npm, Yarn, or Bun â€” or a nub-identity project â€” Nub does not read `pnpm-workspace.yaml`, the `package.json` `pnpm.*` namespace, pnpm's global `~/.config/pnpm/auth.ini`, or a stray `.pnpmfile.cjs`. Each is another tool's state, and a `.pnpmfile.cjs` in particular shapes resolution, so suppressing it is a correctness call. When such a file is present but ignored, Nub prints one warning naming it and the fix (delete it, name it explicitly with `--pnpmfile`, or `nub pm use pnpm`). An explicit `--pnpmfile <path>` always loads.

### Install behavior

The default `node_modules` layout is **isolated** â€” the same symlink-into-a-virtual-store scheme pnpm uses. Nub's virtual store directory is `node_modules/.nub/` rather than pnpm's `node_modules/.pnpm/`; the two never collide, but they are not interchangeable, so switching tools relinks the tree. The `--node-linker hoisted` flag gives the flat npm-style layout. There is **no `pnp` linker** â€” `--node-linker pnp` (and `nodeLinker: pnp`) is rejected with a clear error:

```console
$ nub install --node-linker pnp
  Ă— node-linker=pnp is not supported by nub; use `isolated` (default) or
  â”‚ `hoisted`
```

Lifecycle scripts for dependencies are **skipped by default**, exactly as pnpm 10 does. A package runs its install scripts only when it is allowlisted via `pnpm.onlyBuiltDependencies` (or `pnpm.allowBuilds`), or when Nub's gated default-trust floor vouches for it (see [Install-script trust](#install-script-trust) above); `nub approve-builds` is the interactive way to add an entry. `pnpm.neverBuiltDependencies` is an explicit denylist that wins over any allow â€” and over the floor.

### Gaps and partial support

- **Lockfile format is v9-only.** v6 (pnpm 8) and v5.4 (pnpm 7) lockfiles are not parsed â€” Nub silently treats them as an empty/satisfied project. Migrate with pnpm 9+ first.
- **No `nodeLinker: pnp`.** Only `isolated` (default) and `hoisted`; `--node-linker pnp` errors.
- **Virtual store is `node_modules/.nub/`, not `.pnpm/`.** Layouts are equivalent in shape but not byte-shared; alternating tools relinks.
- **pnpm config surface is incumbent-gated.** A non-pnpm-incumbent project ignores `pnpm-workspace.yaml`, `pnpm.*`, and `.pnpmfile.cjs` (with a warning for the pnpmfile).
- **A `time:` block appears in the lockfile.** Nub writes per-package timestamps pnpm does not; harmless and preserved across the round-trip, but a `git diff` against a pnpm-written lockfile will show it.

### `package-lock.json`

When `package-lock.json` is the project's incumbent lockfile, Nub reads it, resolves against it, and writes it back in npm's exact format â€” no `pnpm-lock.yaml` is introduced.

```jsonc
// package-lock.json written by `nub install` â€” accepted by `npm ci` byte-for-byte
{
  "name": "app",
  "version": "1.0.0",
  "lockfileVersion": 3,
  "requires": true,
  "packages": {
    "": { "name": "app", "version": "1.0.0", "dependencies": { "is-odd": "3.0.1" } },
    "node_modules/is-odd": {
      "version": "3.0.1",
      "resolved": "https://registry.npmjs.org/is-odd/-/is-odd-3.0.1.tgz",
      "integrity": "sha512-CQpnWPrDwmP1+SMHXZhtLtJv90yiyVfluGsX5iNCVkrhQtU3TQHsUWPG9wkdk9Lgd5yNpAg9jQEo90CBaXgWMA==",
      "license": "MIT",
      "dependencies": { "is-number": "^6.0.0" },
      "engines": { "node": ">=4" }
    }
  }
}
```

Supported reading covers `lockfileVersion` 2 and 3. Version 1 (npm 6 and earlier) is rejected with a clear error rather than silently misread:

```console
# captured: nub 0.0.44 on a lockfileVersion 1 package-lock.json
$ nub install
ERR_NUB_LOCKFILE_PARSE

  Ă— failed to parse lockfile
  â•°â”€â–¶ failed to parse lockfile /path/to/package-lock.json:
      package-lock.json lockfileVersion 1 is not supported (need v2 or v3)
```

Round-trip fidelity is the headline. On a no-op install (the graph already matches the lock), Nub reproduces npm's writer byte-for-byte â€” including npm's two-pass key ordering (`@npmcli/arborist` â†’ `json-stringify-nice`), the `devOptional` collapse, `peerDependenciesMeta.optional`, and platform fields (`os`/`cpu`/`libc`) â€” so the file does not churn in diffs and `npm ci` accepts it unchanged. Verified against nub 0.0.44 / npm 11.13.0: a no-op `nub install` over an npm-written lock produced a `diff`-identical file, and `npm ci` exited 0 on it. (The original audit reproduced the same on graphs up to a 45-package tree with scoped, dev, and platform-gated `fsevents` packages.)

One honest caveat on version drift. Nub preserves the incoming `lockfileVersion` on a no-op rewrite â€” a v2 lock stays v2, legacy `dependencies` block and all. But a **mutating** operation (`nub add`, `nub remove`, anything that changes the graph) rewrites the lock as `lockfileVersion: 3`. A v2 lock is therefore upgraded to v3 the first time you change a dependency. This was confirmed empirically: a v2 lock survived `nub install` as v2, then became v3 after `nub add is-number`. The result is still valid and `npm ci`-accepted (v3 has been npm's default since npm 7), but the version field changes â€” a real, expected diff, not a corruption.

### `.npmrc`

When npm is the incumbent PM, Nub reads `.npmrc` at two scopes:

```
1. project   ./.npmrc          (all keys)
2. user      ~/.npmrc          (all keys, relocatable via NPM_CONFIG_USERCONFIG)
```

At those scopes everything works: default and scoped registries (`@scope:registry`), auth tokens (`//host/:_authToken`), TLS (`strict-ssl`, `cafile`/`ca`), proxy settings, and install-shaping keys like `engine-strict` and `save-exact`. The `npm_config_*` environment variables are honored for install behavior, and CLI-passed config wins over both files. Precedence matches npm where the scopes overlap: project over user over env over CLI (project highest of the file scopes). Verified: with `registry` set in both files, `nub config get registry` returns the project value; a key set only in `~/.npmrc` falls back to the user value.

**The gap â€” global and builtin scopes are not read.** npm's full config chain is `default â†’ builtin â†’ project â†’ user â†’ global â†’ env â†’ cli`. Nub covers project, user, env, and cli; it does **not** read:

- the **global** npmrc at `$PREFIX/etc/npmrc` (what `npm config set --global ...` writes), and
- npm's **builtin** npmrc (the one shipped inside the npm install, holding CI/corporate defaults).

As a consequence, `NPM_CONFIG_GLOBALCONFIG` and `prefix`-based relocation of the global file are also not honored. This is a real, verified divergence: a private registry, auth token, CA, or `legacy-peer-deps` set **only** in the global npmrc is invisible to Nub, which can mean resolving against the wrong registry or failing auth on a machine that relies on global config (common on corporate/CI setups). The fix is workable â€” put those settings in `~/.npmrc` (user scope), which Nub does read. Closing the global/builtin gap is tracked; it is not yet shipped.

A minor cosmetic note: `npm_config_*` env vars correctly drive install behavior but are not reflected in Nub's config-display output (the display reads files only). The install is correct; only the readout omits env overrides.

### Install behavior

- **Lifecycle scripts / build trust.** Postinstall and other lifecycle scripts run under Nub's deny-by-default trust posture plus the gated default-trust floor described under [Install-script trust](#install-script-trust) â€” **not** npm's run-everything default. A curated, registry-resolved, age-vetted package builds automatically; everything else is skipped with `WARN_NUB_IGNORED_BUILD_SCRIPTS` until you `nub approve-builds`.
- **overrides.** The `package.json` `"overrides"` field is read and applied. The `"resolutions"` field (which npm itself ignores) is dropped with a warning â€” matching npm's effective behavior.
- **engines / os / cpu / libc.** Enforced. Platform-gated optional dependencies (e.g. `fsevents`) resolve and lock correctly, and the platform fields round-trip into `package-lock.json`.
- **workspaces.** The neutral `"workspaces"` field is honored.

### Unsupported npm features

- **No global/builtin `.npmrc`.** Settings that live only in `$PREFIX/etc/npmrc` or npm's builtin config are ignored (see above). Move them to `~/.npmrc`.
- **lockfileVersion 1** (npm â‰¤ 6) is rejected, not migrated. Run `npm install` once to upgrade the lock to v2/v3 first.
- **No drop-in `npm` CLI grammar.** npm-specific subcommands/flags and `npm-shrinkwrap.json` are not emulated; the CLI is pnpm-shaped.
- **A v2 lock becomes v3 on the first mutating install.** Preserved on no-op installs only.

## Bun

When Bun is the incumbent package manager, Nub installs against it: the `bun.lock` text lockfile round-trips unbroken, top-level `trustedDependencies` is honored for build-script trust, and `overrides` / `resolutions`, `patchedDependencies`, and workspaces + catalogs all resolve the way Bun resolves them. Nub does not write its own lockfile into a Bun project.

| `bun.lock` (text) | `bun.lockb` (binary) | `bunfig.toml` | `trustedDependencies` | `overrides` | `resolutions` | `patchedDependencies` | `catalog` | `workspaces` | `.npmrc` |
|:---:|:---:|:---:|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| âś“ read+write | âś— | âś— | âś“ | âś“ | âś“ | âś“ | âś“ | âś“ | âś“ |

(Both `overrides` and `resolutions` are honored; when both pin the same package, top-level `overrides` wins â€” Bun's own precedence. `patchedDependencies` patches apply at materialize time, and `workspaces` + catalogs resolve to local workspace packages.)


The legacy binary lockfile (`bun.lockb`, pre-1.2) is not supported. If a project has only a `bun.lockb` and no text `bun.lock`, Nub errors with an actionable hint rather than guessing or falling through to another format.

### `trustedDependencies`

Bun gates dependency build scripts behind a `trustedDependencies` allowlist in `package.json`; only listed packages run their `postinstall`/`install` scripts. Nub honors that array when Bun is the incumbent. A package not on the list is installed but its build scripts are skipped, with a warning; adding it to `trustedDependencies` lets the scripts run.

```jsonc
// package.json
{
  "dependencies": { "esbuild": "0.21.5" },
  "trustedDependencies": ["esbuild"]
}
```

```console
# captured: nub 0.0.44 â€” WITHOUT the package in trustedDependencies (bun.lock incumbent) â€” scripts skipped
$ nub install
dependencies:
+ esbuild@0.21.5

nub 0.0.44 Â· âś“ installed 2 packages in 33ms

# captured: nub 0.0.44 â€” WITH "esbuild" in trustedDependencies â€” scripts run (no warning)
$ nub install

+ esbuild@0.21.5

nub 0.0.44 Â· âś“ installed 2 packages in 35ms
```

How the default-trust floor interacts with Bun:

- **The curated default-trust floor is on, but inert over an existing `bun.lock`.** Nub's `defaultTrust` floor (the curated allowlist described under [Install-script trust](#install-script-trust)) is enabled by default. But its cooling-window gate needs per-package publish-time data, and a `bun.lock` carries none. Over an incumbent `bun.lock` the floor therefore fails closed: a curated package like `esbuild` is **not** auto-trusted (the A/B above shows it skipped with `WARN_NUB_IGNORED_BUILD_SCRIPTS`, not the `WARN_NUB_DEFAULT_TRUST_BUILDS` you'd see on a pnpm-incumbent or fresh-resolve install). So in a Bun project the effective trust model is exactly Bun's: **`trustedDependencies` is the allowlist; nothing else builds automatically.**
- **Bun's own `trustedDependencies` is exactly mirrored.** Listed packages build; unlisted ones don't.

One forward-looking note: the engine honors `trustedDependencies` only when the project's declared Bun major is below 10 (`bun@10+` ignores it). Bun is currently at 1.x and has not shipped a v10 or removed the field â€” this is a forward-compatibility guard modeled on pnpm@10's `onlyBuiltDependencies` rename, **not** an observed change in any released Bun. For every current Bun version, `trustedDependencies` is honored.

### `bunfig.toml` is not read

This is the one real config gap. Nub does not read `bunfig.toml` â€” neither the project file nor the global `~/.bunfig.toml`. Registry URLs, `cafile`, and `[install.scopes]` scoped-registry/auth declared *only* in `bunfig.toml` are not picked up.

In practice this is mitigated: Nub reads `.npmrc` (project and user), and Bun reads `.npmrc` too. Keep registry, auth tokens, and scoped-registry config in `.npmrc` and both tools agree. The gap bites only config that lives exclusively in `bunfig.toml` â€” there Nub falls back to the public npm registry and misses private-scope auth. (Verified: with Bun incumbent, `nub config get registry` returns the project `.npmrc` value.)

```ini
# .npmrc â€” honored by both nub and bun; prefer this over bunfig.toml for shared config
//registry.example.com/:_authToken=${NPM_TOKEN}
@myscope:registry=https://registry.example.com/
```

## Yarn

Yarn is supported **read-only**. Nub reads a project's `yarn.lock` â€” Classic (v1) and Berry (v2+) â€” to install and run it, but never writes `yarn.lock`. Treat a Yarn project as something Nub *consumes*, not something Nub maintains: keep using `yarn` for anything that changes the dependency graph. This is the narrowest compatibility tier of any package manager Nub recognizes.

| `yarn.lock` read | `yarn.lock` write | `.yarnrc.yml` / `.yarnrc` | `resolutions` | `packageManager` | `nodeLinker: pnp`/`pnpm` install | `dependenciesMeta.*.built` | `packageExtensions` | `supportedArchitectures` | `.npmrc` |
|:---:|:---:|:---:|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| âś“ v1+Berry | âś— refused | âś— | âś“ | âś“ | âś— hoisted | âś— | âś— | âś— | âś“ |

### Running a Yarn project

Run an existing Yarn project's install when `yarn.lock` already satisfies `package.json`:

```bash
nub install        # reads yarn.lock, links node_modules, runs lifecycle scripts â€” does NOT write yarn.lock
nub ci             # frozen install: yarn.lock is law; drift is a hard error
nub run <script>   # runs package.json scripts
nubx <bin>         # one-off binary execution
```

Nub resolves the dependency graph from `yarn.lock` (both Classic and Berry formats parse), links a hoisted `node_modules`, and runs scripts. The lockfile is read; it is never rewritten.

Top-level `resolutions` in `package.json` is honored â€” Nub applies it the way Yarn does. So is the `packageManager` field, which is what tells Nub the project is a Yarn project.

### `yarn.lock` is read-only

Any command that would change `yarn.lock` is refused with a message pointing you back to Yarn (captured against nub 0.0.44, `nub add lodash` in a Yarn project â€” the verb, reason, and remedy are interpolated):

```console
$ nub add lodash
Error: nub add: refusing to modify yarn.lock â€” adding a dependency re-resolves and rewrites yarn.lock
  yarn.lock write fidelity is unproven in the embedded engine, so commands
  that would rewrite it are blocked. Run it with yarn directly:
    yarn add lodash
```

Refused (run these with `yarn`):

```bash
nub add <pkg>       # â†’ yarn add
nub remove <pkg>    # â†’ yarn remove
nub update          # â†’ yarn upgrade
nub dedupe          # â†’ yarn dedupe
```

The gate also fires on an `install` / `ci` that *would* rewrite the lockfile rather than just read it:

- a project that declares Yarn but has no `yarn.lock` yet (a fresh install would create one),
- `nub install --force`, `--no-frozen-lockfile`, or `--lockfile-only`,
- a `yarn.lock` that no longer satisfies `package.json` (the install would have to re-resolve).

Captured (declared-yarn project with no `yarn.lock`):

```console
$ nub install
Error: nub install: refusing to modify yarn.lock â€” this project declares yarn but has no yarn.lock yet â€” a fresh install would create it
  yarn.lock write fidelity is unproven in the embedded engine, so commands
  that would rewrite it are blocked. Run it with yarn directly:
    yarn install
```

In all of these, run `yarn install` instead. The reason is honesty about fidelity: Nub's embedded engine parses `yarn.lock` reliably but its Yarn-format *writer* is unproven, so Nub blocks the write rather than risk a corrupt or churned lockfile.

There is no path that converts *to* `yarn.lock`. `nub import` produces Nub's own `pnpm-lock.yaml`, not a `yarn.lock`:

```console
$ nub import        # in a yarn project
Imported 2 packages from yarn.lock to pnpm-lock.yaml
```

<Callout title="nub pm use yarn is a different surface">
The `nub pm use yarn` command is the [meta-manager](/docs/pm) provisioning command, and it *does* work: it fetches and pins Yarn via the `packageManager` field. It does not produce a `yarn.lock` or convert your install engine; it just provisions the Yarn binary.
</Callout>

### Yarn config files are not read

This is the headline gap. Nub reads **`.npmrc` only** for registry, auth, scopes, CA, and proxy settings. It does not read Yarn's config files at all:

- `.yarnrc.yml` (Berry) â€” project **and** global (`~/.yarnrc.yml`) â€” is not parsed,
- classic `.yarnrc` (Yarn 1) is not parsed.

In practice, none of these Yarn-config keys take effect under Nub:

```yaml
# .yarnrc.yml â€” ALL IGNORED by nub
npmRegistryServer: https://registry.internal/   # ignored â€” nub reads .npmrc `registry=`
npmScopes:                                       # ignored â€” per-scope registries unread
  myorg:
    npmRegistryServer: https://npm.myorg.dev/
    npmAuthToken: ...                            # ignored â€” private-dep auth NOT applied
npmAuthToken: ...                                # ignored
httpsCaFilePath: ./ca.pem                        # ignored
httpProxy: http://proxy:8080                     # ignored
```

If your private-registry auth or custom registry lives in `.yarnrc.yml` (which for Berry is the *only* place it lives), Nub will not pick it up and private installs fail with 401/403. The workaround today is to put the equivalent settings in `.npmrc`, which Nub does read.

### PnP and pnpm node-linkers are not produced

Nub-the-package-manager only produces a **hoisted `node_modules`** for a Yarn project. Because Nub does not read `.yarnrc.yml` at all, a `nodeLinker` setting there is never seen â€” and Nub's default layout for a Yarn-incumbent project is hoisted regardless:

```yaml
# .yarnrc.yml
nodeLinker: pnp     # nub installs hoisted node_modules instead
nodeLinker: pnpm    # also produces hoisted
```

Running `nub install` in a `nodeLinker: pnp` project does **not** produce a PnP install â€” it builds a hoisted `node_modules`, a different on-disk layout than Yarn would produce.

Keep this distinction crisp: **Nub-the-runtime fully supports Plug'n'Play.** If a project was *already* installed by Yarn in PnP mode, Nub can run it â€” `nub <file>`, `nub run`, and `nubx` honor `.pnp.cjs` across all supported Node versions. What Nub cannot do is *produce* a PnP install itself. So the supported workflow for a PnP project is: install with `yarn`, run with `nub`.

### Unsupported Yarn manifest fields

Several Yarn `package.json` and `.yarnrc.yml` fields have no effect under Nub:

- `dependenciesMeta.<pkg>.built` â€” Yarn's per-package build gate is not read; Nub uses its own build-script approval (`nub approve-builds`),
- `packageExtensions` â€” Yarn's resolution patches are not applied,
- `supportedArchitectures` â€” Yarn's platform/arch download controls are not read.

## Store and disk layout

Regardless of the incumbent, Nub installs through a global content-addressed store and links into an isolated virtual store â€” the same scheme aube uses, under Nub's own directory names.

### Global content store

Package files are deduplicated by content hash in a global store at `$XDG_DATA_HOME/nub/store/v1/` (defaulting to `~/.local/share/nub/store/v1/`). Every install imports from it, so the same package version on disk once is shared across every project. Run `nub store path` to print the resolved location. Files materialize into `node_modules` by reflink (APFS/btrfs), hardlink (ext4), or copy â€” whichever the filesystem supports â€” so a populated tree costs little extra disk.

### Virtual store

The default `node_modules` layout is **isolated**: direct dependencies sit at the top level, and transitive packages link into a per-project virtual store so phantom dependencies fail instead of resolving by accident. Nub's virtual store is `node_modules/.nub/` (where pnpm uses `node_modules/.pnpm/`) â€” equivalent in shape, but not byte-shared, so alternating tools relinks the tree. The `--node-linker hoisted` flag produces a flat, npm-style layout instead; npm, Yarn, and Bun incumbents default to `hoisted`, while pnpm and nub-identity projects keep `isolated`.

### Offline and the global virtual store

A global virtual store reuses fully materialized package directories across projects (on by default for local installs, off under CI). A handful of build-heavy frameworks opt out of it by default (`@sveltejs/kit`, `next`, `nuxt`, `parcel`). Installs run offline when the store already holds every needed package; `--offline` forces it and `--prefer-offline` tries the cache first. See [aube's store docs](https://aube.jdx.dev/package-manager/global-virtual-store) for the full model.
