Import a config or data file directly — no `npm install yaml` / `json5`, no `fs.readFile` + `parse` plumbing. The default export is the parsed value; for an object, its top-level keys are also available as named exports. Same shape as `import data from "./data.json"`, which Node has supported since 2017. Loaders work on every Node version Nub supports (18.19+).

```ts
import config from "./config.yaml";          // parsed object
import flags from "./feature.jsonc";         // parsed object (comments stripped)
import pkg from "./Cargo.toml";              // parsed object
import schema from "./schema.json5";         // parsed object (JSON5 superset)
import prompt from "./prompt.txt";           // string
import { host, port } from "./config.yaml";  // named exports mirror top-level keys
```

## Extensions

Loaders are keyed on file extension:

- `.jsonc` — JSON with comments
- `.json5` — the JSON5 superset
- `.toml`
- `.yaml` / `.yml`
- `.txt` — loaded as a string, no parsing

The `.json` extension is intentionally not in this set — it is Node-native (`resolveJsonModule`), so Nub leaves it to Node.

## Default and named exports

The default export is the parsed value. When the parsed value is an object, each top-level key is also emitted as a named export, so you can destructure at the import site.

```yaml
# config.yaml
host: localhost
port: 5432
```

```ts
import config from "./config.yaml";          // { host: "localhost", port: 5432 }
import { host, port } from "./config.yaml";  // "localhost", 5432
```

Only keys that are valid JavaScript binding identifiers become named exports. A key that is a reserved word (`package`, `class`, `default`, …) or not an identifier stays reachable through the default export — so a `Cargo.toml` with a `[package]` table imports cleanly, with `package` available as `pkg.package`.
