# AGENTS.md ‚Äî agent orientation for the Nub repo

This file is the entry point for AI coding agents working in this repository. It mirrors the [`AGENTS.md` convention](https://agents.md) used by other AI tools. [`CLAUDE.md`](CLAUDE.md) is a symlink to this file ‚Äî Claude Code and the `AGENTS.md` convention read identical content.

## Core design positions ‚Äî ALWAYS HOLD THESE IN CONTEXT (read before any CLI/PM/runtime change)

These are the load-bearing, always-true architectural premises of nub. They are easy to violate by reflex (e.g. adding an npm flag, augmenting hijacked `node`), so keep them resident ‚Äî do NOT design or dispatch against this surface without them. When a change touches the CLI grammar, the PM, or the runtime, FIRST re-state the relevant position here and design to it.

- **What nub is:** a **Rust CLI that AUGMENTS the user's installed Node** via Node's own extension surfaces (`--import`/`--require` preload, `module.registerHooks`, env, N-API addons, V8-flag injection). It is **NOT a fork**, ships no patched Node, embeds no libnode. The test for any augmentation: "would a user on plain Node + the corresponding `module.register()`/`--import`/npm-addon get the same result?"
- **The PM CLI frontend is pnpm-compatible ‚Äî ONLY pnpm.** The package-manager command grammar mirrors **pnpm's, exclusively** ‚Äî NOT npm, NOT yarn, NOT bun. When adding/checking a flag/alias/positional on a PM verb, the ONLY question is *"does real pnpm accept it?"* ‚Äî never "does npm?". Do not add npm-isms (`--omit`, `--no-save`, `-S`/`--save`, npm's `-w <name>`/`--workspaces` selectors). Gotcha: pnpm `-w` = `--workspace-root` (boolean); member selection is `--filter`, recursion is `-r`. (This premise was violated once by grounding a compat audit against npm+pnpm ‚Äî never again.)
  - **The DELIBERATE exceptions to pnpm-compat (decided 2026-06-20):** (1) the **top-level file-run command** (`nub <file>`) is nub's own surface, not pnpm's; (2) **`nub upgrade`** is nub's SELF-UPDATE (not pnpm's `upgrade`‚Üí`update` alias) ‚Äî keep as-is, `nub upgrade <pkg>` does not route to `update`; (3) **NO implicit script shortcuts, EVER** ‚Äî nub ALWAYS requires explicit `nub run <script>`; it never accepts `nub test`/`nub start`/`nub t`/etc. as run-shortcuts even though pnpm does. These three are the known, intentional divergences; everything else mirrors pnpm.
- **Lockfile-compatibility is a SEPARATE axis from CLI-compat.** nub is lockfile-compatible with whatever the project already uses (npm/pnpm/bun round-trip, yarn read-only) ‚Äî it does not impose its own lockfile format. "pnpm-only" governs the CLI grammar; the lockfile axis is multi-PM.
- **The PM is the vendored `aube` engine, embedded as a Rust LIBRARY** (path dep `vendor/aube`, called in-process ‚Äî no aube subprocess). Fork = `nubjs/aube` branch `nub-fork`; upstream = `jdx/aube`. The pin is a `nub-fork` commit; **push `nub-fork` before bumping the pin**; sync upstream by MERGE never rebase. Fork changes are default-preserving (standalone aube byte-for-byte unchanged; the nub embedder opts in) or a justified latent-bug fix.
- **Config identity model:** a **nub-identity** project consumes ONLY neutral, cross-tool config (`overrides`, `resolutions`, `catalog`, `workspaces`) ‚Äî never another PM's branded fields/files. A **compat** project (incumbent npm/pnpm/yarn/bun) is MIRRORED faithfully, branded fields included. pnpm-NAMED files are never read unless pnpm is the incumbent.
- **Compat targets are PER-MAJOR-VERSION of each package manager (decided 2026-06-20).** Each major version is its OWN compat target because config homes, lockfile formats, and behavior shift across majors ‚Äî the canonical example: **pnpm 10 reads/writes scalar settings from `.npmrc`; pnpm 11 moved them to `pnpm-workspace.yaml`/global `config.yaml`.** So treat **npm**, **pnpm 9**, **pnpm 10**, **pnpm 11**, **yarn 1 (classic)**, **yarn 2/3/4 (berry ‚Äî lump ONLY if byte-identical behavior)**, and **bun** as distinct targets. When MIRRORING an incumbent (config-writes, lockfile, behavior), nub DETECTS the incumbent's specific major version (`packageManager` field ‚Üí `devEngines` ‚Üí installed-PM `--version` ‚Üí lockfile-version signal) and mirrors THAT major ‚Äî never assumes "latest." When the version is unknown but the PM is known, default to the **dominant / most-compatible** major (e.g. pnpm ‚Üí the v10 `.npmrc` model, the larger install base + the lowest-common-denominator that even v11 still reads for auth). Architect this as a version‚Üímodel MAP, not a two-branch special-case, so a new major slots in cleanly. (This governs the multi-PM compat-mirroring axis; the CLI-grammar axis stays pnpm-only and tracks pnpm's current stable.) **WITHIN REASON:** split a major into its own target ONLY where its behavior MATERIALLY differs AND that version is in real-world use; LUMP majors that are identical/near-identical; do NOT engineer distinct targets for versions nobody runs or speculative future majors. The map is the architecture so new targets slot in cheaply ‚Äî not a mandate to pre-populate every historical/hypothetical major. Cover what people actually use (today: pnpm 9/10/11, npm, yarn-classic, yarn-berry, bun), pragmatically.
- **Brand boundary (public surfaces only):** no public `globalThis.nub`, no `nub:*` module namespace, no `@nub/*` scope, no `"nub"` user-authored config field, no documented `NUB_*` user knob. Internal mechanism is EXEMPT ‚Äî internal `NUB_*`/`__NUB_*` plumbing vars, error codes, cache/dir names may carry the brand.
- **`node` PATH-hijack contract (under active design ‚Äî see [[node-hijack-compat-design]]):** when something invokes `node` via nub's shim it should RESPECT node/PATH semantics (the hijack's legitimate job is version resolution/pinning); augmentation belongs to the `nub`/`nubx` entrypoints. `--node` is the explicit zero-augmentation escape hatch on `run`/`nubx`.
- **v0 verb surface:** `nub <file>` ¬∑ `run` ¬∑ `watch` ¬∑ `nubx`/`dlx` ¬∑ `upgrade` ¬∑ `nub node` (version management) ¬∑ the pnpm-compatible PM (`install`/`i`/`ci` + `add`/`remove`/`update`/`import`/`dedupe`/`pm ‚Ä¶`). No `nub node <file>` passthrough; `--node` flag is live, and a truthy `NODE_COMPAT` env var is the tree-wide augmentation opt-out (= `--node`, persistent).

## Default to a PR flow (from a git WORKTREE) ‚Äî never branch/reset/stash the SHARED `main` tree

**Substantive nub-repo work lands via a PULL REQUEST opened from an isolated git WORKTREE; the SHARED working tree always stays on `main` and is NEVER branched, reset, or stashed.** (This REVERSES the long-standing "all work on `main`, no branches/worktrees" rule ‚Äî but ONLY for the nub repo's own work; `vendor/aube` keeps its existing `nub-fork` workflow, below.) The PR flow buys visibility, trackability, and a shareable link to announce a feature landing. The old rule existed because branching/resetting/stashing a SHARED working tree clobbers parallel agents and orphans their WIP ‚Äî the PR flow MUST NOT reintroduce that, so the isolation is structural: each landing agent works in its OWN worktree, and the shared tree is left alone.

**The proven worktree workflow (copy-pasteable; validated end-to-end 2026-06-20, submodule + build + push + PR all clean).** Steps 1‚Äì3 are encoded in `scripts/new-worktree.ts` ‚Äî `nub scripts/new-worktree.ts <slug>` (or `node scripts/new-worktree.ts <slug>`) creates the worktree off `origin/main`, inits the `vendor/aube` submodule, applies `.worktreeinclude`, and prints the `CARGO_TARGET_DIR` to export. Prefer it over hand-rolling the recipe (full detail: the `worktree` skill, `.claude/skills/worktree/SKILL.md`). The manual equivalent:

```sh
# 1. From the shared main tree ‚Äî create a worktree on a new branch. This checks out ONLY tracked
#    source (NOT target/ .repos/ node_modules) and shares the .git object store ‚Äî no re-clone, no
#    re-fetch. The shared main tree is UNTOUCHED and stays on main.
git worktree add /tmp/nub-wt-<slug> -b <branch-slug> origin/main

# 2. The aube submodule is NOT auto-populated in a new worktree ‚Äî init it explicitly (re-clones into
#    the worktree's own vendor/aube; required for any build that touches the PM engine).
git -C /tmp/nub-wt-<slug> submodule update --init vendor/aube

# 3. Give the worktree its OWN target dir so a sibling's build can't contaminate it.
cd /tmp/nub-wt-<slug> && export CARGO_TARGET_DIR=/tmp/nub-wt-<slug>-target
cargo check -p nub-cli                       # or the scoped test for what changed

# 4. Commit your work, push the branch, open the PR ‚Äî then REPORT the URL. Do NOT merge your own PR.
git add <your files> && git commit -m "<subject>"
git push -u origin <branch-slug>
gh pr create --title "<title>" --body "<body>"   # report the returned URL to the orchestrator

# 5. After the orchestrator merges, clean up (orchestrator or agent):
git worktree remove /tmp/nub-wt-<slug> --force && rm -rf /tmp/nub-wt-<slug>-target

# 6. Eagerly pull the SHARED tree so it tracks origin (see the eagerly-pull rule below).
git -C <shared-tree> pull --ff-only
```

The rules around it:

- **NEVER `git checkout -b` / `git reset` / `git checkout <ref>` / `git stash` / `git branch` on the SHARED working tree.** Those bypass the file-tool clobber safety and wipe concurrent agents' uncommitted WIP. The shared tree stays on `main`; all branching happens in a worktree.
- **The orchestrator reviews + merges the PR.** A landing agent OPENS the PR and reports the URL; it does NOT merge its own PR. Merge once green + reviewed.
- **EXCEPTIONS that still commit DIRECTLY to `main`** (no PR): the orchestrator's own control-surface edits (`.fray/` threads, the fray skill, hooks, `AGENTS.md`, memory); trivial doc/typo fixes; and the version-bump + tag release commits (the release flow is its own tag-triggered thing ‚Äî see [Releasing](#releasing)). Use judgment: a PR is for substantive code/feature/behavior changes where a reviewable, announceable diff adds value.
- **`vendor/aube` changes do NOT use this PR flow** ‚Äî they keep the `nub-fork` workflow (commits on `nub-fork`, push before pin bump, contribution PRs to `jdx/aube`; see the aube section below).
- **Full `git clone --depth 1 "file://$PWD"` + own `CARGO_TARGET_DIR` is reserved for the fully-isolated trustworthy-build case** (aube fork patches whose deliverable is "passes tests + preserves upstream behavior"). Worktrees are the default for ordinary PR-landing work; a clone is the heavier tool for when an agent needs a tree no sibling can touch at all.
- **Commit small and often within your worktree; never stash.** In an interactive foreground session, do not push/PR unless the user asks; leave verified edits committed in the worktree (or uncommitted on the shared tree for tiny exception-class edits) and report status. Committed work cannot be clobbered; if you need to preserve WIP, *commit* it.
- **EAGERLY PULL the shared tree ‚Äî after ANY merge or push to origin, `git -C <shared-tree> pull --ff-only` (the drift fix).** Every landing goes worktree ‚Üí push ‚Üí squash-merge, and nothing pulls the SHARED checkout back, so it silently falls behind `origin/main` (it once drifted 21 commits, and the running session kept loading stale `.claude/` hooks). After merging any PR or pushing to origin, fast-forward the shared tree. The merge-queue drain is the natural automatic home for this ‚Äî it already auto-merges and flips threads done, so it should also `pull --ff-only` the shared tree after each merge. NOTE: this keeps the FILES current; loaded hooks still need a session restart (the fray plugin makes hook-drift moot ‚Äî hooks load from the plugin cache, not host `.claude/`). **Corollary: do NOT commit directly in the shared tree's checkout ‚Äî even control-surface/release commits go via a worktree push, so the shared tree stays clean and always fast-forwardable.** Direct shared-tree commits are what make it *diverge* (not merely fall behind), which breaks `pull --ff-only`.
- **PUSH-THEN-EXIT for a BACKGROUND/dispatched landing agent ‚Äî never wait on CI in-process.** A backgrounded sub-agent doing landing work pushes its branch the instant a commit exists (safe pre-CI ‚Äî CI runs on the pushed commit), BEFORE any verification wait or rest, then reports `"pushed <sha>, awaiting CI"` and EXITS. It NEVER arms a CI watcher / poll loop / `sleep`-on-CI ‚Äî those are bounded by the harness's background-command kill and strand the work; the orchestrator's heartbeat owns merge-on-green via the durable `.fray/merge-queue.jsonl` (see the fray skill). The `fray-push-before-rest` SubagentStop hook enforces this ‚Äî it refuses to let a dispatched agent rest holding a committed-but-unpushed change in a PR worktree. (This is for the autonomous/background flow; it does not override the interactive-foreground "don't push unless asked" rule above.)

A busy `main` with several open PRs and in-progress worktrees is expected and fine.

### Worktree dev loop ‚Äî fast incremental builds + the real build/test incantations

Full runbook + crate map: the `nub-dev` skill (`.claude/skills/nub-dev/SKILL.md`). The essentials, so a worktree iterates fast (measured 2026-06-20):

- **Set up the worktree, init the submodule, give it its OWN target dir** (per the proven workflow above): `git worktree add ‚Ä¶ -b <slug> origin/main`, then `git -C ‚Ä¶ submodule update --init vendor/aube`, then `export CARGO_TARGET_DIR=/tmp/<slug>-target`. The per-worktree target dir is what keeps a sibling's build from contaminating yours ‚Äî keep it.
- **Iterate with the `fast` profile, NEVER `release`.** `cargo build -p nub-cli --profile fast` builds the dev CLI to `target/fast/nub`. Cold ‚âà 3 min; every subsequent rebuild in the same target dir ‚âà 5s via cargo's native incremental. The `release` profile's `lto=thin` + `codegen-units=1` makes a cold build ‚âà 15 min and re-LTOs the whole binary on every change ‚Äî do not use it to iterate. For the full dev binary + N-API addon on PATH, `make install-dev` (symlinks `nub-dev`/`nubx-dev` ‚Üí `target/fast/nub`); for the addon alone, `make addon-fast`. There is no `nub build` command.
- **Pay the cold build ONCE per worktree, then keep the target dir stable.** Do NOT clean, move, re-clone, or seed the target dir between iterations ‚Äî cargo's incremental fingerprints are keyed to the absolute target path, so any of those forces a full ~3-min rebuild. The fast path is one stable target dir per worktree for the whole session.
- **A shared cross-worktree compile cache does NOT help here.** sccache was measured against this Rust workspace and delivered a 0% Rust cache-hit rate across worktrees (rustc embeds per-target-dir paths in its cache keys; `--remap-path-prefix` does not fix it) ‚Äî no speedup over a cold build. Do not reach for it; the `fast` profile + a stable per-worktree target dir is the whole answer.
- **Tests:** a specific file is `cargo test -p nub-cli --test <file_stem>` (e.g. `--test pm_verbs`); a single test by name substring is `cargo test -p nub-cli <substring>` (`-- --exact <full::path>` pins one). Before pushing, run the [pre-push local verification loop](#implementation-quality-discipline) ‚Äî the *exact* CI gates are `cargo clippy --all-targets --all-features -- -D warnings` and `cargo fmt --check`.
- **Ad-hoc e2e probing** (build the dev `nub`, run your subcommand against a `/tmp` fixture, verify the effect, probe variants) is a valid way to verify new functionality ‚Äî the `ad-hoc-test` skill (`.claude/skills/ad-hoc-test/SKILL.md`) has the loop. Promote durable checks into the suite per the pre-push loop.

## The repo is PUBLIC ‚Äî never commit internal discussion (HIGH PRIORITY)

`github.com/nubjs/nub` is a **public open-source repo with real stars**. Everything committed ‚Äî code, comments, commit messages, `tests/**` READMEs, every tracked markdown file ‚Äî is world-readable forever, and **survives in git history even after you delete it**. So: **never put internal product / marketing / competitive discussion or considerations into a tracked file.** Banned from anything git tracks:

- **Benchmark-presentation strategy** ‚Äî which competitors or numbers to show vs. omit, "include X only where we're competitive," competitive-band thresholds, anything about how results are framed for publication.
- **Candid hedges that read as gaming** ‚Äî "load-bearing for honesty," "this is the headline," framing a default (e.g. GVS auto-disable for next/nuxt/parcel) as something to be sheepish about, or any sentence a competitor/skeptic could screenshot as "they juice their benchmarks."
- **Private attributions / quotes / first-person strategy** ‚Äî by-name attributions, internal quotes, "our number," "don't publish X," "reads as cherry-picking." Public copy states what the code DOES, factually and neutrally ‚Äî never *why a number was framed a certain way*.

**Commit messages are professional, factual, and neutral ‚Äî NEVER braggy, competitive, or editorializing.** A commit message states WHAT CHANGED, not how great it is or how it stacks up against a competitor. Banned: competitive bragging ("nub clearly beats bun", "crushes pnpm", "fastest"), superlatives/hype, and any framing a skeptic could screenshot. Write `site: update warm-install bench to 36-run numbers`, NOT `site: ‚Ä¶ (nub clearly beats bun)`. Benchmark commits name the change (new run, new sample size, updated bars) ‚Äî never the verdict. This applies to every commit subject + body; the rule is the same as for any tracked file (above), and commit messages are world-readable forever and cannot be cleanly scrubbed once pushed.

All internal deliberation lives in the **LOCAL-ONLY, gitignored** surfaces ‚Äî `.fray/`, `epics/`, `wiki/` (none are published). Keep strategy, competitive analysis, and decision rationale there, never in a tracked source/doc/test file. When in doubt, leave it out of the commit.

## Model tiering for sub-agents (HIGH PRIORITY ‚Äî apply to EVERY dispatch)

Two levers: WHICH model, and (where exposed) the `effort` parameter. Cost ladder cheapest ‚Üí priciest: **Haiku 4.5 < Sonnet 4.6 < Opus 4.8 < Fable 5** (Haiku‚ÜíFable ‚âà 10√ó in/out). This applies to every sub-agent dispatch ‚Äî single one-off calls as much as multi-agent workflows. Set `model:` per dispatch by cognitive load (JUDGMENT REQUIRED), not by task type:

**EFFORT IS NOT PER-DISPATCH SETTABLE ON THE `Agent` TOOL ‚Äî quality rides on the MODEL tier (HIGH PRIORITY; corrected 2026-06-19 after the orchestrator over-claimed it was setting workers to xhigh).** The background-dispatch `Agent` tool exposes `model` but **no `effort` param** ‚Äî so for fray dynamic dispatch (individually-dispatched background sub-agents, the normal mode) you CANNOT set a worker to high/xhigh, and must not claim to. It is UNKNOWN whether a background worker inherits the orchestrator session's `/effort` (so dropping the orchestrator chat to `medium` to economize narration may or may not pull workers down ‚Äî do not rely on either). The ONLY effort knob that exists is **Workflow's `agent({effort})`** ‚Äî use it there, set it `high`/`xhigh` for the hard stages. Everywhere else, worker quality is protected by **(1) MODEL TIER and (2) a rigorous self-contained prompt** that demands the verification/adversarial-rigor you'd want from high effort. THE RULE: **a worker doing judgment / engineering / design / review work ALWAYS gets Opus (or Fable) ‚Äî never economize it down to Sonnet/Haiku because the orchestrator chat is on `medium`.** The "Effort `xhigh`/`high`/`medium`" defaults below apply where an effort knob actually exists (Workflow `agent()`); on the `Agent` tool they are the *intended rigor*, achieved via model + prompt, not a settable parameter.

- **Haiku** ‚Äî fully-scripted mechanical only: run THESE exact commands, harvest THIS output, trace THIS path, where every decision is pre-made by you. Cannot self-steer ‚Äî do not give Haiku an open-ended investigation and expect a reliable verdict. Give it a script, not a question. Caveats: 200K context (vs 1M for the others) ‚Äî route >~150K-token jobs to Sonnet; and Haiku takes NO `effort`/adaptive-thinking param (passing them 400s).
- **Sonnet** ‚Äî the supporting cast: differential probes requiring light judgment to recognize a divergence, test scaffolding, gates/settle/CI-watching, doc updates, mechanical-but-not-trivial edits. Effort `medium` default. (Trivial fix-loop iterations can stay here to save the Opus premium.)
- **Opus** ‚Äî THE DEFAULT for actual software engineering: diagnosis and writing/fixing the code that LANDS, architecture & design, adversarial review, gnarly debugging. The ~1.67√ó premium over Sonnet is cheap because code is low-volume vs the Haiku-dominated grunt, and it is exactly where the quality bar lives. Effort `xhigh` for coding/agentic.
- **Fable** ‚Äî only the very hardest synthesis/judgment (priciest; use sparingly). Effort `high` default.
- **Orchestrator (you)** ‚Äî push real work DOWN to the cheapest capable tier, then VERIFY what the doing-agents produce against the quality bar. If you are on Fable (the expensive flagship), keep your own footprint lean: coordinate + delegate + spot-check, do not do heavy lifting yourself.

Pattern: *haiku gathers & packages ‚Üí opus does the real engineering ‚Üí sonnet handles the supporting cast ‚Üí you verify.* Sub-agent prompts must be SELF-CONTAINED (a model switch starts a fresh cache ‚Äî the orchestrator's cached prefix does not carry over). Re-verify cheap-tier load-bearing claims yourself ‚Äî a Haiku "this is a security bug" is a *lead, not a fact*; confirm it against code or a foreground experiment before acting on it. Cheap-tier agents should emit an explicit escalation marker on low confidence so the orchestrator re-routes up a tier. Long Opus/Fable loops: use Task Budgets to cap spend; big Haiku fan-outs: consider the Batch API. Fuller detail in the `claude-api` skill. Do NOT spin up all-one-model workflows.

**Never `cp -r` the repo to isolate work** (a benchmark, a throwaway build, a sandbox). The tree carries `target/`, `.repos/`, and `node_modules` ‚Äî *tens of GB* ‚Äî so a recursive copy fills the disk. Isolate with `git clone --depth 1 "file://$PWD" <dest>` (tracked files only; add `--recurse-submodules` for `vendor/aube`) or `git archive`, and give any isolated build its own `CARGO_TARGET_DIR`. (A recursive copy of the multi-GB tree fills the disk.)

**Detecting a GitHub fork ‚Äî check `.fork`, never an empty `.parent`.** `gh repo view <owner>/<repo> --json parent` can return a null/empty `parent` even for a genuine fork (that field isn't reliably populated through `repo view`). The authoritative signal is the boolean `.fork`. Use the REST API: `gh api repos/<owner>/<repo> --jq '{fork:.fork, parent:.parent.full_name, source:.source.full_name}'` ‚Äî `.fork == true` means it's a fork, `.parent.full_name` is the immediate upstream, `.source.full_name` is the root. **Never conclude "not a fork" from an empty `parent` alone**, and never block a cross-repo PR on that (an empty `parent` does not mean the repo isn't a fork).

## Workflow lifecycle ‚Äî never kill mid-edit; run queues as a continuous series (HIGH PRIORITY)

**NEVER `TaskStop` a workflow while it may be mid-file-edit.** Killing an agent mid-edit orphans uncommitted WIP ‚Äî the exact stranded-work failure this repo's top rule warns about, and a mistake made repeatedly. Only stop at a SAFE point: the active agent has committed, or you are between phases. To shed load, prefer letting the running agent reach its commit over killing it. If you genuinely must stop mid-flight, the FIRST thing you do afterward is `git status` and **commit any buildable WIP (commit, don't stash)** ‚Äî including inside `vendor/aube` + the pin ‚Äî so nothing is lost. Workflows resume from committed state (`resumeFromRunId`), so the correct way to "pause" is to wait for a stopping point, not to kill a live editor.

**Act on clear next actions, not just observations.** When the necessary next action is clear and within existing safety constraints, do it immediately: apply the fix, run the verification, dispatch or steer the right child, or update the tracker. Treat an outcome-shaped user ask as authorization to proceed through the safe implementation/verification loop; ask only when a real human-owned decision blocks progress. P0/blockers/known required fixes are not next steps to merely describe ‚Äî launch or steer the work now, then report what happened. Keep following outstanding threads until they are done, blocked, or explicitly deprioritized.

**NEVER block the orchestrator foreground on long-running work.** The main loop must stay responsive: do not run polling loops, CI/deploy watches, builds, tests, benchmarks, dev servers, git/Vercel/GitHub polling, sleeps, browser sessions, or any command expected to take more than a short moment as a foreground tool call. Use a child agent, an existing background/notify mechanism, or a detached background command that writes to a named log path, then report the log path. Foreground tool calls are for quick state checks only; the orchestrator returns control or reports progress immediately, then reconciles/checks completion later.

**Close GitHub issues with a brief factual comment via `gh`.** Whenever closing an issue, use the GitHub CLI close path with a comment, e.g. `gh issue close <number> --comment <text>`. Keep the comment as short as possible while stating what fixed the issue, or why no code fix is needed. Do not close silently.

**Never respond to automated review bots / CI bots / dependabot / code-review bots as if they were human.** No "thanks", no niceties, no conversational replies. For an automated bot comment, either address the substantive finding with a terse factual note (or a code change) or don't reply at all. Conversational tone is for humans only.

**Get sign-off before launching an OPINIONATED workflow ‚Äî one that bakes in a design decision.** Classify every workflow before you launch it. A BROAD INVESTIGATION (profile, explore, diagnose, gather ‚Äî commits to no design and lands nothing consequential) is safe to launch autonomously. An OPINIONATED TASK / FIX / IMPL (prescribes a specific solution, or lands changes that touch an important design consideration ‚Äî a default, a security posture, a product behavior, an API/config/env surface, an architecture) requires the user's guidance FIRST: flesh the design out together, and be explicit about whether the work is investigation-scope or opinionated-implementation-scope, before any agent is told to build it. Separately, a workflow prompt must NEVER empower a sub-agent to autonomously land a change to a default / security posture / product decision the user owns ‚Äî those are recommend-only until the user signs off; the agent profiles and proposes, the user decides. An investigation prompt must not quietly carry a phase that prescribes-and-lands a design decision; the profiling is fine, the decision is a conversation first.

**Run a multi-item queue as a CONTINUOUS SERIES ‚Äî one at a time, but never stalled.** Parallel fan-out burns the quota fast (three concurrent multi-agent workflows can exhaust even Max 20√ó in ~an hour), so run ONE workflow at a time ‚Äî but the instant it finishes, launch/resume the next, driven by each completion notification, with a `ScheduleWakeup` heartbeat as the backup that catches stalls and survives quota resets. The failure to avoid is the series stalling after one item (it looks done; it isn't). Record the ordered sequence durably in the epic so any re-invocation ‚Äî completion or heartbeat ‚Äî can read "what's next" and continue with no further user prompting.

**The serialization rule is about BUDGET, not about clobbering ‚Äî and the governor is WITHIN-workflow:**

The thing that burns the budget is **ONE workflow fanning out 20+ parallel Opus agents at once.** That is the failure mode to never create. The fix is internal serialization, NOT avoiding concurrency:

1. **WITHIN a workflow ‚Äî exactly ONE sub-agent runs at a time. Absolute, non-negotiable.** A workflow (the `Workflow` tool, or an Interactive-Workflow effort) dispatches its sub-agents strictly one at a time: dispatch one, wait, verify, then the next ‚Äî a serial waterfall with synthesis and staging between steps. NEVER `parallel()`/`pipeline()` fan-out inside one workflow. This single-agent-at-a-time discipline is the **budget governor**: it caps a workflow's heavy-agent draw at one. (It also keeps the human able to follow + intervene at every step.)

2. **ACROSS workflows ‚Äî several MAY run concurrently. This is FINE and not gated.** Because each workflow is internally serial (governor above), running multiple internally-serial workflows side by side draws only a bounded number of concurrent heavy agents. Concurrent workflows are allowed by default. **There is NO fixed heavy-lane ceiling** (the old "‚âà5 heavy lanes" cap is removed ‚Äî fan out as wide as the work genuinely parallelizes); the only real governor is **429 ‚Üí back off**. The ONLY thing that is banned is the within-workflow fan-out in (1).

**Codebase-EDITING work parallelizes too ‚Äî the clobbering fear was overblown (corrected 2026-06-12).** The file tools enforce read-before-write, reject stale edits, and write atomically, so concurrent edits to one tree do NOT silently clobber and a concurrent build reads a consistent old-or-new file. Do not blanket-serialize codebase work. Three NARROW guards are the whole constraint: (1) **destructive raw git ops** (`reset`/`checkout`/`stash`) on a shared tree bypass the file-tool safety and WILL wipe a sibling's WIP ‚Äî forbid them in parallel-agent prompts (stage only own files, commit fast); (2) an agent whose DELIVERABLE is a *trustworthy build/test result* (a benchmark, or a `vendor/aube` patch proving it passes tests + preserves upstream behavior) needs its OWN isolated tree ‚Äî give it a `git clone --depth 1 file://‚Ä¶` + its own `CARGO_TARGET_DIR` so a sibling's edits can't contaminate its result; that is HOW you parallelize landing work, not a reason to serialize it; (3) serialize only the final **assembly/commit onto the shared branch** (cherry-pick/apply the isolated patches onto `nub-fork`, bump the pin once, integrated-test once).

And tier EVERY agent by the model-selection guidelines (judgment / the-fix-that-lands ‚Üí Opus; supporting cast / mechanical-but-not-trivial ‚Üí Sonnet; fully-scripted harvest ‚Üí Haiku) ‚Äî choose per agent, every time.

## Git & GitHub maintainer hygiene (HIGH PRIORITY)

nubjs/nub is a public open-source project with real reporters and contributors watching. Maintainer responsiveness on issues and PRs is a visible signal of the project's health, so these are not optional courtesies ‚Äî they are mandatory hygiene. The end-to-end issue playbook lives in the `address-issue` skill (`.claude/skills/address-issue/SKILL.md`); the GitHub-comment tone + the durable conventions below are also captured in the cross-project guide [`.github/prose-style.md`](.github/prose-style.md). Every comment here follows the project's existing tone bar ‚Äî factual, neutral, professional, never braggy or competitive (see [The repo is PUBLIC](#the-repo-is-public--never-commit-internal-discussion-high-priority) and the commit-message tone rule); these rules do not relax it.

- **Acknowledge external issues the moment you start ‚Äî with proper tone.** When you begin work on an EXTERNAL (non-maintainer) issue, post a brief "Investigating" (or similarly worded) comment so the reporter knows it's been seen and is being looked at. A short, sincere acknowledgement, never boilerplate or over-promising ‚Äî state that you're looking into it, not when it'll be fixed. (Internal/self-filed issues don't need the acknowledgement.)
- **Reference the associated issue(s) in every PR body.** A PR that resolves a bug uses a closing keyword ‚Äî `Closes #123` / `Fixes #123` ‚Äî so the merge auto-closes the issue; a PR that merely relates to an issue (touches the area, partial work, follow-up) uses `Refs #123`. Never land a fix-PR that leaves its issue unlinked.
- **On EVERY release, comment the release link + version on ALL closed issues and merged PRs that shipped in it.** This is MANDATORY maintainer hygiene ‚Äî always do it, no matter what. For each issue closed and each PR merged since the previous release tag, post a comment with the version it shipped in and a link to the release, e.g. `Shipped in v0.1.8: https://github.com/nubjs/nub/releases/tag/v0.1.8`. The fix being merged is not the same as it being on a published binary; this comment closes that gap for the reporter. (The `release` skill's Step 5 carries the concrete enumeration + commands.)
- **Close issues with a brief factual comment ‚Äî never silently.** Use `gh issue close <number> --comment <text>` (see the dedicated rule in [Workflow lifecycle](#workflow-lifecycle--never-kill-mid-edit-run-queues-as-a-continuous-series-high-priority) above), stating what fixed it or why no code fix is needed.

## fray (orchestrator-first, sub-agents as instruments) ‚Äî a named methodology (HIGH PRIORITY)

Full methodology: `.claude/skills/fray/SKILL.md`.

This is **fray** ‚Äî the preferred way to run a large, mixed set of efforts (investigations + decided fixes + verifications) toward a goal like a launch. Reference it by name. It is the default for any multi-effort push where the work is part *find-out-what's-true* and part *land-the-decided-thing*, and where the human wants to stay in the loop on the decisions the investigations surface. Use it **instead of the `Workflow` tool** for these ‚Äî workflows hardcode a DAG up front and fan out hard (expensive, and they bury the decision points); fray stays cheap and dynamic by driving everything from the main session with individually-dispatched sub-agents.

**The operational essentials:**

- **You (the main session) are the orchestrator, and the ONLY decider.** You hold the whole picture, dispatch sub-agents as *instruments*, ingest what they return, and decide the next move. You do not hand the steering wheel to a workflow script.
- **A living tracker document is your control surface.** One markdown doc (e.g. `epics/<epic>/HANDOFF.md` or `STATUS.md`) holds: a status board (one row per effort), a detail card per effort, a **Questions-awaiting-human** queue, and a reusable **codebase map** so future dispatches are cheap. You OWN this file ‚Äî after every sub-agent returns, fold its facts into the card, advance the status, and re-derive "what's next" from the board. The doc is extensible by design: new efforts get a new ID and a new row. Status vocab: `todo ¬∑ probing ¬∑ needs-decision ¬∑ ready ¬∑ landing ¬∑ verifying ¬∑ done ¬∑ blocked`.
- **Thread write-ownership: the dispatched sub-agent edits ITS OWN thread doc directly.** A doing-agent has the full context on its effort and best represents that thread's current truth, so it edits its own thread's status/decisions/next-step IN PLACE (single-voice, current-truth ‚Äî no changelog appends), and writes the DEPTH into the thread itself ‚Äî **the thread is ONE self-contained, unified document, never a stub that points at scattered findings sidecars.** A `.fray/<thread>.findings/<id>.md` sidecar is justified ONLY for a PARALLEL fan-out (N agents feeding ONE effort, where concurrent single-file writes would clobber), and even then it is a TRANSIENT working artifact the orchestrator MUST integrate into the unified thread doc (the resting state is one mega-doc). The orchestrator does NOT re-transcribe what a single agent wrote into its own thread (reconstructing status from a packaged summary is lossy), but DOES integrate parallel-fan-out sidecars into the one doc. The file tools enforce read-before-write + reject stale edits + atomic writes, so even a concurrent same-file edit can't silently clobber; the guideline is to keep one live agent per thread (dispatch chains are sequential), and an agent edits only its own dispatched thread, never another. The orchestrator's RESIDUAL thread role is what a local-context agent structurally can't do: cross-thread linkage and reversals, the human-decision queue, dispatch and synthesis across efforts, and editing the globals doc. The dispatch-binding bookkeeping is NO LONGER manual ‚Äî a PostToolUse hook (`fray-record-binding.mjs`) auto-records each dispatched agent's `{id, label}` into its thread's `agents:` frontmatter, and the merge-queue drain auto-sets a thread `status: done` on a confirmed PR merge; the orchestrator hand-edits thread status only for non-merge terminal cases or to fold a non-PR return.
- **NEVER put verbatim quotes from the user into ANY document** (fray threads, tracker docs, research write-ups, code comments, commit messages ‚Äî anything). The user dislikes seeing their own words quoted back. Record the *decision or fact*, paraphrased in neutral third-person ("Decided: bump to 98.8" / "Direction: add cafile support"), never `User: "..."` or a `"..."` block attributed to them. This holds for LOCAL docs too, not just tracked/public files (that's the separate brand-boundary rule). Capture what was decided, not who said it or how they phrased it.
- **Sub-agents are instruments, not deciders.** A probe returns *facts* ‚Äî divergences, traces, measurements ‚Äî not verdicts. No sub-agent autonomously lands a change to a default / security posture / product behavior / API-config-env surface / error-contract; those route back to the human as a question. Mechanical / clearly-a-bug fixes may land (you review the diff). Every sub-agent prompt is **self-contained** (a model switch starts a fresh cache); embed the relevant slice of the codebase map so the agent doesn't re-derive it.
- **Model-tier every dispatch by JUDGMENT REQUIRED, not just task type.** Haiku *cannot self-steer* ‚Äî give it only fully-scripted mechanical work where every decision is pre-made. A differential probe that must recognize a divergence and judge real-bug-vs-intended needs judgment ‚Üí Sonnet (or Opus), never Haiku. Opus does the fix that lands; you verify. Re-verify cheap-tier load-bearing claims yourself ‚Äî a Haiku verdict is a *lead, not a fact*. Background sub-agents are HEADLESS (no TTY): they cannot drive a REPL or answer a permission prompt; any interactive-surface probe must run in the orchestrator's FOREGROUND. (See the model-tiering section above and the full skill for detail.)
- **ALWAYS STEER ‚Äî DON'T RE-DISPATCH (ABSOLUTE TOP-PRIORITY RULE; agent-teams is ON).** `SendMessage` works on RUNNING sub-agents (delivered at the next tool round) AND resumes COMPLETED ones with full prior context (warm resume ‚Äî message the `agentId`). The DEFAULT for every course-correction is to STEER the existing agent, never to let it die and cold-redispatch a replacement (that throws away its runbook + accumulated context). Reach for it constantly: fold a new follow-up into the agent already editing a file (preserves one-editor-per-file), feed a just-landed decision or superseding fact to every running agent it affects, answer a question an agent surfaced (`to:"main"` ‚Üí reply `to:"<agentId>"`), redirect/re-scope mid-flight, or ‚Äî when an agent finished/halted on a TEMPORARY blocker (a HOLD you issued, a transient CI failure, a now-available input) ‚Äî RESUME it the instant the blocker clears. The tell you're violating this: catching yourself writing "I'll re-dispatch X once Y lands" ‚Äî instead resume X with "Y is landing, continue." NEVER kill an agent (orphans WIP), NEVER spawn a clobbering sibling against a file another agent owns. Reserve the `enqueued` thread-status ONLY for a *separate* dispatch that genuinely must wait for a named agent's completed output. (This reverses the obsolete "no in-flight steering / agent-teams off" guidance ‚Äî verify with `echo $CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS`.)
- **Eagerly ask, and re-surface.** Push questions to the human the moment an investigation surfaces one ‚Äî both in the tracker's queue and in chat. If a question goes unanswered, assume it was *missed, not declined*, and re-surface it the next turn.
- **Parallelize aggressively ‚Äî codebase-editing work included.** Read-only probes parallelize without limit. Landing agents that EDIT code parallelize too: the file tools enforce read-before-write, reject stale edits, and write atomically, so concurrent edits to one tree can't silently clobber and a concurrent build reads a consistent old-or-new file (at worst a spurious re-run it retries past) ‚Äî NOT a reason to serialize. The narrow guards: (1) forbid destructive raw git ops (`reset`/`checkout`/`stash`) in parallel-agent prompts ‚Äî they bypass the file-tool safety and wipe a sibling's WIP; (2) an agent whose deliverable is a *trustworthy build/test result* on `vendor/aube` (does my patch pass tests + preserve upstream behavior?) gets its OWN isolated clone + `CARGO_TARGET_DIR` so siblings can't contaminate the result ‚Äî that is how you fan out fork-patch landing work; (3) serialize only the final assembly of those isolated patches onto `nub-fork` (apply, bump the pin once, integrated-test once). Everything else fans out.
- **Dynamic, not pre-planned.** There is no committed step list to march through. Each turn: read the board, ingest returns, surface new questions, dispatch the next round honoring deps + the build-correctness constraint, update the board. The human can re-prioritize or stop after any round ‚Äî the tracker means nothing is lost.

The win over a hardcoded workflow: the human answers the questions the investigations raise *as they arise*, spend stays low (cheap sub-agents, no fan-out tax), and the orchestrator keeps a coherent mental model of the whole effort instead of delegating it to a script.

## What Nub is, briefly

- **A Rust CLI that orchestrates the user's installed Node.** v0 is a fast script runner plus a TypeScript-just-works runtime. Mechanism: Node's own extension surfaces ‚Äî `module.registerHooks()`, `--import` preload, `--require`, env vars, N-API addons, V8 flag injection.
- **TypeScript-first developer supertool.** Own transpiler (oxc-based). TS, JSX, non-erasable syntax, `emitDecoratorMetadata` all supported.
- **Compatibility is paramount.** Code targeting Node must run on Nub byte-for-byte. The contract: **default mode = Nub runtime augmentation active; `NODE_COMPAT=1` or `--node` = augmentation disabled, plain Node behavior**.
- **Behavioral additivity** is the rule for augmentation ‚Äî [`wiki/philosophy.md#additivity`](wiki/philosophy.md#additivity) for the canonical statement. "Additive" means doesn't break code written for vanilla Node; mechanism is restricted to Node's own extension surfaces (not source patches).

## The augmenter / fork distinction is load-bearing

A previous direction (2026-05-17) positioned Nub as a soft fork of Node, with patches to Node source on the table. **That direction was reversed 2026-05-18.** Nub is now a Rust CLI augmenter via Node's extension surfaces only. Source patches are off the table.

If you're editing a plan doc and find "soft fork" or "patched Node" language, it predates 2026-05-18 and should be updated. The authoritative statement of the current direction is [`wiki/architecture.md#augmenter-not-fork`](wiki/architecture.md#augmenter-not-fork).

The reason the distinction is load-bearing for agents specifically: implementation approaches that *sound* clean ‚Äî "just patch `lib/internal/modules/esm/resolve.js` to do X" ‚Äî were valid under the old posture and are no longer. The current mechanism rule is **"would a user on plain Node, plus the corresponding `module.register()` / `--import` / npm-addon, get the same result?"** If yes, in scope. If no, find a different mechanism or scope the feature out.

### The brand boundary ‚Äî public surfaces only (internals are exempt)

The brand boundary protects **public-facing surfaces only** ‚Äî anything a user imports, installs, types, or authors as configuration, or any API they call by a documented name. **Internal mechanisms are explicitly exempt:** error/warning codes, environment variables Nub sets for its own cross-process / shim plumbing, internal `globalThis.__nub*` sentinels, and internal cache/dir names ‚Äî the brand may leak there freely and chasing it out is wasted effort. (Error codes and the internal cross-process env vars Nub sets to change shim behavior are internal mechanism; the concern is only the brand leaking into public-facing APIs ‚Äî don't over-police the boundary for internals.) The rules below govern the *public* surface and are absolute there ‚Äî do not add a public nub-branded API even "behind a flag."

- **No PUBLIC `globalThis.nub`.** Nub never injects a *documented, user-callable* global named after itself; `Bun.*`-style conveniences, if they exist at all, go in importable modules. Internal markers are FINE ‚Äî a `globalThis.__nub*` sentinel used for runtime/cross-process plumbing that user code never references is not a leak (reversed 2026-06-11; the old "no internal markers either" was too strict).
- **No `nub:*` module namespace.** Nub never registers synthetic specifiers users would `import` (`nub:test`, `nub:sqlite`, `nub:serve`). The `node:*` namespace is upstream Node's and closed; Nub does not squat a sibling. (Public surface ‚Äî users type it.)
- **No `@nub/*` npm scope** for user-facing packages or APIs ‚Äî the bare `@nub` scope is unused entirely. The CLI ships under the project's own `@nubjs` org: `@nubjs/nub` (the package users `npm install -g`) plus its `@nubjs/nub-<platform>` `optionalDependencies`, which carry the per-platform binary + N-API addon and are install-time plumbing never imported by user code (same shape as `@biomejs/biome` + `@biomejs/cli-*`, `@rollup/rollup-*`, `@esbuild/*`). Nub's ambient TypeScript declarations ship as `@nubjs/types` ‚Äî our own org, a types-only devDep, not a user-imported API. (We do not own the npm package `nub`, so `@types/nub` on DefinitelyTyped is not a viable home.) `@nubjs` is deliberately distinct from the prohibited bare `@nub`. See [`wiki/philosophy.md#the-brand-boundary`](wiki/philosophy.md#the-brand-boundary) for the full rationale.
- **Internal `NUB_*` env vars are FINE; never a brand env var as a documented USER knob.** Reversed 2026-06-11: Nub MAY set `NUB_*` / `__NUB_*` environment variables for its own cross-process plumbing ‚Äî telling a child shim how to behave, detection sentinels, debug toggles ‚Äî these are internal mechanism, not a public API, and the old "no `NUB_*` ever, it shows up in `printenv`" rule was over-strict. What remains: don't *document* a `NUB_*` var as a knob users are meant to set, and for genuinely user-facing toggles prefer `NODE_*` (where Node doesn't claim the name, e.g. `NODE_COMPAT`) and neutral conventions (`NO_COLOR`, `FORCE_COLOR`, `PORT`, `NODE_ENV`).
- **The nub runtime respects ZERO `AUBE_*` env vars ‚Äî the embedded engine's brand is never nub's config surface (nub is the package manager).** DISTINCT from the exemption above. That exemption covers internal vars nub *sets* for its own children (detection sentinels, shim signaling ‚Äî a user never touches them). THIS rule covers vars the nub runtime *reads from the environment to decide behavior* ‚Äî those are user-facing knobs, and when aube runs *as nub* the user sees nub while aube is an invisible implementation detail, so an `AUBE_`-branded knob is a brand leak. aube currently exposes ~30 such runtime vars (`AUBE_DISABLE_CLONEDIR` ‚Äî already documented as the clonefile opt-out ‚Äî `AUBE_CONCURRENCY`, `AUBE_CACHE_DIR`, `AUBE_NO_UPDATE_CHECK`, `AUBE_PRIMER_PICK_GATE`, the `AUBE_DISABLE_*` perf toggles, `AUBE_DIAG_*`, `AUBE_SELF_*`/`AUBE_VERSIONS_HOST`, ‚Ä¶); under nub, none of these may be the surface. Surface preference (now that nub fully owns package management, `NUB_*` is legitimate, no longer a last resort): use a **neutral npmrc field** where one reads naturally (config any tool could honor ‚Äî also reachable as `npm_config_<field>`), else a **`NUB_*`** env var / nub-namespaced config is FINE for nub's own PM knobs (e.g. `NUB_CACHE_DIR`, `NUB_PRIMER_TTL`). The hard line stays: **never `AUBE_*`** (the embedded engine's brand) for a nub-facing knob. Two disciplines on top: (a) **keep the exposed set as SMALL as possible** ‚Äî every knob is a maintenance + doc + support surface; expose one only on real demand, default the rest internal or dropped; (b) **design each knob to GENERALIZE, not special-case** ‚Äî a `NUB_PRIMER_TTL` *duration* (default unlimited), never a boolean `primer_evergreen` flag (config should be as generalizable as possible; a binary is a degenerate special-case of the right abstraction). MECHANISM (structural fix, not var-by-var whack-a-mole): extend the **Embedder profile** ‚Äî the same compile-time pluggability jdx merged in #862 ‚Äî with an env-prefix / env-resolution hook, so standalone aube keeps reading `AUBE_*` while the NUB profile reads neutral/`npm_config_*`/`NUB_*`; the boundary is enforced once, structurally. (Build-time `AUBE_*` knobs in aube's `build.rs` ‚Äî `AUBE_PRIMER_TOP`, `AUBE_REQUIRE_PRIMER`, `AUBE_PRIMER_GENERATED_AT` ‚Äî are aube's own build machinery our CI sets; out of scope for this *runtime* rule, a low-priority consistency cleanup at most.) Audit + migration tracked as the `brand-boundary-env` thread.
- **Prefer neutral names for USER-AUTHORED config (a public surface).** A `package.json` field or config file a user writes is public, so Nub reads a *neutral-named* `tasks.json` / `"tasks"` field rather than a `"nub"` field or `nub.toml` ‚Äî the config a user authors shouldn't be brand-locked (the `"workspaces"` precedent: yarn coined a generic field, npm/bun adopted it). This is a public-config preference, distinct from the internal-exemption above. Carve-out: Nub may adopt a vendor-neutral, any-tool-could-implement-it field by decision record ‚Äî currently only `"tasks"` ([`wiki/commands/tasks.md`](wiki/commands/tasks.md)).
- **The brand boundary is SYMMETRIC ‚Äî a true nub project consumes only un-branded, cross-tool config.** Just as Nub never *emits* its own brand into your config (no `"nub"` field), **when Nub is the project's active PM it never *consumes* config branded with another package manager** ‚Äî `pnpm.overrides`, `pnpm.*`, yarn-namespaced config, Bun-branded fields, etc. are all ignored. A nub-identity project respects only the neutral fields any tool could read: `overrides`, `resolutions`, `catalog`, `workspaces`. (When the inferred active PM is npm/pnpm/yarn/bun ‚Äî a *compat* project, not a nub project ‚Äî Nub MIRRORS that PM faithfully, *including* its branded fields, because the project genuinely is that PM's. The no-other-brand rule is specifically for nub-identity projects.) This is the config-side of the brand boundary; the mechanism lives in the config-scoping policy (`crates/nub-cli/src/pm_engine/config_scope.rs`, "mirror the active PM, never silently").
- **pnpm-NAMED files and paths are NEVER read unless pnpm is the incumbent PM (non-negotiable).** When the project's incumbent package manager is NOT pnpm, Nub does not read any file or path that contains the literal string "pnpm" in its name ‚Äî this includes `pnpm-workspace.yaml`, `.pnpmfile.cjs`, `.pnpmfile.mjs`, `.pnpmrc`, `~/.config/pnpm/` (including `~/.config/pnpm/auth.ini`), and any `pnpm.*` field in `package.json`. The gate is on the **NAME**, not the semantic effect ‚Äî "it's auth-only / harmless / no resolution impact" exempts nothing. Crucial distinction: **pnpm-NAMED ‚âÝ pnpm-specific.** A *generically-named* field that only pnpm happens to support (e.g. `overrides`) is a separate, case-by-case decision ‚Äî it is not pnpm-branded by name and may be honored in a nub-identity project. The file-name half of the symmetric boundary: just as Nub never emits pnpm-branded names into your config, it never reads them unless pnpm is provably the active PM.
- **Error codes are internal ‚Äî don't brand-sweep them.** `ERR_NUB_*` / `ERR_AUBE_*` / `WARN_*` identifiers (and the engine errors that surface them) are internal mechanism; a brand string in a code or message is acceptable and not worth chasing. *Adding* a missing code is good UX; rebranding an existing one for purity is not required.
- **No vendored Node patches.** Nub does not patch Node source, ship a custom-built Node binary, or embed `libnode`. Every augmentation rides on Node's public extension surfaces. (This is an architecture/additivity rule, not a brand rule.) If a feature can't be expressed through those, find a different feature.

### `nub`-named npm packages, explained

Two families of `nub`-named packages exist on npm. Neither is the prohibited bare `@nub/*` scope, and neither is user-facing application code:

1. **`@nubjs/types`** ‚Äî Nub's ambient TypeScript declarations ship as a types-only package under the `@nubjs` org we own and control. (We do NOT own the npm package `nub`, so `@types/nub` on DefinitelyTyped is not a viable home ‚Äî `@types/<pkg>` maps to an npm package `<pkg>` we don't control, and DefinitelyTyped is community-controlled. The `@types/*` auto-load is also not the right pickup mechanism.) `@nubjs/types` is the correct home for any declarations user code references ‚Äî `import.meta.hot` for hot-mode consumers, Nub-specific globals, etc. ‚Äî as a devDep, never a runtime import. Brand-clean: under our own org, no bare `@nub` claim.

2. **`@nubjs/nub` + `@nubjs/nub-<platform>`** ‚Äî the CLI itself and its platform-specific binary distribution packages (`@nubjs/nub-darwin-arm64`, `@nubjs/nub-linux-x64`, ‚Ä¶ ‚Äî 8 platforms total). `@nubjs/nub` is the package users `npm install -g`; the `@nubjs/nub-<platform>` packages are its `optionalDependencies`, each containing the compiled Rust binary + N-API addon for one platform, selected at install time by npm's `os`/`cpu` filters and copied into place by `postinstall.js`. They are install-time plumbing ‚Äî never `import`ed, never referenced in application source, never a thing users type. Same distribution pattern as `@biomejs/biome` + `@biomejs/cli-*`, `@oxlint/cli-*`, `@rollup/rollup-*`, `@esbuild/*`. The org scope is `@nubjs` (the project's own npm org), deliberately distinct from the prohibited bare `@nub`.

## The aube vendoring + upstreaming ‚Äî MEMORIZE THIS (forgotten twice; ground every aube claim here, not memory)

nub's package manager IS the vendored **aube** engine, embedded as a Rust LIBRARY. The structure (re-read this before asserting anything about how the integration works):

- **Embed, not subprocess.** `vendor/aube` is a path dependency ‚Äî a checkout of *our fork*. nub has its OWN CLI (clap dispatch + `pm_engine::ENGINE_VERBS`) and calls `aube::commands::<verb>::run(typed_opts)` IN-PROCESS. There is NO aube subprocess and NO aube CLI under nub; aube's own `cli_main` + tool subcommands are DEAD under nub. All engine output is rebranded through `crates/nub-cli/src/pm_engine/present.rs` (`ERR_AUBE_*`‚Üí`ERR_NUB_*`, `aube`‚Üí`nub`, jdx URLs stripped).
- **Fork vs upstream.** The fork is **`nubjs/aube`**; upstream is **`jdx/aube`**. `vendor/aube` is pinned to a fork commit (`nub-fork` branch); bumping the pin updates the embedded engine.
- **UPSTREAMING ‚Äî contribution PRs target `jdx/aube` (UPSTREAM), NEVER the fork (the thing forgotten 2026-06-14).** A contribution PR is `jdx/aube:main ‚Üê nubjs/aube:<branch>` (e.g. #888 = `embedder-profile-fixes`; #897 = the `process-exit-sweep` stack). Do NOT open a fork-internal `nubjs/aube ‚Üê nubjs/aube` PR to upstream ‚Äî it upstreams nothing. **Stacked PRs also target `jdx/aube`** (base `main`; note the stack + which commit to review in the body; the diff auto-cleans once the parent PR merges). Cross-fork stack creation: `gh pr create --repo jdx/aube --base main --head nubjs:<branch>`.
- **What upstreams vs stays fork-only:** PLUGGABILITY / additive changes that are NO-OP for standalone aube (the embedder profile, env-resolution hooks, the `prog()`/`cmd()` source-branding helpers, the exit-code sweep) ‚Üí upstream to jdx/aube. MATERIAL PM-behavior / nub-specific changes (primer-TTL, etc.) ‚Üí stay in the fork (merge-synced on main, never rebased ‚Äî see the workflow below), upstreamed later only if jdx wants them.
- **The embedder profile** (jdx merged it as #862): compile-time pluggability. nub sets its own profile (`cache_namespace` "nub/pm", `env_prefix`/`config_env_prefix`, banner-gating, pnpm-surface off, `read_branded_pnpm_config` gating). The NUB profile is what makes the brand + config boundary hold; standalone aube keeps its `AUBE_*` behavior.
- **Fork-discipline for EVERY vendor/aube change:** default-preserving (standalone aube byte-for-byte unchanged on the default path; the embedder opts in) OR a justified latent-bug-fix jdx would accept unconditionally. A trustworthy-build/test deliverable gets its OWN isolated clone + `CARGO_TARGET_DIR`; the final assembly onto `nub-fork` + the pin bump is serialized.

### The canonical `nub-fork` workflow ‚Äî commits, not patches; push before pin (MEMORIZE; an inconsistency here just caused a real CI-killing bug, 2026-06-15)

This is the durable, non-negotiable convention for every change to the vendored engine. Be totally consistent ‚Äî the commits-vs-patches drift caused a pin pointing at an unpushed commit that would have died at CI submodule-init.

- **ONE blessed branch = all our aube work, as real COMMITS.** `nubjs/aube` branch `nub-fork` is the single source of truth holding every nub-specific change to the embedded engine. Changes land as **real git commits on `nub-fork`** ‚Äî NEVER as parked `.patch` files (`git format-patch` dumped into `.fray/‚Ä¶` "to apply later"). Commits, not patches: a transient patch while moving a commit between clones is fine; a `.patch` as the *resting representation* of fork work is not. (Companion memory: `aube-fork-changes-as-commits`.)
- **The pin tracks a `nub-fork` commit, and you PUSH `nub-fork` BEFORE bumping the pin.** The `vendor/aube` submodule gitlink (the "pin") must point at a commit ALREADY on the `nub-fork` remote. Bumping the pin to a local-only/unpushed commit makes every CI job die at submodule-init. The order is absolute: **push `nub-fork` ‚Üí bump the pin ‚Üí push `main`.** (Reinforces `vendor-aube-pin-push-order`.)
- **Sync with upstream by MERGE, never rebase.** Keep `nub-fork` current with `jdx/aube:main` via `git merge upstream/main` (resolve conflicts preferring upstream for files we don't own). Do NOT rebase `nub-fork` ‚Äî rebasing rewrites SHAs, forces a force-push, and invalidates the pin on every sync. Merge keeps SHAs stable so the pin stays valid; this is the standard long-lived-downstream pattern.
- **Upstreaming: cherry-pick onto a FRESH branch off `jdx/aube:main`, PR, then converge on merge.** To contribute a fork change upstream, cherry-pick the specific commit(s) onto a fresh branch cut off `jdx/aube:main` (NOT off `nub-fork` ‚Äî so the PR diff is only that change, not the whole fork delta) and open the PR to `jdx/aube:main` (e.g. audit #900). The change is then two commits ‚Äî one on `nub-fork` (ships to users), one on the upstream branch (for jdx). When jdx merges, the next `merge upstream/main` into `nub-fork` CONVERGES it: git dedups identical content (or a trivial conflict if jdx tweaked it in review ‚Äî resolve by taking upstream, dropping the now-redundant fork commit). The change graduates OUT of the fork-only delta, keeping `nub-fork`'s diff-vs-upstream as thin as possible.
- **Parallel fork work:** an agent making a fork change works in a clone of `nub-fork` (or the live `vendor/aube`), commits, and PUSHES to `nub-fork`; the orchestrator fetches and bumps the pin. The serialized step is only that final **pin-bump + `main` push** (one assembly: apply/converge the pushed commits, bump the pin once, integrated-test once) ‚Äî the fork edits themselves parallelize (each in its own clone + `CARGO_TARGET_DIR` when the deliverable is a trustworthy build/test result).

## Testing philosophy

**Minimum number of tests, comprehensively covering the API surface.** This is the load-bearing rule for the v0.1 implementation, and the antidote to AI-generated test bloat. Quality of coverage matters; volume of tests does not.

> **Before you push, run the pre-push local verification loop** (see [Implementation quality discipline](#implementation-quality-discipline) below): incremental worktree build ‚Üí the *exact* CI `clippy --all-targets`/`fmt`/scoped tests ‚Üí an e2e tmp-fixture run of the specific feature ‚Üí Docker for global-cache/config behavior ‚Üí promote durable checks into the suite. Get it green locally and push ONCE ‚Äî pushing fix-after-fix to "see if CI passes" starves the shared runner pool.

- **Comprehensive coverage, not exhaustive coverage.** Each behavior gets tested once, well, against the cases that matter: golden path, the one or two failure modes that have user-facing implications, the boundary condition that's easy to get wrong. Stop there. Don't write `it("should not throw when input is null")` after you've already written `it("rejects null input")`.
- **Hand-crafted feel.** Tests must read as if a careful engineer wrote them, not as if an agent enumerated possibilities. Concrete signs of agent bloat to avoid: identical assertions across multiple `describe` blocks with different setup; per-input parametrization where one assertion would do; test names that paraphrase the implementation instead of describing the contract; descriptions like "should handle X" without naming what "handle" means.
- **Carefully considered abstractions.** Test helpers and fixtures are part of the contract. Don't bury behavior in clever shared setup; don't copy-paste either. Pick the shape that lets a reader skim a test file in 30 seconds and understand what's being verified. If a test file is over 300 lines, you're probably testing too many things in one place or each test is too verbose ‚Äî split or trim.
- **Some things are untestable, and that's fine.** Perf-shaped behavior, OS-specific corners, race conditions that require infrastructure we don't have, behaviors that depend on Node's internal scheduling. Leaving them untested is honest; writing ceremonial fake tests for them is sloppification. Document the gap in a comment if it'll surprise a future reader.
- **Pull from upstream where it makes sense.** Node ships an extensive test suite at `nodejs/node/test/`. The executable-level subset (tests that just spawn `node` and assert behavior) is the strongest possible compat-validation against Nub. We run it black-box in two modes: `nub --node` (expected ~100% pass ‚Äî passthrough mode) and `nub` augmented (expected near-100% with documented divergences). See `wiki/research/node-test-suite-leverage.md` for the harness design and CI cadence.
- **Test names describe the contract, not the implementation.** `test("emits ECONNRESET when peer closes mid-write")` is good; `test("emits ECONNRESET")` is too short to mean anything; `test("calls socket.destroy() with the EconnReset constructor when..."` is paraphrasing the code.
- **Failure messages must be self-debugging.** When a test fails in CI, the assertion message should make the cause obvious without rerunning locally. Asserting `expect(result).toEqual({ ok: true })` is fine when it passes and useless when it fails ("expected {ok: true}, got {ok: false}"). Use `expect(result.ok).toBe(true)` or write a custom message.

## Docker is available ‚Äî use it instead of declaring things "untestable"

**Docker may be available.** Before you write off a behavior as "hard to test" or "can't verify locally," check whether a container closes the gap ‚Äî several things the testing philosophy lists as untestable corners actually *are* reachable through Docker:

- **A clean, dependency-free environment.** A fresh `node:22.15-slim` (or no-Node `debian:slim`) container with no `~/.cache/nub`, no global Node, no local dev state ‚Äî the honest way to test first-run install, the curl `install.sh` flow, `nub upgrade`'s `~/.nub` channel, provisioning a Node from `nodejs.org`, and "works on a clean machine."
- **A specific / floor Node version.** Pin `node:22.15` to exercise the **floor-only** defects a modern host Node (24/26) masks ‚Äî `using`/`await using` down-leveling, the Temporal `toTemporalInstant` path, version-gated polyfills, the async-tier `module.register` loader. This is the local complement to the Node-22.15 CI leg.
- **Linux-specific paths.** musl-vs-glibc detection (`node:22-alpine` vs `node:22-slim`), arch matching, signal/`PATH` behavior on Linux.

**What Docker does NOT give you:** it runs **Linux containers only** (on a macOS/arm64 host ‚Üí `linux/aarch64` by default, `--platform linux/amd64` via slow QEMU emulation). **Windows containers require a Windows host**, so cmd.exe behavior, the `--shell-emulator`, `.cmd`/`.bat` resolution, and `nub.exe` vs `bin/nub` still can only be verified on the **`windows-latest` CI leg** ‚Äî Docker is not a substitute there. When an item's *Verify* says "on Windows," it means Windows CI; do not claim a Docker run verified it.

Keep containers ephemeral (`docker run --rm`), mount the repo read-only where you only need to read it, and never leave long-running containers behind.

## Probing methodology ‚Äî differential fixtures, empirical over source (learned 2026-06-10)

The highest-yield way to find correctness bugs in nub is a **differential fixture**: a minimal fixture isolating ONE behavior, run against nub AND the reference tools it claims parity with (npm / pnpm / bun / node) on identical inputs ‚Äî a divergence is the finding. This consistently beats happy-path app rounds: days of running real apps under nub surfaced almost nothing, while one five-minute `.env`-visibility fixture found two real bugs (a secret-leak over-reach where nub injected `.env` into non-node tools, and a NODE_ENV-cascade bug). Build the fixture, run every relevant tool, diff. Always compare against the thing you assert parity with ‚Äî "nub does X" is unverified until "‚Ä¶and npm/pnpm/bun do Y on the same fixture" is in hand.

Three supporting disciplines, each earned the hard way:

- **Test empirically before reading source, and before deciding.** A throwaway fixture in a sandbox answers "what does it actually do?" faster and more reliably than tracing Rust/Zig. When a decision hinges on a tool's real behavior, reproduce it ‚Äî don't infer it from code.
- **Read the COMMENTS and the resolution, not just the issue body.** A bug's body says what someone wanted; the maintainer comments and *why it was closed* say what's true. An "open feature request" can be a deliberate rejection with the real rationale in the thread ‚Äî Bun's auto-`.env`-for-scripts looked like unmet demand in the body, but the comments showed it was intentionally rejected, with the core maintainer's root-cause analysis. Bodies alone will make you confidently wrong.
- **Ground every claim about the system in code or an experiment ‚Äî never memory.** A confidently-designed probe this session referenced `nub test`, which is not a command. Verify the actual surface (read `cli.rs`, or run it) before you build against it.

And: **reversing a decision on new evidence is correct, not flip-flopping.** Architecture calls routinely dissolve or flip once the facts arrive ‚Äî a "minimal vs elegant" split collapsed when investigation showed the elegant option was also the only *correct* one; a "keep this feature / lean in" call reversed twice as comments and a fixture came in. Present the decision, then keep probing; let evidence move you rather than defending the first answer.

## Iterating across Node versions and tiers

nub's behavior splits by **tier** ‚Äî the fast tier (Node 22.15+, sync `module.registerHooks`) and the compat tier (18.19‚Äì22.14, async loader-worker via `module.register`) take different code paths and break differently ‚Äî so a green run on a single modern Node (often 26) routinely masks compat-tier and floor-only defects. nub discovers its Node from `PATH`, so the cheapest way to drive it onto a specific version is `PATH="$HOME/.nvm/versions/node/v20.19.0/bin:$PATH" nub ‚Ä¶`; sweep several to cover both tiers, and reach for Docker (per the section above) for Linux + floor confirmation before trusting a result. **Do not claim cross-version support from one Node.**

Feature-specific harnesses that encode this loop live under `tests/<feature>/` ‚Äî e.g. `tests/pnp/` (Yarn PnP) builds a real fixture (`make-fixture.sh`), runs a scenario matrix across Node versions on the host (`run-pnp-matrix.sh`) and on Linux in per-version containers (`docker-matrix.sh`), and documents the whole loop + the tier model in `tests/pnp/README.md`. When you build or extend a system for iterating on a hard-to-unit-test feature, document it in-place like that (a `README.md` next to the scripts, plus comments in the scripts themselves) so the next agent reproduces it in minutes instead of rediscovering it.

## Implementation quality discipline

**Quality over velocity. Always.** Don't move fast, check boxes, and ship stubs as complete implementations. The specific failure modes:

- **Never mark a task `[x]` without verifying the behavior end-to-end.** Running `cargo test` is necessary but not sufficient. If the task says "implement per-line stream prefixing matching pnpm," you must run `nub -r --stream run build` on a real fixture and compare the output to `pnpm -r --stream run build`. A green test suite with a stubbed implementation is worse than an unchecked task.
- **Never claim "parity" without evidence.** "Complete workspace parity with pnpm" means every flag, every edge case, every output format. If you haven't tested a flag, don't mark it done. If the implementation is a simplified version, say so explicitly in the task note ‚Äî don't use language like "implemented" for something that's "scaffolded."
- **Name what you actually built.** If you shipped `Stdio::inherit()` with a header line, that's not "stream prefixing." If you partitioned into fixed batches, that's not "work-stealing." Use precise language so the next reader knows the actual state.
- **If you're completing 10 tasks per hour, you're not testing them.** A thoughtfully implemented and tested task takes 15-30 minutes. If a phase has 12 tasks and you completed it in 20 minutes, audit yourself.
- **Shipping a user-facing feature or behavior change MUST include updating the docs.** A feature is not "done" until `site/content/docs/` reflects it. The code change and the doc update are the same effort ‚Äî do not close the task or mark it complete until both are landed.
- **VERIFY LOCALLY BEFORE PUSHING ‚Äî don't outsource verification to CI (the pre-push loop) (HIGH PRIORITY ‚Äî protects shared CI capacity).** CI is the backstop, not the first place a defect should surface. This is a RUNNER-CONTENTION rule, not just a quality one: the repo has a finite GitHub Actions runner pool, every push fires ~5 workflows √ó the 8-platform matrix (~30-40min each), and several open PRs pushing fix-after-fix to "see if CI is green" SATURATES the pool ‚Äî head-commit jobs then queue for HOURS (this actually happened: a config PR's run sat queued ~1h40m with zero jobs running while fix-commits churned). A red CI on something locally-catchable (the #34 `clippy --all-targets` miss, a fmt diff) is doubly wasteful ‚Äî it burns a full matrix AND a restart. So: get it green LOCALLY, push ONCE. Before pushing a code change, run the loop in your worktree:
  1. **Incremental build** in the worktree (own `CARGO_TARGET_DIR` so it's warm/fast across iterations) ‚Äî `cargo build` for the touched crates.
  2. **Run EXACTLY what CI runs** for the cheap gates: `cargo clippy --all-targets --all-features -- -D warnings` (scoped `-p` WITHOUT `--all-targets` misses test-code lints ‚Äî the exact #34 gap) AND `cargo fmt --check` AND the scoped `cargo test` for what changed. Match the invocations in `.github/workflows/ci.yml`, don't approximate them.
  3. **End-to-end test the SPECIFIC functionality with a tmp fixture** ‚Äî build the release (or debug) `nub` binary and exercise the actual feature against a throwaway fixture in `/tmp`, diffing against the reference tool (npm/pnpm/yarn/bun/node) where parity is claimed. "Tests pass" is not "the feature works"; run the feature.
  4. **Use Docker for behavior that touches the GLOBAL cache / config / a clean machine** ‚Äî global `~/.npmrc`/config homes, the CAS store, first-run/clean-env install, a Node-version floor ‚Äî run it in an ephemeral `docker run --rm` container so host state doesn't mask or pollute the result (see the Docker section above). Linux-only; Windows still rides CI.
  5. **INCORPORATE the verification into the actual test suite where reasonable** ‚Äî a tmp-fixture e2e check that caught a real behavior should become a committed integration test (or a documented harness under `tests/<feature>/`), not a one-off you run and discard. A throwaway probe verifies *this* change; a committed test prevents the *next* regression. Promote it when the behavior is stable and the fixture is reusable; leave it a throwaway only for genuinely one-shot/environback checks.
  Push only after the local loop is green. This is the implementer's half of the "scale review to blast radius" rule ‚Äî local e2e + the incorporated test is what makes the reviewer's pass cheap and the CI run a formality.

- **Bailout commands are sub-section asides.** Surfaces that exist for completeness (`nub node`, `nub pm`) get a short `###` under the section whose implicit behavior they back up, never their own `##` ‚Äî "a Node version manager you should never need" framing.
- **Frame replacements as toolchain, not "npm packages."** What nub displaces is project-level tooling ‚Äî `nvm`, `corepack`, the PM CLI ‚Äî much of which was never an npm package. Keep the PM's two compat axes distinct: pnpm-compatible CLI surface; lockfile-compatible with whatever the project already uses (npm / pnpm / bun round-trip, yarn read-only). Concrete product comparisons are encouraged ("the uv experience, for Node"; "uv for Node" is a good framing).
- **Marketing asides use the styled treatment** (`.blog-prose blockquote`); default blockquote styling is unacceptable on the marketing site.
- **Version-gated claims must trace to code, not the wiki.** Every site claim about a Node-version-gated feature must map to a named constant/function in `crates/nub-core/src/node/flags.rs` or `spawn.rs` (no symbol ‚Üí the claim cannot ship). The wiki's `status: v0.1 "Confirmed"` tracks intent, not implementation ‚Äî copy written from it shipped false claims (WebSocket, webstorage, 2026-06-10).
- **"Every supported Node version" is a banned phrase** unless the feature is `typeof`-feature-detected in the shared preload. Flag-injected features are tier- and version-banded; state the floor per row instead of glossing it.
- **Brand-boundary copy.** "The brand stops at the binary boundary" is wiki-internal shorthand, never user-facing ‚Äî and absolute "the name nub never appears anywhere" claims are false (`node_modules/.nub`, `~/.cache/nub`, internal `NUB_*` plumbing vars, error codes). The public claim is about *public surfaces only*: no nub-branded API you import or call, no `nub:*` import namespace, no `@nub/*` scope, no `"nub"` config field you author. **"No `NUB_*` environment variables" is a fine public claim** ‚Äî in user-facing copy "`NUB_*` env vars" means vars *the user sets to change behavior*, and nub has none; its internal cross-process plumbing vars aren't user knobs and don't contradict it. Don't hedge that copy. The only thing to actually avoid is the absolute "the name nub never appears anywhere." The promise is "zero nub-specific *public* APIs / no lock-in."
- **Protective-refusal demos use real nub output.** Capture from `target/release/nub` or ground the exact string in `crates/nub-cli/src/pm_engine/` ‚Äî never invent. (The general pattern ‚Äî show the refusal as a feature, red ‚ùå on failure lines, ‚úì on success ‚Äî is in the shared guide.)

Every doc under `wiki/runtime/` and `wiki/commands/` carries a YAML front-matter block describing its status, mechanism, and dependencies. **Run the indexer before opening files individually:**

Front-matter schema:


Status values:

## Releasing

Nub is published to npm as `@nubjs/nub` with 8 platform-specific binary packages. The release is fully automated via GitHub Actions.

```bash
make version V=0.0.6          # sets version in all 9 npm packages + Cargo.toml
make version-check             # verify consistency
git add -A && git commit -m "v0.0.6"
git tag v0.0.6
git push origin main --tags    # CI builds 8 platforms, publishes to npm, creates GitHub release
```

**Makefile targets:**
- `make version V=<ver>` ‚Äî set version across all packages atomically.
- `make version-check` ‚Äî verify all package versions are consistent.
- `make npm-build` ‚Äî build + package for the current platform (local testing).
- `make npm-publish` ‚Äî publish all packages manually (prefer CI).
- `make npm-publish-dry` ‚Äî dry-run publish.

**CI release workflow** (`.github/workflows/release.yml`): triggers on `v*` tags. Builds for darwin-arm64, darwin-x64, linux-x64, linux-x64-musl, linux-arm64, linux-arm64-musl, win32-x64, win32-arm64. Publishes via npm OIDC trusted publishing (no secrets needed). Creates GitHub Release with binary artifacts.

**Version regime:** stay in `0.0.x` until ready for public launch. Bump to `0.1.0` only when the whitepaper, benchmarks, and install experience are polished.

## Reference checkouts under `.repos/`

`.repos/` holds local-only clones of other runtimes and tools used as reading material: `.repos/node/` (the Node source ‚Äî previously the contents of this repo's root, since Nub was originally a Node fork), `.repos/bun/`, `.repos/tsx/`, and `.repos/pnpm/` (the pnpm monorepo, which now absorbs pacquet's in-progress Rust port at `.repos/pnpm/pacquet/crates/`). They are gitignored and never modified. When a research question requires reading the actual source ("how does Node's `lib/internal/modules/esm/resolve.js` handle X?"; "how does `pnpm -r` implement `--resume-from`?"), read from the corresponding `.repos/<name>/` directory. The Nub repo itself contains no vendored Node source, no Node patches, and no upstream-Node remote ‚Äî see [`wiki/architecture.md#augmenter-not-fork`](wiki/architecture.md#augmenter-not-fork).

**Default to `.repos/` for third-party source research; clone eagerly; never WebFetch individual GitHub files.** First check whether the reference checkout already exists under `.repos/<name>` and read/search it locally. When you need another tool's source ‚Äî to compare an API surface, check a flag's behavior, or read an implementation ‚Äî `git clone --depth 1 <repo> .repos/<name>` and then `Read`/`Grep` it locally. Do **not** pull files one at a time off GitHub: a clone is cheap, gives you the entire tree to search, stays internally consistent, and is the established pattern here (`.repos/node`, `.repos/bun`, `.repos/tsx`, `.repos/pnpm`, `.repos/ni`, ‚Ä¶). Add a new reference clone the moment a question needs one ‚Äî don't fetch. Curl official docs when useful for current published behavior, but for source/implementation questions `.repos/` is the default starting point.

When citing pnpm's implementation in epic todos or research docs, link to the specific file in `.repos/pnpm/` (e.g. [`.repos/pnpm/exec/commands/src/runRecursive.ts`](.repos/pnpm/exec/commands/src/runRecursive.ts)). The canonical workflow for pnpm-parity work is: read the source first, then write a fixture against real pnpm, then verify nub matches.
