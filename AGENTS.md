## Work directly on `main` — no branches, no stashes, no worktrees

**All work happens on `main`.** This is a pre-release product with multiple agents often running in parallel, and the trunk is `main`. Do **not** `git checkout -b`, do **not** `git stash`, do **not** switch branches, and do **not** create git worktrees for isolation. Those operations on a shared working tree cause parallel agents to clobber each other's uncommitted work and orphan commits onto dead branches. The rule that prevents it: **commit small and often, directly to `main`.** Committed work cannot be clobbered; frequent commits shrink the window where loose uncommitted work can be lost to near-zero. If you need to preserve WIP, *commit* it — never stash it. Releases are tag-triggered and deliberate (see [Releasing](#releasing)), so a busy `main` with in-progress work is expected and fine.

## The repo is PUBLIC — never commit internal discussion (HIGH PRIORITY)

`github.com/nubjs/nub` is a **public open-source repo with real stars**. Everything committed — code, comments, commit messages, `tests/**` READMEs, every tracked markdown file — is world-readable forever, and **survives in git history even after you delete it**. So: **never put internal product / marketing / competitive discussion or considerations into a tracked file.** Banned from anything git tracks:

- **Benchmark-presentation strategy** — which competitors or numbers to show vs. omit, "include X only where we're competitive," competitive-band thresholds, anything about how results are framed for publication.
- **Candid hedges that read as gaming** — "load-bearing for honesty," "this is the headline," framing a default (e.g. GVS auto-disable for next/nuxt/parcel) as something to be sheepish about, or any sentence a competitor/skeptic could screenshot as "they juice their benchmarks."
- **Private attributions / quotes / first-person strategy** — by-name attributions, internal quotes, "our number," "don't publish X," "reads as cherry-picking." Public copy states what the code DOES, factually and neutrally — never *why a number was framed a certain way*.

**Commit messages are professional, factual, and neutral — NEVER braggy, competitive, or editorializing.** A commit message states WHAT CHANGED, not how great it is or how it stacks up against a competitor. Banned: competitive bragging ("nub clearly beats bun", "crushes pnpm", "fastest"), superlatives/hype, and any framing a skeptic could screenshot. Write `site: update warm-install bench to 36-run numbers`, NOT `site: … (nub clearly beats bun)`. Benchmark commits name the change (new run, new sample size, updated bars) — never the verdict. This applies to every commit subject + body; the rule is the same as for any tracked file (above), and commit messages are world-readable forever and cannot be cleanly scrubbed once pushed.

All internal deliberation lives in the **LOCAL-ONLY, gitignored** surfaces — `.fray/`, `epics/`, `wiki/` (none are published). Keep strategy, competitive analysis, and decision rationale there, never in a tracked source/doc/test file. When in doubt, leave it out of the commit.

## Model tiering for sub-agents (HIGH PRIORITY — apply to EVERY dispatch)

Two levers: WHICH model, and the `effort` parameter (often the bigger cost/quality knob on a fixed model). Cost ladder cheapest → priciest: **Haiku 4.5 < Sonnet 4.6 < Opus 4.8 < Fable 5** (Haiku→Fable ≈ 10× in/out). This applies to every sub-agent dispatch — single one-off calls as much as multi-agent workflows. Set `model:` per `agent()` call by cognitive load (JUDGMENT REQUIRED), not by task type:

- **Haiku** — fully-scripted mechanical only: run THESE exact commands, harvest THIS output, trace THIS path, where every decision is pre-made by you. Cannot self-steer — do not give Haiku an open-ended investigation and expect a reliable verdict. Give it a script, not a question. Caveats: 200K context (vs 1M for the others) — route >~150K-token jobs to Sonnet; and Haiku takes NO `effort`/adaptive-thinking param (passing them 400s).
- **Sonnet** — the supporting cast: differential probes requiring light judgment to recognize a divergence, test scaffolding, gates/settle/CI-watching, doc updates, mechanical-but-not-trivial edits. Effort `medium` default. (Trivial fix-loop iterations can stay here to save the Opus premium.)
- **Opus** — THE DEFAULT for actual software engineering: diagnosis and writing/fixing the code that LANDS, architecture & design, adversarial review, gnarly debugging. The ~1.67× premium over Sonnet is cheap because code is low-volume vs the Haiku-dominated grunt, and it is exactly where the quality bar lives. Effort `xhigh` for coding/agentic.
- **Fable** — only the very hardest synthesis/judgment (priciest; use sparingly). Effort `high` default.
- **Orchestrator (you)** — push real work DOWN to the cheapest capable tier, then VERIFY what the doing-agents produce against the quality bar. If you are on Fable (the expensive flagship), keep your own footprint lean: coordinate + delegate + spot-check, do not do heavy lifting yourself.

Pattern: *haiku gathers & packages → opus does the real engineering → sonnet handles the supporting cast → you verify.* Sub-agent prompts must be SELF-CONTAINED (a model switch starts a fresh cache — the orchestrator's cached prefix does not carry over). Re-verify cheap-tier load-bearing claims yourself — a Haiku "this is a security bug" is a *lead, not a fact*; confirm it against code or a foreground experiment before acting on it. Cheap-tier agents should emit an explicit escalation marker on low confidence so the orchestrator re-routes up a tier. Long Opus/Fable loops: use Task Budgets to cap spend; big Haiku fan-outs: consider the Batch API. Fuller detail in the `claude-api` skill. Do NOT spin up all-one-model workflows.

**Never `cp -r` the repo to isolate work** (a benchmark, a throwaway build, a sandbox). The tree carries `target/`, `.repos/`, and `node_modules` — *tens of GB* — so a recursive copy fills the disk. Isolate with `git clone --depth 1 "file://$PWD" <dest>` (tracked files only; add `--recurse-submodules` for `vendor/aube`) or `git archive`, and give any isolated build its own `CARGO_TARGET_DIR`. (A recursive copy of the multi-GB tree fills the disk.)

**Detecting a GitHub fork — check `.fork`, never an empty `.parent`.** `gh repo view <owner>/<repo> --json parent` can return a null/empty `parent` even for a genuine fork (that field isn't reliably populated through `repo view`). The authoritative signal is the boolean `.fork`. Use the REST API: `gh api repos/<owner>/<repo> --jq '{fork:.fork, parent:.parent.full_name, source:.source.full_name}'` — `.fork == true` means it's a fork, `.parent.full_name` is the immediate upstream, `.source.full_name` is the root. **Never conclude "not a fork" from an empty `parent` alone**, and never block a cross-repo PR on that (an empty `parent` does not mean the repo isn't a fork).

## Workflow lifecycle — never kill mid-edit; run queues as a continuous series (HIGH PRIORITY)

**NEVER `TaskStop` a workflow while it may be mid-file-edit.** Killing an agent mid-edit orphans uncommitted WIP — the exact stranded-work failure this repo's top rule warns about, and a mistake made repeatedly. Only stop at a SAFE point: the active agent has committed, or you are between phases. To shed load, prefer letting the running agent reach its commit over killing it. If you genuinely must stop mid-flight, the FIRST thing you do afterward is `git status` and **commit any buildable WIP (commit, don't stash)** — including inside `vendor/aube` + the pin — so nothing is lost. Workflows resume from committed state (`resumeFromRunId`), so the correct way to "pause" is to wait for a stopping point, not to kill a live editor.

**Get sign-off before launching an OPINIONATED workflow — one that bakes in a design decision.** Classify every workflow before you launch it. A BROAD INVESTIGATION (profile, explore, diagnose, gather — commits to no design and lands nothing consequential) is safe to launch autonomously. An OPINIONATED TASK / FIX / IMPL (prescribes a specific solution, or lands changes that touch an important design consideration — a default, a security posture, a product behavior, an API/config/env surface, an architecture) requires the user's guidance FIRST: flesh the design out together, and be explicit about whether the work is investigation-scope or opinionated-implementation-scope, before any agent is told to build it. Separately, a workflow prompt must NEVER empower a sub-agent to autonomously land a change to a default / security posture / product decision the user owns — those are recommend-only until the user signs off; the agent profiles and proposes, the user decides. An investigation prompt must not quietly carry a phase that prescribes-and-lands a design decision; the profiling is fine, the decision is a conversation first.

**Run a multi-item queue as a CONTINUOUS SERIES — one at a time, but never stalled.** Parallel fan-out burns the quota fast (three concurrent multi-agent workflows can exhaust even Max 20× in ~an hour), so run ONE workflow at a time — but the instant it finishes, launch/resume the next, driven by each completion notification, with a `ScheduleWakeup` heartbeat as the backup that catches stalls and survives quota resets. The failure to avoid is the series stalling after one item (it looks done; it isn't). Record the ordered sequence durably in the epic so any re-invocation — completion or heartbeat — can read "what's next" and continue with no further user prompting.

**The serialization rule is about BUDGET, not about clobbering — and the governor is WITHIN-workflow:**

The thing that burns the budget is **ONE workflow fanning out 20+ parallel Opus agents at once.** That is the failure mode to never create. The fix is internal serialization, NOT avoiding concurrency:

1. **WITHIN a workflow — exactly ONE sub-agent runs at a time. Absolute, non-negotiable.** A workflow (the `Workflow` tool, or an Interactive-Workflow effort) dispatches its sub-agents strictly one at a time: dispatch one, wait, verify, then the next — a serial waterfall with synthesis and staging between steps. NEVER `parallel()`/`pipeline()` fan-out inside one workflow. This single-agent-at-a-time discipline is the **budget governor**: it caps a workflow's heavy-agent draw at one. (It also keeps the human able to follow + intervene at every step.)

2. **ACROSS workflows — several MAY run concurrently. This is FINE and not gated.** Because each workflow is internally serial (governor above), running multiple internally-serial workflows side by side draws only a bounded number of concurrent heavy agents. Concurrent workflows are allowed by default. **There is NO fixed heavy-lane ceiling** (the old "≈5 heavy lanes" cap is removed — fan out as wide as the work genuinely parallelizes); the only real governor is **429 → back off**. The ONLY thing that is banned is the within-workflow fan-out in (1).

**Codebase-EDITING work parallelizes too — the clobbering fear was overblown (corrected 2026-06-12).** The file tools enforce read-before-write, reject stale edits, and write atomically, so concurrent edits to one tree do NOT silently clobber and a concurrent build reads a consistent old-or-new file. Do not blanket-serialize codebase work. Three NARROW guards are the whole constraint: (1) **destructive raw git ops** (`reset`/`checkout`/`stash`) on a shared tree bypass the file-tool safety and WILL wipe a sibling's WIP — forbid them in parallel-agent prompts (stage only own files, commit fast); (2) an agent whose DELIVERABLE is a *trustworthy build/test result* (a benchmark, or a `vendor/aube` patch proving it passes tests + preserves upstream behavior) needs its OWN isolated tree — give it a `git clone --depth 1 file://…` + its own `CARGO_TARGET_DIR` so a sibling's edits can't contaminate its result; that is HOW you parallelize landing work, not a reason to serialize it; (3) serialize only the final **assembly/commit onto the shared branch** (cherry-pick/apply the isolated patches onto `nub-fork`, bump the pin once, integrated-test once).

And tier EVERY agent by the model-selection guidelines (judgment / the-fix-that-lands → Opus; supporting cast / mechanical-but-not-trivial → Sonnet; fully-scripted harvest → Haiku) — choose per agent, every time.

## fray (orchestrator-first, sub-agents as instruments) — a named methodology (HIGH PRIORITY)

Full methodology: `.claude/skills/fray/SKILL.md`.

This is **fray** — the preferred way to run a large, mixed set of efforts (investigations + decided fixes + verifications) toward a goal like a launch. Reference it by name. It is the default for any multi-effort push where the work is part *find-out-what's-true* and part *land-the-decided-thing*, and where the human wants to stay in the loop on the decisions the investigations surface. Use it **instead of the `Workflow` tool** for these — workflows hardcode a DAG up front and fan out hard (expensive, and they bury the decision points); fray stays cheap and dynamic by driving everything from the main session with individually-dispatched sub-agents.

**The operational essentials:**

- **You (the main session) are the orchestrator, and the ONLY decider.** You hold the whole picture, dispatch sub-agents as *instruments*, ingest what they return, and decide the next move. You do not hand the steering wheel to a workflow script.
- **A living tracker document is your control surface.** One markdown doc (e.g. `epics/<epic>/HANDOFF.md` or `STATUS.md`) holds: a status board (one row per effort), a detail card per effort, a **Questions-awaiting-human** queue, and a reusable **codebase map** so future dispatches are cheap. You OWN this file — after every sub-agent returns, fold its facts into the card, advance the status, and re-derive "what's next" from the board. The doc is extensible by design: new efforts get a new ID and a new row. Status vocab: `todo · probing · needs-decision · ready · landing · verifying · done · blocked`.
- **CRITICAL: only the orchestrator edits the tracker/handoff doc.** Sub-agents must NEVER directly modify it — not the HANDOFF.md, not any STATUS.md used as the control surface. Sub-agents return results to you; you update the doc. Parallel agents writing the same tracker clobber each other's updates, which is how efforts get lost and decisions disappear.
- **NEVER put verbatim quotes from the user into ANY document** (fray threads, tracker docs, research write-ups, code comments, commit messages — anything). The user dislikes seeing their own words quoted back. Record the *decision or fact*, paraphrased in neutral third-person ("Decided: bump to 98.8" / "Direction: add cafile support"), never `User: "..."` or a `"..."` block attributed to them. This holds for LOCAL docs too, not just tracked/public files (that's the separate brand-boundary rule). Capture what was decided, not who said it or how they phrased it.
- **Sub-agents are instruments, not deciders.** A probe returns *facts* — divergences, traces, measurements — not verdicts. No sub-agent autonomously lands a change to a default / security posture / product behavior / API-config-env surface / error-contract; those route back to the human as a question. Mechanical / clearly-a-bug fixes may land (you review the diff). Every sub-agent prompt is **self-contained** (a model switch starts a fresh cache); embed the relevant slice of the codebase map so the agent doesn't re-derive it.
- **Model-tier every dispatch by JUDGMENT REQUIRED, not just task type.** Haiku *cannot self-steer* — give it only fully-scripted mechanical work where every decision is pre-made. A differential probe that must recognize a divergence and judge real-bug-vs-intended needs judgment → Sonnet (or Opus), never Haiku. Opus does the fix that lands; you verify. Re-verify cheap-tier load-bearing claims yourself — a Haiku verdict is a *lead, not a fact*. Background sub-agents are HEADLESS (no TTY): they cannot drive a REPL or answer a permission prompt; any interactive-surface probe must run in the orchestrator's FOREGROUND. (See the model-tiering section above and the full skill for detail.)
- **Steer in-flight agents with `SendMessage` (confirmed working — use it extensively).** You can enqueue a message to a RUNNING background sub-agent (delivered at its next tool round) or resume a COMPLETED one with full prior context. Reach for it constantly: fold a new follow-up into the agent already editing a file (preserves one-editor-per-file), feed a just-landed decision or superseding fact to every running agent it affects, answer a question an agent surfaced (it messages `to:"main"`), or redirect/re-scope mid-flight — instead of killing an agent (orphans WIP) or spawning a clobbering sibling. When a new ask arrives and an agent already owns that surface, message it rather than dispatching fresh. Reserve the `enqueued` thread-status for a *separate* dispatch that genuinely must wait for a named agent to finish.
- **Eagerly ask, and re-surface.** Push questions to the human the moment an investigation surfaces one — both in the tracker's queue and in chat. If a question goes unanswered, assume it was *missed, not declined*, and re-surface it the next turn.
- **Parallelize aggressively — codebase-editing work included.** Read-only probes parallelize without limit. Landing agents that EDIT code parallelize too: the file tools enforce read-before-write, reject stale edits, and write atomically, so concurrent edits to one tree can't silently clobber and a concurrent build reads a consistent old-or-new file (at worst a spurious re-run it retries past) — NOT a reason to serialize. The narrow guards: (1) forbid destructive raw git ops (`reset`/`checkout`/`stash`) in parallel-agent prompts — they bypass the file-tool safety and wipe a sibling's WIP; (2) an agent whose deliverable is a *trustworthy build/test result* on `vendor/aube` (does my patch pass tests + preserve upstream behavior?) gets its OWN isolated clone + `CARGO_TARGET_DIR` so siblings can't contaminate the result — that is how you fan out fork-patch landing work; (3) serialize only the final assembly of those isolated patches onto `nub-fork` (apply, bump the pin once, integrated-test once). Everything else fans out.
- **Dynamic, not pre-planned.** There is no committed step list to march through. Each turn: read the board, ingest returns, surface new questions, dispatch the next round honoring deps + the build-correctness constraint, update the board. The human can re-prioritize or stop after any round — the tracker means nothing is lost.

The win over a hardcoded workflow: the human answers the questions the investigations raise *as they arise*, spend stays low (cheap sub-agents, no fan-out tax), and the orchestrator keeps a coherent mental model of the whole effort instead of delegating it to a script.

## What Nub is, briefly

- **A Rust CLI that orchestrates the user's installed Node.** v0 is a fast script runner plus a TypeScript-just-works runtime. Mechanism: Node's own extension surfaces — `module.registerHooks()`, `--import` preload, `--require`, env vars, N-API addons, V8 flag injection.
- **TypeScript-first developer supertool.** Own transpiler (oxc-based). TS, JSX, non-erasable syntax, `emitDecoratorMetadata` all supported.
- **Compatibility is paramount.** Code targeting Node must run on Nub byte-for-byte. The contract: **default mode = Nub runtime augmentation active; `NODE_COMPAT=1` or `--node` = augmentation disabled, plain Node behavior**.
- **Behavioral additivity** is the rule for augmentation — [`wiki/philosophy.md#additivity`](wiki/philosophy.md#additivity) for the canonical statement. "Additive" means doesn't break code written for vanilla Node; mechanism is restricted to Node's own extension surfaces (not source patches).

## The augmenter / fork distinction is load-bearing

A previous direction (2026-05-17) positioned Nub as a soft fork of Node, with patches to Node source on the table. **That direction was reversed 2026-05-18.** Nub is now a Rust CLI augmenter via Node's extension surfaces only. Source patches are off the table.

If you're editing a plan doc and find "soft fork" or "patched Node" language, it predates 2026-05-18 and should be updated. The authoritative statement of the current direction is [`wiki/architecture.md#augmenter-not-fork`](wiki/architecture.md#augmenter-not-fork).

The reason the distinction is load-bearing for agents specifically: implementation approaches that *sound* clean — "just patch `lib/internal/modules/esm/resolve.js` to do X" — were valid under the old posture and are no longer. The current mechanism rule is **"would a user on plain Node, plus the corresponding `module.register()` / `--import` / npm-addon, get the same result?"** If yes, in scope. If no, find a different mechanism or scope the feature out.

### The brand boundary — public surfaces only (internals are exempt)

The brand boundary protects **public-facing surfaces only** — anything a user imports, installs, types, or authors as configuration, or any API they call by a documented name. **Internal mechanisms are explicitly exempt:** error/warning codes, environment variables Nub sets for its own cross-process / shim plumbing, internal `globalThis.__nub*` sentinels, and internal cache/dir names — the brand may leak there freely and chasing it out is wasted effort. (Error codes and the internal cross-process env vars Nub sets to change shim behavior are internal mechanism; the concern is only the brand leaking into public-facing APIs — don't over-police the boundary for internals.) The rules below govern the *public* surface and are absolute there — do not add a public nub-branded API even "behind a flag."

- **No PUBLIC `globalThis.nub`.** Nub never injects a *documented, user-callable* global named after itself; `Bun.*`-style conveniences, if they exist at all, go in importable modules. Internal markers are FINE — a `globalThis.__nub*` sentinel used for runtime/cross-process plumbing that user code never references is not a leak (reversed 2026-06-11; the old "no internal markers either" was too strict).
- **No `nub:*` module namespace.** Nub never registers synthetic specifiers users would `import` (`nub:test`, `nub:sqlite`, `nub:serve`). The `node:*` namespace is upstream Node's and closed; Nub does not squat a sibling. (Public surface — users type it.)
- **No `@nub/*` npm scope** for user-facing packages or APIs — the bare `@nub` scope is unused entirely. The CLI ships under the project's own `@nubjs` org: `@nubjs/nub` (the package users `npm install -g`) plus its `@nubjs/nub-<platform>` `optionalDependencies`, which carry the per-platform binary + N-API addon and are install-time plumbing never imported by user code (same shape as `@biomejs/biome` + `@biomejs/cli-*`, `@rollup/rollup-*`, `@esbuild/*`). Nub's ambient TypeScript declarations ship as `@nubjs/types` — our own org, a types-only devDep, not a user-imported API. (We do not own the npm package `nub`, so `@types/nub` on DefinitelyTyped is not a viable home.) `@nubjs` is deliberately distinct from the prohibited bare `@nub`. See [`wiki/philosophy.md#the-brand-boundary`](wiki/philosophy.md#the-brand-boundary) for the full rationale.
- **Internal `NUB_*` env vars are FINE; never a brand env var as a documented USER knob.** Reversed 2026-06-11: Nub MAY set `NUB_*` / `__NUB_*` environment variables for its own cross-process plumbing — telling a child shim how to behave, detection sentinels, debug toggles — these are internal mechanism, not a public API, and the old "no `NUB_*` ever, it shows up in `printenv`" rule was over-strict. What remains: don't *document* a `NUB_*` var as a knob users are meant to set, and for genuinely user-facing toggles prefer `NODE_*` (where Node doesn't claim the name, e.g. `NODE_COMPAT`) and neutral conventions (`NO_COLOR`, `FORCE_COLOR`, `PORT`, `NODE_ENV`).
- **The nub runtime respects ZERO `AUBE_*` env vars — the embedded engine's brand is never nub's config surface (nub is the package manager).** DISTINCT from the exemption above. That exemption covers internal vars nub *sets* for its own children (detection sentinels, shim signaling — a user never touches them). THIS rule covers vars the nub runtime *reads from the environment to decide behavior* — those are user-facing knobs, and when aube runs *as nub* the user sees nub while aube is an invisible implementation detail, so an `AUBE_`-branded knob is a brand leak. aube currently exposes ~30 such runtime vars (`AUBE_DISABLE_CLONEDIR` — already documented as the clonefile opt-out — `AUBE_CONCURRENCY`, `AUBE_CACHE_DIR`, `AUBE_NO_UPDATE_CHECK`, `AUBE_PRIMER_PICK_GATE`, the `AUBE_DISABLE_*` perf toggles, `AUBE_DIAG_*`, `AUBE_SELF_*`/`AUBE_VERSIONS_HOST`, …); under nub, none of these may be the surface. Surface preference (now that nub fully owns package management, `NUB_*` is legitimate, no longer a last resort): use a **neutral npmrc field** where one reads naturally (config any tool could honor — also reachable as `npm_config_<field>`), else a **`NUB_*`** env var / nub-namespaced config is FINE for nub's own PM knobs (e.g. `NUB_CACHE_DIR`, `NUB_PRIMER_TTL`). The hard line stays: **never `AUBE_*`** (the embedded engine's brand) for a nub-facing knob. Two disciplines on top: (a) **keep the exposed set as SMALL as possible** — every knob is a maintenance + doc + support surface; expose one only on real demand, default the rest internal or dropped; (b) **design each knob to GENERALIZE, not special-case** — a `NUB_PRIMER_TTL` *duration* (default unlimited), never a boolean `primer_evergreen` flag (config should be as generalizable as possible; a binary is a degenerate special-case of the right abstraction). MECHANISM (structural fix, not var-by-var whack-a-mole): extend the **Embedder profile** — the same compile-time pluggability jdx merged in #862 — with an env-prefix / env-resolution hook, so standalone aube keeps reading `AUBE_*` while the NUB profile reads neutral/`npm_config_*`/`NUB_*`; the boundary is enforced once, structurally. (Build-time `AUBE_*` knobs in aube's `build.rs` — `AUBE_PRIMER_TOP`, `AUBE_REQUIRE_PRIMER`, `AUBE_PRIMER_GENERATED_AT` — are aube's own build machinery our CI sets; out of scope for this *runtime* rule, a low-priority consistency cleanup at most.) Audit + migration tracked as the `brand-boundary-env` thread.
- **Prefer neutral names for USER-AUTHORED config (a public surface).** A `package.json` field or config file a user writes is public, so Nub reads a *neutral-named* `tasks.json` / `"tasks"` field rather than a `"nub"` field or `nub.toml` — the config a user authors shouldn't be brand-locked (the `"workspaces"` precedent: yarn coined a generic field, npm/bun adopted it). This is a public-config preference, distinct from the internal-exemption above. Carve-out: Nub may adopt a vendor-neutral, any-tool-could-implement-it field by decision record — currently only `"tasks"` ([`wiki/commands/tasks.md`](wiki/commands/tasks.md)).
- **The brand boundary is SYMMETRIC — a true nub project consumes only un-branded, cross-tool config.** Just as Nub never *emits* its own brand into your config (no `"nub"` field), **when Nub is the project's active PM it never *consumes* config branded with another package manager** — `pnpm.overrides`, `pnpm.*`, yarn-namespaced config, Bun-branded fields, etc. are all ignored. A nub-identity project respects only the neutral fields any tool could read: `overrides`, `resolutions`, `catalog`, `workspaces`. (When the inferred active PM is npm/pnpm/yarn/bun — a *compat* project, not a nub project — Nub MIRRORS that PM faithfully, *including* its branded fields, because the project genuinely is that PM's. The no-other-brand rule is specifically for nub-identity projects.) This is the config-side of the brand boundary; the mechanism lives in the config-scoping policy (`crates/nub-cli/src/pm_engine/config_scope.rs`, "mirror the active PM, never silently").
- **pnpm-NAMED files and paths are NEVER read unless pnpm is the incumbent PM (non-negotiable).** When the project's incumbent package manager is NOT pnpm, Nub does not read any file or path that contains the literal string "pnpm" in its name — this includes `pnpm-workspace.yaml`, `.pnpmfile.cjs`, `.pnpmfile.mjs`, `.pnpmrc`, `~/.config/pnpm/` (including `~/.config/pnpm/auth.ini`), and any `pnpm.*` field in `package.json`. The gate is on the **NAME**, not the semantic effect — "it's auth-only / harmless / no resolution impact" exempts nothing. Crucial distinction: **pnpm-NAMED ≠ pnpm-specific.** A *generically-named* field that only pnpm happens to support (e.g. `overrides`) is a separate, case-by-case decision — it is not pnpm-branded by name and may be honored in a nub-identity project. The file-name half of the symmetric boundary: just as Nub never emits pnpm-branded names into your config, it never reads them unless pnpm is provably the active PM.
- **Error codes are internal — don't brand-sweep them.** `ERR_NUB_*` / `ERR_AUBE_*` / `WARN_*` identifiers (and the engine errors that surface them) are internal mechanism; a brand string in a code or message is acceptable and not worth chasing. *Adding* a missing code is good UX; rebranding an existing one for purity is not required.
- **No vendored Node patches.** Nub does not patch Node source, ship a custom-built Node binary, or embed `libnode`. Every augmentation rides on Node's public extension surfaces. (This is an architecture/additivity rule, not a brand rule.) If a feature can't be expressed through those, find a different feature.

### `nub`-named npm packages, explained

Two families of `nub`-named packages exist on npm. Neither is the prohibited bare `@nub/*` scope, and neither is user-facing application code:

1. **`@nubjs/types`** — Nub's ambient TypeScript declarations ship as a types-only package under the `@nubjs` org we own and control. (We do NOT own the npm package `nub`, so `@types/nub` on DefinitelyTyped is not a viable home — `@types/<pkg>` maps to an npm package `<pkg>` we don't control, and DefinitelyTyped is community-controlled. The `@types/*` auto-load is also not the right pickup mechanism.) `@nubjs/types` is the correct home for any declarations user code references — `import.meta.hot` for hot-mode consumers, Nub-specific globals, etc. — as a devDep, never a runtime import. Brand-clean: under our own org, no bare `@nub` claim.

2. **`@nubjs/nub` + `@nubjs/nub-<platform>`** — the CLI itself and its platform-specific binary distribution packages (`@nubjs/nub-darwin-arm64`, `@nubjs/nub-linux-x64`, … — 8 platforms total). `@nubjs/nub` is the package users `npm install -g`; the `@nubjs/nub-<platform>` packages are its `optionalDependencies`, each containing the compiled Rust binary + N-API addon for one platform, selected at install time by npm's `os`/`cpu` filters and copied into place by `postinstall.js`. They are install-time plumbing — never `import`ed, never referenced in application source, never a thing users type. Same distribution pattern as `@biomejs/biome` + `@biomejs/cli-*`, `@oxlint/cli-*`, `@rollup/rollup-*`, `@esbuild/*`. The org scope is `@nubjs` (the project's own npm org), deliberately distinct from the prohibited bare `@nub`.

## The aube vendoring + upstreaming — MEMORIZE THIS (forgotten twice; ground every aube claim here, not memory)

nub's package manager IS the vendored **aube** engine, embedded as a Rust LIBRARY. The structure (re-read this before asserting anything about how the integration works):

- **Embed, not subprocess.** `vendor/aube` is a path dependency — a checkout of *our fork*. nub has its OWN CLI (clap dispatch + `pm_engine::ENGINE_VERBS`) and calls `aube::commands::<verb>::run(typed_opts)` IN-PROCESS. There is NO aube subprocess and NO aube CLI under nub; aube's own `cli_main` + tool subcommands are DEAD under nub. All engine output is rebranded through `crates/nub-cli/src/pm_engine/present.rs` (`ERR_AUBE_*`→`ERR_NUB_*`, `aube`→`nub`, jdx URLs stripped).
- **Fork vs upstream.** The fork is **`nubjs/aube`**; upstream is **`jdx/aube`**. `vendor/aube` is pinned to a fork commit (`nub-fork` branch); bumping the pin updates the embedded engine.
- **UPSTREAMING — contribution PRs target `jdx/aube` (UPSTREAM), NEVER the fork (the thing forgotten 2026-06-14).** A contribution PR is `jdx/aube:main ← nubjs/aube:<branch>` (e.g. #888 = `embedder-profile-fixes`; #897 = the `process-exit-sweep` stack). Do NOT open a fork-internal `nubjs/aube ← nubjs/aube` PR to upstream — it upstreams nothing. **Stacked PRs also target `jdx/aube`** (base `main`; note the stack + which commit to review in the body; the diff auto-cleans once the parent PR merges). Cross-fork stack creation: `gh pr create --repo jdx/aube --base main --head nubjs:<branch>`.
- **What upstreams vs stays fork-only:** PLUGGABILITY / additive changes that are NO-OP for standalone aube (the embedder profile, env-resolution hooks, the `prog()`/`cmd()` source-branding helpers, the exit-code sweep) → upstream to jdx/aube. MATERIAL PM-behavior / nub-specific changes (primer-TTL, etc.) → stay in the fork (merge-synced on main, never rebased — see the workflow below), upstreamed later only if jdx wants them.
- **The embedder profile** (jdx merged it as #862): compile-time pluggability. nub sets its own profile (`cache_namespace` "nub/pm", `env_prefix`/`config_env_prefix`, banner-gating, pnpm-surface off, `read_branded_pnpm_config` gating). The NUB profile is what makes the brand + config boundary hold; standalone aube keeps its `AUBE_*` behavior.
- **Fork-discipline for EVERY vendor/aube change:** default-preserving (standalone aube byte-for-byte unchanged on the default path; the embedder opts in) OR a justified latent-bug-fix jdx would accept unconditionally. A trustworthy-build/test deliverable gets its OWN isolated clone + `CARGO_TARGET_DIR`; the final assembly onto `nub-fork` + the pin bump is serialized.

### The canonical `nub-fork` workflow — commits, not patches; push before pin (MEMORIZE; an inconsistency here just caused a real CI-killing bug, 2026-06-15)

This is the durable, non-negotiable convention for every change to the vendored engine. Be totally consistent — the commits-vs-patches drift caused a pin pointing at an unpushed commit that would have died at CI submodule-init.

- **ONE blessed branch = all our aube work, as real COMMITS.** `nubjs/aube` branch `nub-fork` is the single source of truth holding every nub-specific change to the embedded engine. Changes land as **real git commits on `nub-fork`** — NEVER as parked `.patch` files (`git format-patch` dumped into `.fray/…` "to apply later"). Commits, not patches: a transient patch while moving a commit between clones is fine; a `.patch` as the *resting representation* of fork work is not. (Companion memory: `aube-fork-changes-as-commits`.)
- **The pin tracks a `nub-fork` commit, and you PUSH `nub-fork` BEFORE bumping the pin.** The `vendor/aube` submodule gitlink (the "pin") must point at a commit ALREADY on the `nub-fork` remote. Bumping the pin to a local-only/unpushed commit makes every CI job die at submodule-init. The order is absolute: **push `nub-fork` → bump the pin → push `main`.** (Reinforces `vendor-aube-pin-push-order`.)
- **Sync with upstream by MERGE, never rebase.** Keep `nub-fork` current with `jdx/aube:main` via `git merge upstream/main` (resolve conflicts preferring upstream for files we don't own). Do NOT rebase `nub-fork` — rebasing rewrites SHAs, forces a force-push, and invalidates the pin on every sync. Merge keeps SHAs stable so the pin stays valid; this is the standard long-lived-downstream pattern.
- **Upstreaming: cherry-pick onto a FRESH branch off `jdx/aube:main`, PR, then converge on merge.** To contribute a fork change upstream, cherry-pick the specific commit(s) onto a fresh branch cut off `jdx/aube:main` (NOT off `nub-fork` — so the PR diff is only that change, not the whole fork delta) and open the PR to `jdx/aube:main` (e.g. audit #900). The change is then two commits — one on `nub-fork` (ships to users), one on the upstream branch (for jdx). When jdx merges, the next `merge upstream/main` into `nub-fork` CONVERGES it: git dedups identical content (or a trivial conflict if jdx tweaked it in review — resolve by taking upstream, dropping the now-redundant fork commit). The change graduates OUT of the fork-only delta, keeping `nub-fork`'s diff-vs-upstream as thin as possible.
- **Parallel fork work:** an agent making a fork change works in a clone of `nub-fork` (or the live `vendor/aube`), commits, and PUSHES to `nub-fork`; the orchestrator fetches and bumps the pin. The serialized step is only that final **pin-bump + `main` push** (one assembly: apply/converge the pushed commits, bump the pin once, integrated-test once) — the fork edits themselves parallelize (each in its own clone + `CARGO_TARGET_DIR` when the deliverable is a trustworthy build/test result).

## Testing philosophy

**Minimum number of tests, comprehensively covering the API surface.** This is the load-bearing rule for the v0.1 implementation, and the antidote to AI-generated test bloat. Quality of coverage matters; volume of tests does not.

- **Comprehensive coverage, not exhaustive coverage.** Each behavior gets tested once, well, against the cases that matter: golden path, the one or two failure modes that have user-facing implications, the boundary condition that's easy to get wrong. Stop there. Don't write `it("should not throw when input is null")` after you've already written `it("rejects null input")`.
- **Hand-crafted feel.** Tests must read as if a careful engineer wrote them, not as if an agent enumerated possibilities. Concrete signs of agent bloat to avoid: identical assertions across multiple `describe` blocks with different setup; per-input parametrization where one assertion would do; test names that paraphrase the implementation instead of describing the contract; descriptions like "should handle X" without naming what "handle" means.
- **Carefully considered abstractions.** Test helpers and fixtures are part of the contract. Don't bury behavior in clever shared setup; don't copy-paste either. Pick the shape that lets a reader skim a test file in 30 seconds and understand what's being verified. If a test file is over 300 lines, you're probably testing too many things in one place or each test is too verbose — split or trim.
- **Some things are untestable, and that's fine.** Perf-shaped behavior, OS-specific corners, race conditions that require infrastructure we don't have, behaviors that depend on Node's internal scheduling. Leaving them untested is honest; writing ceremonial fake tests for them is sloppification. Document the gap in a comment if it'll surprise a future reader.
- **Pull from upstream where it makes sense.** Node ships an extensive test suite at `nodejs/node/test/`. The executable-level subset (tests that just spawn `node` and assert behavior) is the strongest possible compat-validation against Nub. We run it black-box in two modes: `nub --node` (expected ~100% pass — passthrough mode) and `nub` augmented (expected near-100% with documented divergences). See `wiki/research/node-test-suite-leverage.md` for the harness design and CI cadence.
- **Test names describe the contract, not the implementation.** `test("emits ECONNRESET when peer closes mid-write")` is good; `test("emits ECONNRESET")` is too short to mean anything; `test("calls socket.destroy() with the EconnReset constructor when..."` is paraphrasing the code.
- **Failure messages must be self-debugging.** When a test fails in CI, the assertion message should make the cause obvious without rerunning locally. Asserting `expect(result).toEqual({ ok: true })` is fine when it passes and useless when it fails ("expected {ok: true}, got {ok: false}"). Use `expect(result.ok).toBe(true)` or write a custom message.

## Docker is available — use it instead of declaring things "untestable"

**Docker may be available.** Before you write off a behavior as "hard to test" or "can't verify locally," check whether a container closes the gap — several things the testing philosophy lists as untestable corners actually *are* reachable through Docker:

- **A clean, dependency-free environment.** A fresh `node:22.15-slim` (or no-Node `debian:slim`) container with no `~/.cache/nub`, no global Node, no local dev state — the honest way to test first-run install, the curl `install.sh` flow, `nub upgrade`'s `~/.nub` channel, provisioning a Node from `nodejs.org`, and "works on a clean machine."
- **A specific / floor Node version.** Pin `node:22.15` to exercise the **floor-only** defects a modern host Node (24/26) masks — `using`/`await using` down-leveling, the Temporal `toTemporalInstant` path, version-gated polyfills, the async-tier `module.register` loader. This is the local complement to the Node-22.15 CI leg.
- **Linux-specific paths.** musl-vs-glibc detection (`node:22-alpine` vs `node:22-slim`), arch matching, signal/`PATH` behavior on Linux.

**What Docker does NOT give you:** it runs **Linux containers only** (on a macOS/arm64 host → `linux/aarch64` by default, `--platform linux/amd64` via slow QEMU emulation). **Windows containers require a Windows host**, so cmd.exe behavior, the `--shell-emulator`, `.cmd`/`.bat` resolution, and `nub.exe` vs `bin/nub` still can only be verified on the **`windows-latest` CI leg** — Docker is not a substitute there. When an item's *Verify* says "on Windows," it means Windows CI; do not claim a Docker run verified it.

Keep containers ephemeral (`docker run --rm`), mount the repo read-only where you only need to read it, and never leave long-running containers behind.

## Probing methodology — differential fixtures, empirical over source (learned 2026-06-10)

The highest-yield way to find correctness bugs in nub is a **differential fixture**: a minimal fixture isolating ONE behavior, run against nub AND the reference tools it claims parity with (npm / pnpm / bun / node) on identical inputs — a divergence is the finding. This consistently beats happy-path app rounds: days of running real apps under nub surfaced almost nothing, while one five-minute `.env`-visibility fixture found two real bugs (a secret-leak over-reach where nub injected `.env` into non-node tools, and a NODE_ENV-cascade bug). Build the fixture, run every relevant tool, diff. Always compare against the thing you assert parity with — "nub does X" is unverified until "…and npm/pnpm/bun do Y on the same fixture" is in hand.

Three supporting disciplines, each earned the hard way:

- **Test empirically before reading source, and before deciding.** A throwaway fixture in a sandbox answers "what does it actually do?" faster and more reliably than tracing Rust/Zig. When a decision hinges on a tool's real behavior, reproduce it — don't infer it from code.
- **Read the COMMENTS and the resolution, not just the issue body.** A bug's body says what someone wanted; the maintainer comments and *why it was closed* say what's true. An "open feature request" can be a deliberate rejection with the real rationale in the thread — Bun's auto-`.env`-for-scripts looked like unmet demand in the body, but the comments showed it was intentionally rejected, with the core maintainer's root-cause analysis. Bodies alone will make you confidently wrong.
- **Ground every claim about the system in code or an experiment — never memory.** A confidently-designed probe this session referenced `nub test`, which is not a command. Verify the actual surface (read `cli.rs`, or run it) before you build against it.

And: **reversing a decision on new evidence is correct, not flip-flopping.** Architecture calls routinely dissolve or flip once the facts arrive — a "minimal vs elegant" split collapsed when investigation showed the elegant option was also the only *correct* one; a "keep this feature / lean in" call reversed twice as comments and a fixture came in. Present the decision, then keep probing; let evidence move you rather than defending the first answer.

## Iterating across Node versions and tiers

nub's behavior splits by **tier** — the fast tier (Node 22.15+, sync `module.registerHooks`) and the compat tier (18.19–22.14, async loader-worker via `module.register`) take different code paths and break differently — so a green run on a single modern Node (often 26) routinely masks compat-tier and floor-only defects. nub discovers its Node from `PATH`, so the cheapest way to drive it onto a specific version is `PATH="$HOME/.nvm/versions/node/v20.19.0/bin:$PATH" nub …`; sweep several to cover both tiers, and reach for Docker (per the section above) for Linux + floor confirmation before trusting a result. **Do not claim cross-version support from one Node.**

Feature-specific harnesses that encode this loop live under `tests/<feature>/` — e.g. `tests/pnp/` (Yarn PnP) builds a real fixture (`make-fixture.sh`), runs a scenario matrix across Node versions on the host (`run-pnp-matrix.sh`) and on Linux in per-version containers (`docker-matrix.sh`), and documents the whole loop + the tier model in `tests/pnp/README.md`. When you build or extend a system for iterating on a hard-to-unit-test feature, document it in-place like that (a `README.md` next to the scripts, plus comments in the scripts themselves) so the next agent reproduces it in minutes instead of rediscovering it.

## Chrome DevTools — use it eagerly to evaluate UI

**A `chrome-devtools` MCP server may be configured, and the marketing site dev server runs on `localhost:3000`.** Whenever you touch the site (`site/`) — any layout, spacing, color, typography, or copy change that renders — do not declare it done by reading the JSX. Drive the actual browser and look: navigate/reload the page, take a screenshot, and visually confirm the change is correct. For anything alignment- or spacing-sensitive, don't trust the eyeball alone — measure with `evaluate_script` (`getBoundingClientRect`, `getComputedStyle`) so the assessment is grounded in real numbers, not a guess. Geometry checks and visual checks catch different bugs: a box can be mathematically centered while serif font metrics make the ink ride visually high, so use both. Also check `list_console_messages` for runtime errors after a change — a green HTTP 200 is not proof the page didn't throw. The cost of a screenshot is trivial; shipping a UI change you never looked at is not.

## Implementation quality discipline

**Quality over velocity. Always.** Don't move fast, check boxes, and ship stubs as complete implementations. The specific failure modes:

- **Never mark a task `[x]` without verifying the behavior end-to-end.** Running `cargo test` is necessary but not sufficient. If the task says "implement per-line stream prefixing matching pnpm," you must run `nub -r --stream run build` on a real fixture and compare the output to `pnpm -r --stream run build`. A green test suite with a stubbed implementation is worse than an unchecked task.
- **Never claim "parity" without evidence.** "Complete workspace parity with pnpm" means every flag, every edge case, every output format. If you haven't tested a flag, don't mark it done. If the implementation is a simplified version, say so explicitly in the task note — don't use language like "implemented" for something that's "scaffolded."
- **Name what you actually built.** If you shipped `Stdio::inherit()` with a header line, that's not "stream prefixing." If you partitioned into fixed batches, that's not "work-stealing." Use precise language so the next reader knows the actual state.
- **If you're completing 10 tasks per hour, you're not testing them.** A thoughtfully implemented and tested task takes 15-30 minutes. If a phase has 12 tasks and you completed it in 20 minutes, audit yourself.

## Research eagerly

For non-trivial technical or strategic questions, spawn a research sub-agent rather than guessing. Each meaningful research thread produces a write-up under `wiki/research/<topic>.md`.

**Research docs are canonical living documents.** When findings update, supersede earlier conclusions, or reverse on new evidence, edit the doc in place — but record the change at the bottom in a `## Changelog` section as a dated bullet. Format: `- YYYY-MM-DD — <what changed and why>`. The first entry on a new doc is `- YYYY-MM-DD — Initial write-up.` Major reversals get a leading `**REVERSAL:**` marker so future agents skimming the changelog see the shift immediately. Don't blow away prior conclusions silently — note that the doc previously said X and now says Y, and what evidence drove the update.

This matters because the codebase has gone through posture shifts (soft-fork-on, soft-fork-off, etc.). The git history of the research doc plus its changelog tell you the *shape of the evolution*; the doc body always reflects the current best understanding. When two research docs cover overlapping territory, prefer merging into a single canonical and superseding the older — record the merge in the surviving doc's changelog and note the supersession at the top of the doc being absorbed before deleting it (or just delete and reference the merge in the survivor).

## User-facing docs (`site/content/docs/`): terse, command-headed

Register: zod.dev — to the point, code-first, no marketing fluff inside docs pages. **Any GENERAL copy-style feedback gets applied EVERYWHERE, not just at the one spot it was raised.** When the feedback is a general writing rule (not a one-off fix to a single sentence), dispatch a sub-agent to sweep ALL docs (and the homepage/blog where it applies) and integrate the rule — then record the rule here so it sticks. **Never start a sentence with inline code (see the absolute rule in the Blog & marketing section) — lead with prose, put the `code` inside the sentence.** **Section headings are the flag/subcommand spellings, not prose, wherever a section maps cleanly onto one:** `## --filter`, `## nub pm use`, `## --node` — never "Filter packages" / "Pin a version". Prose headings are for sections with no command surface (concepts like "Lifecycle hooks", "What triggers a restart"). Same rule for variants: nest selector/sub-syntax as `###` under the owning flag. **A heading that IS a code token is rendered as inline code — backtick the whole token (any heading that should be inline code is inline code, e.g. `baseUrl`):** `` ## `--filter` ``, `` ### `baseUrl` ``, `` ### `extends` ``, `` ## `nub pm use` ``, `` ## `.npmrc` ``. Covers flags, subcommands, config/property fields, file names, env vars, and API identifiers. The "never START a heading with inline code" rule governs PROSE headings only (don't open a prose heading with a backtick); a heading that is *purely* a code token backticks the entire token and that is correct. A prose heading containing a token mid-phrase backticks just that token. **But register must be CONSISTENT within a sibling-heading group, and a heading is NEVER the place for pedantic correctness:** (a) **Don't MIX registers across sibling headings** — a run like `## Non-erasable syntax` (English) · `` ## `using` / `await using` `` (code) · `` ## `--enable-source-maps` `` (flag) is unacceptable; pick ONE register for the group, and for FEATURE/CONCEPT sections English wins — `` `using` / `await using` `` → `## Explicit resource management`, `` `--enable-source-maps` `` → `## Source maps`. A code-token heading is for a group that is genuinely command/flag/field REFERENCE (the `## nub install` / `## nub add` verb run, a flag-reference run); a lone *feature* never gets a code-token heading just because it happens to have a syntax. (b) **Never cram exact syntax into a heading for correctness** — the body carries the precise spelling: `` ## `${VAR}` and `$VAR` expansion `` → `## Variable expansion`. The heading names the topic cleanly; pedantic accuracy lives in the prose. **Features whose API surface relies on nub's OWN ambient TS declarations get a `<Callout>` at the very top:** tell the reader to install `@nubjs/types` as a devDep alongside `@types/node`, and to use `@types/node` **26**. Apply only to pages whose types actually come from `@nubjs/types`/`nub-env.d.ts` (data-format imports, `Worker`, `import.meta.hot`, …) — not pages typed entirely by `@types/node`; word it to match what that feature actually needs. Page slugs are command-aligned too (`/docs/run`, `/docs/pm`; the command-less file runner is `/docs/files`). **Terminal mockups show REAL captured output only — never invented lines** (the burned precedent: the site shipped a `nub run dev --watch` example for a flag that never existed, and invented `nub pm switch` output that the real binary errors on). **No inline-code pileups — applies to DOCS, not just marketing; they are unreadable and banned.** A sentence/paragraph that strings together a pile of back-to-back code chips (a flag list, a command list, a config-field list) is banned — `` `--frozen-lockfile`, `--no-frozen-lockfile`, `--prefer-frozen-lockfile`, `--prod`/`-P`… `` reads as noise. Move the enumeration into a fenced **code block** (one flag/command per line, `#` comments where they help) — or a table if it's genuinely tabular, but default to a code block. Keep any surrounding explanatory prose (a single sentence naming a difference is fine; the *list* is what must leave the paragraph). **Never address a concern nobody has — cut defensive / non-question editorializing.** Lines that pre-empt an imagined worry — "Nub never claims the yaml/toml/json5 import specifiers, so a bare `import` still resolves to the published package", "no minimum version", "not a version floor", "this is the correct, conservative behavior", "rather than silently misread" — answer questions a reader never had. If a reader wasn't going to ask it, don't answer it: delete the sentence. State what the feature DOES; do not defend against objections nobody raised. **Precedence / source / ordering lists are JUST the names — no explanatory paragraph per item (list the file names and that's all; a big explanatory sentence just makes it look complicated).** A resolution/precedence list (pin sources, config-file order, the PM-detection chain) is a bullet list of the file/field NAMES with at most a 2–3 word inline qualifier — never a full sentence per bullet. Long caveats, edge-case sources that technically win but never matter in practice (e.g. a Yarn-Berry signal), and error-on-ambiguity behavior go in a `<Callout>`, not inline in the list. And only list a source the code ACTUALLY consults — verify against the resolver before adding a bullet (don't assume a lockfile-inference step exists; confirm it). **Don't drop confusing cross-cutting-flag asides (esp. `--node`/the compat flag) into sections that never introduce the flag — mentioning it mid-section, or on a page that never alludes to the Node-compatibility flag, just confuses people.** The `--node` escape hatch belongs where it's the actual TOPIC (the runtime overview's `--node` section, the node-command page) or as a documented flag of the command a page is about (`nub run --node`, `nubx --node`) — NEVER as a tangential "passing `--node` disables/skips this" note tacked onto a PnP / TypeScript / web-storage / etc. behavior section. Introduce a cross-cutting flag once, on its own surface; don't sprinkle "and `--flag` turns this off" asides across unrelated feature pages. **No page opens with two big paragraphs before any other block-level element. EVER.** A page must reach a code block / list / table / callout / heading FAST — never two dense back-to-back prose paragraphs as the page opener (the resolution page's "Nub adds a TS-aware resolution layer…" + "If your editor's Go-to-Definition works…" double-open is the banned shape). Lead with one short framing sentence or two, then a block-level element (a code example is ideal — "show the cool thing"); fold the rest of the prose in after. This is the docs-side of the "open with the CLI working" / "no walls of text" rules. **DESCRIPTION fields carry NO inline code — they render raw and ugly. Write each as a regular sentence and find ways to use proper nouns or plain language rather than the actual API names, globals, or flags.** This covers every frontmatter `description:` and every homepage/card/`<Feature>` description-style prop that renders as PLAIN TEXT (inline `code` does not format there — a `` `--localstorage-file` `` shows literal backticks). Rewrite each as a normal sentence and DE-EMPHASIZE the code tokens — replace API names / globals / flags / file names / config fields with proper nouns or plain language ("`sessionStorage` works out of the box; `localStorage` persists once you name a backing file with `--localstorage-file`" → "Session storage works out of the box; persistent local storage turns on once you point it at a backing file"). The precise token names live in the page BODY (where inline code renders), never in the description. When in doubt, strip the token and describe the capability in English. **Know the rendered HTML/CSS before hand-rolling a code block in a React/MDX component.** This site's global CSS styles a bare `<code>` element as INLINE code (background, padding, rounded chip) — so a `<code>` nested inside a custom `<pre>` makes the whole block render like an inline-code pill, NOT a code block. A code block is a `<pre>` containing the text DIRECTLY — no inner `<code>` (`<pre>{`line1\nline2`}</pre>`). (Standard `<pre><code>` is semantically fine in the abstract, but under THIS site's CSS the `<code>` hijacks inline styling, so omit it for hand-built blocks — or use a real MDX code fence, which fumadocs styles correctly.) And never put `select-none` / per-span styling on parts of a code block — it makes those parts unselectable and reads as "weird, not a normal code block." A custom component's raw elements inherit the site's GLOBAL element styles; check how the thing actually renders (drive the browser / read the CSS) instead of assuming the markup is inert.

## Blog & marketing copy (`site/content/blog/`, homepage): structure rules

Distilled from line-edits on the "Introducing Nub" post (2026-06-10). These govern launch posts and long-form marketing prose. The homepage is the canonical register — when a passage or code block already exists there, reuse it rather than rewriting it.

- **NEVER start a sentence (or a heading) with inline code. ABSOLUTE — applies to docs, blog, homepage, captions, and chat.** A sentence may never open with a `code-formatted` token: not a command (`` `nub install` reads…``), not a flag, not a field name, not a filename. Lead with prose words and place the code inside the sentence — "The installer reads the lockfile…" / "Run `nub install` and it reads…", never "`nub install` reads the lockfile…". If a draft sentence begins with backticks, rewrite it before shipping. No exceptions, no "but it's the subject of the sentence."
- **Code blocks carry the argument.** Every feature claim wants a code block; pull the homepage's vetted blocks before writing new ones. A bullet list restating what a code block already shows is strictly worse than the code block.
- **Open with the CLI working; compatibility later.** The first code block lands within a sentence or two of the section heading — show the tool doing the cool thing before any feature/parity enumeration, or the reader is lost (lead with the cool part). Compat and parity notes go at the back of the section. Substantive mechanism prose may follow a code block, but never a one-line closer paragraph — fold closers into the prose above. Don't get into the weeds on internals the reader didn't ask about (e.g. "the only `.npmrc` key the script runner reads").
- **No walls of text.** Never open a section with a long multi-sentence paragraph. Two or more dense paragraphs in a row anywhere is a smell; 1–3 short paragraphs, then code.
- **No inline-code pileups.** A sentence stuffed with flag/command chips (`-r` / `--recursive` / `--workspace <n>` / …) always reads terribly — enumerate flags as lines inside the code block with `#` comments instead.
- **Cut clever-but-empty lines.** "If `tsc --noEmit` accepts your code, Nub runs it" — sounds tight, says nothing. Every sentence must add a fact.
- **Headings must look like headings.** If a rendered heading level sits near body-text size, fix the CSS or stop using a heading. Small repeated items (the polyfill APIs) read better as `` **`Worker`** — blurb `` bold+mdash lead-ins over their code block — which also keeps the TOC clean.
- **Benchmarks are visuals, not tables.** Use the homepage bar treatment (`<Bench>`, registered as a global MDX component). Each benchmark lives in the section it measures — the file-execution bench belongs to the file runner, not `nub run`. Present noise honestly: 44 ms vs 45 ms is a statistical tie, not a win.
- **FAQs are a curated handful.** Keep only the genuinely interesting ones in the post; everything else becomes a list of question-titles linking to `/docs/faq#<anchor>` (verify anchors against the rendered page, don't slug-guess).
- **Sell the implicit default, not the command surface.** Frame the automatic behavior as the product and explicit commands as the rare escape hatch — "a Node version manager you should never need."
- **Bailout commands are sub-section asides, not top-level sections.** Surfaces that exist for completeness (`nub node`, `nub pm`) get a short `###` under the section whose implicit behavior they back up ("there's not much to say, but they're good bailouts — worth noting so the tool feels complete"), never their own `##`.
- **Frame replacements as toolchain, not "npm packages."** What Nub displaces is project-level tooling — `nvm`, `corepack`, the PM CLI — much of which was never an npm package. And keep the PM's two compat axes distinct: pnpm-compatible CLI surface; lockfile-compatible with whatever the project already uses (npm / pnpm / bun round-trip, yarn read-only).
- **Asides are styled and sparing.** Default blockquote styling is unacceptable on the marketing site (`.blog-prose blockquote` carries the styled treatment); one aside per post is plenty.
- **Version-gated claims must trace to code, not the wiki.** Every site claim about a Node-version-gated feature must map to a named constant/function in `crates/nub-core/src/node/flags.rs` or `spawn.rs` (no symbol → the claim cannot ship). The wiki's `status: v0.1 "Confirmed"` tracks intent, not implementation — copy written from it shipped false claims (WebSocket, webstorage, 2026-06-10).
- **"Every supported Node version" is a banned phrase** unless the feature is `typeof`-feature-detected in the shared preload. Flag-injected features are tier- and version-banded; state the floor per row instead of glossing it.
- **No flowery phrasing in user-facing copy.** Aphorisms and metaphors get rewritten plain ("in the same breath", "earned the slow way", "the day the ecosystem cracks" — all cut 2026-06-10). "The brand stops at the binary boundary" is wiki-internal shorthand, never user-facing — and absolute "the name nub never appears anywhere" claims are false (`node_modules/.nub`, `~/.cache/nub`, internal `NUB_*` plumbing vars, error codes). The public claim is about *public surfaces only*: no nub-branded API you import or call, no `nub:*` import namespace, no `@nub/*` scope, no `"nub"` config field you author. **"No `NUB_*` environment variables" is a fine public claim** — in user-facing copy "`NUB_*` env vars" means vars *the user sets to change behavior*, and Nub has none; its internal cross-process plumbing vars aren't user knobs and don't contradict it. Don't go hedging that copy. The only thing to actually avoid is the absolute "the name nub never appears anywhere" (false — `node_modules/.nub`, `~/.cache/nub`). The promise is "zero Nub-specific *public* APIs / no lock-in." Concrete product comparisons are fine and encouraged ("the uv experience, for Node"; "uv for Node" is a good framing).
- **Showcase nub's protective refusals as a FEATURE — with REAL failure code blocks + a red ❌ in the comment on failure-condition lines.** Use these failure code blocks anywhere nub actually prevents unsound configuration. Wherever nub eagerly refuses an unsound/unsupported operation (it can't corrupt a `yarn.lock`, won't read a binary `bun.lockb`, rejects a `lockfileVersion 1`, etc.), SHOW it: the failing command + real captured error + (where useful) `$ echo $?` → exit code, then one tight framing sentence — never a paragraph apologizing for it. Put a **red ❌ in the inline comment** on any line indicating a failure/unsupported path (`nodeLinker: pnp   # ❌ nub installs hoisted node_modules instead`); use ✓/green for what works, ❌/red for what's refused, so the asymmetry reads at a glance. REAL captured output ONLY — capture from `target/release/nub` or ground the exact string in `crates/nub-cli/src/pm_engine/`; never invent. The distilled PM-pages style+tone guide lives in the per-rewrite prompt; this bullet is its durable summary.

## Never hard-wrap markdown paragraphs

**Every paragraph in every `.md` file in this repo is one long line.** Editors do soft-wrap. Hard line breaks inside a paragraph are forbidden. This applies to prose, blockquotes, list items, and table cells. Only code blocks, list-item boundaries, and headers introduce new lines.

If you find yourself wrapping at column 72/80/100, stop. Write the whole paragraph as one line and move on. There is a fixer at `wiki/scripts/unwrap-md.py` if you need to repair an existing document; do not invoke it as a substitute for not making the mistake.

## Releasing

Nub is published to npm as `@nubjs/nub` with 8 platform-specific binary packages. The release is fully automated via GitHub Actions.

```bash
make version V=0.0.6          # sets version in all 9 npm packages + Cargo.toml
make version-check             # verify consistency
git add -A && git commit -m "v0.0.6"
git tag v0.0.6
git push origin main --tags    # CI builds 8 platforms, publishes to npm, creates GitHub release
```

**Makefile targets:**
- `make version V=<ver>` — set version across all packages atomically.
- `make version-check` — verify all package versions are consistent.
- `make npm-build` — build + package for the current platform (local testing).
- `make npm-publish` — publish all packages manually (prefer CI).
- `make npm-publish-dry` — dry-run publish.

**CI release workflow** (`.github/workflows/release.yml`): triggers on `v*` tags. Builds for darwin-arm64, darwin-x64, linux-x64, linux-x64-musl, linux-arm64, linux-arm64-musl, win32-x64, win32-arm64. Publishes via npm OIDC trusted publishing (no secrets needed). Creates GitHub Release with binary artifacts.

**Version regime:** stay in `0.0.x` until ready for public launch. Bump to `0.1.0` only when the whitepaper, benchmarks, and install experience are polished.
